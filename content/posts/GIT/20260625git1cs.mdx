## 1. 분산 시스템 아키텍처 (Distributed System Architecture)
 
### 분산 버전 관리 시스템 (DVCS, Distributed Version Control System)
 
**정의**: 중앙 집중식 서버에만 의존하지 않고, 네트워크에 연결된 모든 클라이언트가
원격 저장소와 동일한 역사적 기록(메타데이터 및 오브젝트 데이터베이스)을
로컬에 완전히 복제하여 보유하는 아키텍처입니다.
 
**CS적 의의**:
 
- **단일 장애점(SPOF, Single Point of Failure) 배제**: 중앙 서버가 마비되더라도
  각 개발자의 로컬 저장소가 완전한 백업 역할을 하므로 시스템의 가용성이 극대화됩니다.
- **연산 최적화**: 이력 조회, 브랜치 전환, 커밋 등의 핵심 연산이
  로컬 파일 시스템에서 직접 수행되므로 네트워크 지연(Latency)의 영향을 받지 않습니다.
  단, `git log`처럼 커밋 전체를 순회하는 연산은 커밋 수에 비례하는 O(n) 복잡도를 가집니다.

## 2. 암호학 및 데이터 무결성 (Cryptography & Data Integrity)
 
### 암호학적 해시 함수 (Cryptographic Hash Function)
 
**정의**: 임의의 길이를 가진 데이터를 입력받아, 고정된 길이의 고유한 고밀도 이진 문자열(지문)을
출력하는 일방향성 알고리즘입니다.
 
Git의 해시 알고리즘 사용 현황:
 
| 버전 | 알고리즘 | 해시 길이 | 비고 |
|---|---|---|---|
| Git 2.28 이하 | SHA-1 | 160 bit | 현재 레거시, SHAttered 취약점 발견 |
| Git 2.29 이상 | SHA-256 | 256 bit | 현재 전환 중인 기본 알고리즘 |
 
> SHA-1은 2017년 구글의 SHAttered 프로젝트에서 충돌 공격이 실증되어
> Git은 SHA-256으로 전환을 진행 중입니다.
> 새 저장소 생성 시 `git init --object-format=sha256` 옵션으로 SHA-256을 명시할 수 있습니다.
 
**CS적 의의**:
 
- **역상 저항성(One-wayness)**: 출력값(해시 ID)을 가지고 입력값(원본 소스코드)을
  역산하는 것이 수학적으로 불가능합니다.
- **충돌 저항성(Collision Resistance)**: 서로 다른 두 입력값이 동일한 해시값을 출력하는
  확률이 극히 낮아, 파일명과 상관없이 내용 자체의 고유성을 완벽히 식별할 수 있습니다.
---
 
### 데이터 무결성 (Data Integrity)
 
**정의**: 시스템에 저장되거나 네트워크로 전송되는 데이터가 임의로 수정, 누락, 추가되지 않고
송신자가 의도한 원래의 상태 그대로 유지되고 있음을 보장하는 성질입니다.
 
**CS적 의의**: Git은 소스코드의 변경 사항뿐 아니라 작성자 정보, 타임스탬프,
부모 커밋의 해시 ID까지 모두 병합하여 하나의 해시 ID를 만듭니다.
이로 인해 과거의 단 1 Byte의 코드 변경도 상위 해시 체인을 깨뜨리게 되므로
데이터 무단 변조를 원천적으로 방지합니다.

## 3. 상태 머신 및 소프트웨어 공학 (State Machine & Software Engineering)
 
### 유한 상태 기계 (FSM, Finite State Machine)
 
**정의**: 컴퓨터 프로그램과 논리 회로를 설계할 때 사용되는 수학적 모델로,
유한한 개수의 상태를 가지며 특정 조건(이벤트)에 따라 다른 상태로 전이(Transition)하는
시스템을 일컫습니다.
 
**CS적 의의**: Git 저장소 내 파일들의 추적 상태는 엄격한 FSM 구조를 따릅니다.
 
```
[Untracked]
     │
     │  git add
     ▼
  [Staged] ◄────────────── git add ──────────────┐
     │                                            │
     │  git commit                                │
     ▼                                            │
[Unmodified]                                      │
     │                                            │
     │  파일 수정                                  │
     ▼                                            │
 [Modified] ─────────────────────────────────────┘
```
 
| 전이 | 이벤트 | 설명 |
|---|---|---|
| Untracked → Staged | `git add` | Git이 처음으로 파일을 인식 |
| Unmodified → Modified | 파일 수정 | 커밋 이후 내용이 바뀜 |
| Modified → Staged | `git add` | 변경분을 대기실에 올림 |
| Staged → Unmodified | `git commit` | 공식 기록으로 저장 완료 |
 
---
 
### 스냅샷 (Snapshot)
 
**정의**: 특정 시점의 데이터 구조나 파일 시스템 전체의 상태를 포착하여 그대로 저장한 가상 사본입니다.
 
**CS적 의의**: SVN과 같은 고전적 버전 관리 도구들이 파일의 '차이점(Delta)'을
누적하여 계산하는 방식을 취한 반면, Git은 변경이 발생한 시점의 파일 시스템 상태 전체를
스냅샷 단위로 기록합니다.
변경되지 않은 파일은 새로운 복사본을 만드는 대신 이전 버전 파일의 링크만을 가리키도록
구조를 최적화하여 저장 공간 효율성과 이력 복구 속도를 동시에 확보합니다.

## Vocabulary
 
| 용어 | 원어 | 정의 |
|---|---|---|
| SPOF | Single Point of Failure, 단일 장애점 | 전체 시스템 중 단 한 군데의 요소가 고장 났을 때 시스템 전체가 다운되는 핵심 취약 지점. 분산 아키텍처로 방지 가능 |
| Metadata | 메타데이터 | 어떤 대상을 설명하기 위해 부여된 '데이터를 위한 부가 데이터'. Git 커밋 객체에 기록되는 작성자명, 이메일, 타임스탬프 등이 해당 |
| Immutable Data Structure | 불변 데이터 구조 | 생성된 이후 내부 상태를 변경할 수 없는 객체 지향 및 기능적 설계 패턴. Git 오브젝트는 한 번 저장되면 수정되지 않고, 변경 시 항상 새로운 해시값을 가진 객체가 생성됨 |
| Delta | 차분 | 이전 버전과 현재 버전의 차이점만 저장하는 방식. SVN이 사용. Git은 스냅샷 방식을 채택 |
| SHAttered | SHA-1 충돌 공격 | 2017년 구글이 실증한 SHA-1 해시 충돌 공격. 서로 다른 두 PDF 파일이 동일한 SHA-1 해시를 갖도록 만드는 데 성공 |

## Key Questions
 
**Q1.** Git에서 두 개의 완전히 다른 경로에 위치한 동일한 내용의 텍스트 파일이 있을 때,
이 두 파일은 Git 내부 저장 공간에 어떻게 관리됩니까?
 
> **A.** Git은 내용 기반 주소 지정(Content-addressable) 방식을 사용합니다.
> 내용이 같으면 해시 결과가 동일하므로, 하나의 데이터 객체(blob)로만 저장되고
> 메타데이터 영역(tree 객체)에서 각각 다른 경로명으로 동일한 객체를 가리킵니다.
> 즉, 저장 공간은 한 번만 사용되고 경로 정보만 중복됩니다.
 
---
 
**Q2.** 분산 버전 관리 시스템이 네트워크 오프라인 상태에서도
'커밋'과 '로그 조회' 연산을 수행할 수 있는 CS적 설계 이유는 무엇입니까?
 
> **A.** 모든 클라이언트가 중앙 서버의 데이터베이스를 복제한 완전한 형태의
> 로컬 저장소를 하드디스크 내에 유지하고 있어, 모든 탐색과 연산이
> 네트워크 지연 없이 로컬 장치 내에서 직접 수행되기 때문입니다.
> 단, `git log`처럼 커밋 전체를 순회하는 연산의 실제 시간 복잡도는
> 커밋 수에 비례하는 O(n)이며, "빠르다"는 것은 네트워크 레이턴시가 없다는 의미입니다.
