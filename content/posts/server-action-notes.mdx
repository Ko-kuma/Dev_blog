## 서버 경계

App Router에서는 서버 컴포넌트와 클라이언트 컴포넌트의 책임을 분명히 나누는 것이 중요합니다. 데이터 조회, 파일 시스템 접근, SEO 메타데이터 생성은 서버에서 처리하고, 입력 상태나 브라우저 저장소처럼 사용자 기기와 직접 연결되는 일은 클라이언트에서 처리합니다.

## 글 데이터 읽기

이 블로그는 게시글을 파일 시스템에서 읽습니다. 빌드 시점에 정적으로 만들기 좋은 형태입니다.

```ts title="src/lib/posts.ts"
export function getPostBySlug(slug: string) {
  const fullPath = path.join(postsDirectory, `${slug}.mdx`);
  const raw = fs.readFileSync(fullPath, "utf8");
  const { data, content } = matter(raw);

  return {
    slug,
    content,
    ...data,
  };
}
```

### 정적 생성

`generateStaticParams`를 사용하면 MDX 파일 목록을 기준으로 상세 페이지 경로를 미리 생성할 수 있습니다.

```ts title="generateStaticParams"
export function generateStaticParams() {
  return getAllPosts().map((post) => ({
    slug: post.slug,
  }));
}
```

## 환경변수

Cloudinary와 Giscus는 공개 키 성격의 값만 클라이언트에서 사용합니다. 그래서 변수 이름에 `NEXT_PUBLIC_` 접두사를 붙였습니다.

<Callout tone="coral">
비밀 키가 필요한 API는 브라우저로 보내지 말고 서버 라우트나 별도 백엔드에서 다뤄야 합니다.
</Callout>

## 운영 메모

무료 환경으로 시작할 때는 GitHub 저장소와 Vercel 자동 배포만으로도 충분합니다. 글을 추가하고 push하면 Vercel이 새 정적 페이지를 빌드합니다.
