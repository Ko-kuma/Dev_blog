## 1. 타입 시스템의 두 축

### 동적 타이핑(Dynamic Typing)과 정적 타이핑(Static Typing)

**정의**: 동적 타이핑은 변수의 타입이 프로그램 실행 중에 결정되고 바뀔 수 있는 방식이고, 정적 타이핑은 코드가 실행되기 전(컴파일 시점)에 타입이 확정되어 있어야 하는 방식입니다.

**CS적 의의**:

- **Python의 본래 성격**: Python은 원래 `x = 3`이라고 쓰면 실행되는 순간 `x`가 정수라는 게 정해지는 동적 타이핑 언어입니다. 변수에 타입을 다시 대입해 문자열로 바꾸는 것도 자유롭게 허용됩니다.
- **트레이드오프**: 동적 타이핑은 코드를 빠르고 유연하게 작성할 수 있게 해주지만, 프로그램이 커질수록 "이 함수에 정확히 뭘 넣어야 하는지" 코드만 보고 파악하기 어려워지는 단점이 있습니다. 정적 타이핑은 그 반대의 트레이드오프를 가집니다.

---

### 점진적 타이핑 (Gradual Typing)

**정의**: 동적 타입 언어에 타입 힌트라는 선택적인 장치를 추가해서, 필요한 곳에만 부분적으로 정적 타이핑의 이점을 가져오는 접근 방식입니다.

**CS적 의의**:

- **Python의 타협점**: Python의 타입 힌트는 언어를 완전히 정적 타입으로 바꾸는 대신, 기존의 동적 타이핑은 그대로 유지하면서 원하는 부분에만 타입 정보를 "덧붙이는" 방식을 택했습니다. 이런 절충안을 점진적 타이핑이라고 부릅니다.
- **호환성 유지**: 이 방식 덕분에 기존에 타입 힌트 없이 작성된 수많은 Python 코드가 전혀 수정 없이도 계속 동작하면서, 새 코드에는 점차 타입 힌트를 추가해나갈 수 있습니다.

## 2. 타입 힌트를 다루는 도구

### 정적 분석(Static Analysis)과 mypy

**정의**: 프로그램을 실제로 실행하지 않고, 코드 자체를 읽어서 오류나 문제를 미리 찾아내는 검사 방식입니다.

**CS적 의의**:

- **런타임 이전에 문제 발견**: 타입 힌트를 어긴 코드는 Python 실행기 입장에서는 아무 문제가 없지만, `mypy` 같은 정적 분석 도구는 코드를 실행하지 않고도 "이 함수에 잘못된 타입이 전달되고 있다"는 것을 미리 잡아낼 수 있습니다. 버그를 사용자가 마주치기 전, 개발 단계에서 걸러내는 것이 핵심 가치입니다.
- **편집기 지원의 기반**: VSCode 등에서 제공하는 자동완성, 타입 오류 경고 같은 기능도 내부적으로 이런 정적 분석 원리를 사용해 코드를 실행하지 않고 타입 정보를 추론합니다.

---

### 런타임 검증(Runtime Validation)과 리플렉션(Reflection)

**정의**: 프로그램이 실제로 실행되는 도중에 타입이 맞는지 확인하는 것이며, 이때 코드가 자기 자신의 타입 정보(타입 힌트)를 스스로 읽어서 활용하는 능력을 리플렉션이라고 부릅니다.

**CS적 의의**:

- **Pydantic의 동작 원리**: mypy 같은 정적 분석 도구는 코드를 실행하지 않고 검사하지만, Pydantic은 반대로 프로그램이 실제로 돌아가는 도중에 `User(name="혜진", age="스물여덟")`처럼 값이 들어오는 그 순간 타입 힌트를 읽어서 실시간으로 검증합니다.
- **메타프로그래밍(Metaprogramming)과의 연결**: 클래스에 적어둔 타입 힌트라는 "코드에 대한 정보"를 그 코드 스스로가 읽어서 동작을 결정하는 것은, 코드가 코드 자신을 다루는 메타프로그래밍의 한 예시입니다. Pydantic과 FastAPI는 이 리플렉션 능력을 이용해 검증 로직과 API 문서를 개발자가 따로 작성하지 않아도 타입 힌트만으로 자동 생성합니다.

## Vocabulary

| 용어 | 원어 | 정의 |
|---|---|---|
| 동적 타이핑 | Dynamic Typing | 실행 중에 타입이 결정되는 방식 |
| 정적 타이핑 | Static Typing | 실행 전에 타입이 확정되어야 하는 방식 |
| 점진적 타이핑 | Gradual Typing | 동적 타입 언어에 선택적으로 타입 힌트를 더하는 절충 방식 |
| 정적 분석 | Static Analysis | 코드를 실행하지 않고 검사하는 방식 |
| 런타임 검증 | Runtime Validation | 프로그램 실행 중에 값이 맞는지 확인하는 것 |
| 리플렉션 | Reflection | 코드가 자기 자신의 정보를 읽어 활용하는 능력 |
| 메타프로그래밍 | Metaprogramming | 코드가 코드 자체를 다루는 프로그래밍 방식 |

## Key Questions

**Q1.** 타입 힌트가 실행 시점에 검사되지 않는다면, 사실상 그냥 주석이랑 다를 게 없는 것 아닌가요?

> **A.** 완전히 같지는 않습니다. 사람만 읽는 주석과 달리, 타입 힌트는 도구가 읽고 활용할 수 있는 구조화된 정보이기 때문입니다.
> 일반 주석은 편집기나 다른 프로그램이 그 내용을 이해하고 활용할 방법이 없지만, 타입 힌트는 `mypy`, VSCode의 자동완성, 그리고 Pydantic처럼 그 정보를 실제로 읽어서 검사·활용하는 도구들의 대상이 됩니다.
> 즉 Python 실행기 자체는 무시하지만, 그 주변 생태계(도구, 라이브러리)가 타입 힌트를 적극적으로 활용한다는 점에서 일반 주석과는 역할이 다릅니다.

---

**Q2.** mypy로 검사하는 것과 Pydantic으로 검증하는 것, 둘 다 타입을 확인하는데 뭐가 다른가요?

> **A.** 검사가 일어나는 시점과 대상이 다릅니다.
> mypy는 코드를 실행하기 전에, 코드 자체를 정적으로 읽어서 "이 코드에 타입이 안 맞는 부분이 있는가"를 미리 찾아냅니다. 실제 값이 아니라 코드의 구조만 봅니다.
> Pydantic은 프로그램이 실제로 실행되는 도중, 사용자가 입력한 실제 값이 들어올 때 그 값이 타입 힌트와 맞는지 그 자리에서 확인합니다. 예를 들어 사용자가 API로 잘못된 타입의 데이터를 보냈을 때 이를 걸러내는 건 mypy가 아니라 Pydantic 같은 런타임 검증 도구의 역할입니다.

---

**Q3.** Python이 애초에 정적 타입 언어였으면 더 간단했을 텐데, 왜 점진적 타이핑이라는 절충안을 택했나요?

> **A.** 기존 코드와 Python 특유의 유연성을 해치지 않기 위해서입니다.
> Python은 이미 타입 힌트 없이 작성된 방대한 코드가 존재하는 언어였기 때문에, 언어 자체를 정적 타입으로 강제로 바꾸면 그 모든 코드가 깨질 위험이 있었습니다.
> 점진적 타이핑은 기존 코드는 그대로 동작하게 두면서, 새로 작성하는 코드나 신뢰도가 중요한 부분에만 선택적으로 타입 힌트를 추가해 안정성을 높일 수 있게 해주는, Python의 철학(유연함)과 실용성을 동시에 지키는 타협안입니다.
