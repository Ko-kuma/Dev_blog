## 1. 동시성의 두 얼굴

### 동시성(Concurrency)과 병렬성(Parallelism)

**정의**: 동시성은 여러 작업을 번갈아 진행해서 "동시에 진행되는 것처럼" 처리하는 것이고, 병렬성은 여러 작업을 물리적으로 정말 같은 순간에 동시에 실행하는 것입니다.

**CS적 의의**:

- **하나의 진행자, 여러 작업**: `asyncio`의 비동기 처리는 동시성에 해당합니다. CPU(정확히는 하나의 스레드)는 한 번에 한 가지 일만 하지만, `A`가 응답을 기다리는 동안 `B`로 넘어가는 식으로 빠르게 번갈아가며 여러 작업을 "동시에 진행 중인 것처럼" 보이게 만듭니다.
- **오해하기 쉬운 차이**: `asyncio.gather(a, b)`로 두 작업이 "동시에" 끝나는 것처럼 보이지만, 실제로 CPU 연산이 물리적으로 병렬 실행되는 것은 아닙니다. 두 작업이 번갈아 기다림과 실행을 주고받는 동시성이지, 여러 코어가 진짜로 동시에 계산하는 병렬성이 아니라는 점이 비동기 프로그래밍을 이해하는 핵심 구분점입니다.

---

### I/O 바운드(I/O Bound)와 CPU 바운드(CPU Bound)

**정의**: I/O 바운드 작업은 네트워크·디스크 응답을 기다리는 시간이 대부분을 차지하는 작업이고, CPU 바운드 작업은 CPU가 쉬지 않고 계산을 수행하는 시간이 대부분을 차지하는 작업입니다.

**CS적 의의**:

- **비동기가 통하는 영역의 한계**: 비동기 프로그래밍이 성능을 개선하는 원리는 "기다리는 시간에 다른 일을 끼워 넣는 것"이기 때문에, 애초에 기다리는 시간이 거의 없는 CPU 바운드 작업에는 효과가 없습니다. 어떤 작업에 비동기를 적용할지 판단하려면 먼저 그 작업이 I/O 바운드인지 CPU 바운드인지부터 구분해야 합니다.
- **적재적소의 도구 선택**: CPU 바운드 작업의 성능을 개선하려면 비동기가 아니라 `multiprocessing`처럼 실제로 여러 CPU 코어를 병렬로 쓰는 접근이 필요합니다. 문제의 성격에 맞는 도구를 고르는 것이 중요합니다.

## 2. 비동기의 내부 동작

### 이벤트 루프(Event Loop)와 논블로킹 I/O(Non-blocking I/O)

**정의**: 이벤트 루프는 여러 코루틴 중 지금 실행 가능한 것을 계속 찾아서 실행해주는 스케줄러 역할을 하는 반복 구조이며, 논블로킹 I/O는 입출력 작업의 응답을 기다리는 동안 프로그램 전체가 멈추지 않는 방식입니다.

**CS적 의의**:

- **`asyncio.run()`이 하는 일**: `asyncio.run(main())`을 호출하면 이벤트 루프가 시작됩니다. 이 루프는 `await`를 만난 코루틴을 잠시 대기 목록에 넣어두고, 그 사이 실행 가능한 다른 코루틴이 있는지 계속 확인하며 작업을 번갈아 진행시킵니다.
- **블로킹과의 대비**: 일반적인 `time.sleep()`이나 동기 방식의 네트워크 호출은 응답이 올 때까지 프로그램 전체를 멈추게(블로킹) 하지만, `asyncio.sleep()`이나 비동기 네트워크 라이브러리는 기다리는 동안 이벤트 루프에게 제어권을 돌려줘서 다른 작업이 진행될 수 있게 합니다.

---

### GIL (Global Interpreter Lock)

**정의**: 표준 Python 인터프리터(CPython)에서, 한 번에 하나의 스레드만 Python 코드를 실행할 수 있도록 제한하는 잠금 장치입니다.

**CS적 의의**:

- **비동기가 필요한 배경 중 하나**: GIL 때문에 Python은 여러 스레드를 만들어도 CPU 연산 자체를 진짜 병렬로 실행하기 어렵습니다. 대신 I/O를 기다리는 동안에는 GIL이 다른 코드에 실행 기회를 넘겨줄 수 있어서, "기다림이 많은" 작업에서는 비동기(또는 멀티스레딩)로 충분한 성능 개선을 얻을 수 있습니다.
- **멀티프로세싱과의 구분**: CPU 바운드 작업에서 진짜 병렬성이 필요하다면, GIL의 제약을 받지 않는 별도의 프로세스를 여러 개 띄우는 `multiprocessing`을 사용해야 합니다. asyncio, 스레딩, 멀티프로세싱은 각각 다른 문제 상황을 위한 도구입니다.

## Vocabulary

| 용어 | 원어 | 정의 |
|---|---|---|
| 동시성 | Concurrency | 여러 작업을 번갈아 진행해 동시에 진행되는 것처럼 처리하는 것 |
| 병렬성 | Parallelism | 여러 작업을 물리적으로 같은 순간에 실행하는 것 |
| I/O 바운드 | I/O Bound | 입출력 응답 대기 시간이 대부분을 차지하는 작업 |
| CPU 바운드 | CPU Bound | CPU 계산 시간이 대부분을 차지하는 작업 |
| 이벤트 루프 | Event Loop | 실행 가능한 코루틴을 찾아 계속 실행해주는 스케줄러 |
| 논블로킹 I/O | Non-blocking I/O | 응답을 기다리는 동안 프로그램이 멈추지 않는 입출력 방식 |
| GIL | Global Interpreter Lock | CPython에서 한 번에 한 스레드만 코드를 실행하게 하는 잠금 |

## Key Questions

**Q1.** `asyncio.gather(a, b)`로 두 작업을 "동시에" 실행한다고 했는데, 진짜로 물리적으로 동시에 실행되는 건가요?

> **A.** 아닙니다. 정확히는 동시성이지 병렬성이 아닙니다.
> CPU(스레드 하나)는 한 순간에는 여전히 한 가지 작업만 실행하고 있습니다. 다만 `A`가 `await`로 응답을 기다리는 동안, 그 대기 시간을 그냥 흘려보내지 않고 재빨리 `B`의 코드로 넘어가 실행을 진행시킵니다.
> 결과적으로 두 작업의 "기다리는 시간"이 겹쳐지기 때문에 전체 소요 시간이 줄어드는 것이지, 두 코드가 정말 같은 순간에 동시에 계산되고 있는 것은 아닙니다.

---

**Q2.** 비동기로 바꾸면 무조건 더 빨라지나요?

> **A.** 아닙니다. 작업의 성격에 따라 다릅니다.
> 네트워크 요청, DB 쿼리, 파일 읽기처럼 "기다리는 시간"이 많은 I/O 바운드 작업이라면 비동기로 확실한 성능 개선을 얻을 수 있습니다.
> 반면 복잡한 수치 계산처럼 CPU가 쉬지 않고 계산만 하는 CPU 바운드 작업이라면, `async`로 감싸도 기다리는 시간 자체가 없기 때문에 개선 효과가 거의 없거나, 오히려 코드만 복잡해질 수 있습니다.

---

**Q3.** GIL이 있으면 Python은 여러 작업을 동시에 처리하지 못하는 건가요?

> **A.** "동시에 계산"은 제한되지만, "동시에 기다림"은 문제가 되지 않습니다.
> GIL은 한 번에 하나의 스레드만 Python 바이트코드를 실행하게 제한하기 때문에, 여러 스레드가 CPU 연산을 진짜로 병렬 처리하는 것은 어렵습니다.
> 하지만 `asyncio`가 다루는 상황은 대부분 "코드 실행"이 아니라 "응답을 기다리는 대기"이기 때문에, 이 대기 시간 동안에는 GIL이 다른 코루틴에게 실행 기회를 넘겨줄 수 있어 문제가 되지 않습니다. GIL의 제약이 실제로 문제가 되는 것은 CPU 바운드 작업을 여러 스레드로 병렬 처리하려 할 때입니다.
