## 1. Python 실행 환경

### 인터프리터 언어 (Interpreted Language)

**정의**: 코드를 사전에 기계어로 번역(컴파일)하지 않고, 한 줄씩 읽으면서 즉시 번역해 실행하는 언어입니다.

**CS적 의의**:

- **컴파일 언어 vs 인터프리터 언어**: C, C++ 같은 언어는 실행 전에 전체 코드를 기계어로 미리 번역합니다. Python은 인터프리터 언어로, 인터프리터가 코드를 한 줄씩 읽으면서 바로 실행합니다.
- **실행 속도 vs 개발 속도**: 컴파일 언어는 실행 속도가 빠르지만 코드를 수정할 때마다 다시 컴파일해야 합니다. 인터프리터 언어는 즉시 실행할 수 있어 개발 속도는 빠르지만, 매번 번역하는 과정 때문에 실행 속도는 상대적으로 느립니다.

```python
print("첫 번째 줄")   # 이 줄을 먼저 번역해서 즉시 실행
print("두 번째 줄")   # 그 다음 이 줄을 번역해서 실행
```

---

### CPython

**정의**: Python 언어의 표준 구현체이자, 가장 널리 쓰이는 인터프리터입니다. C언어로 작성되었습니다.

**CS적 의의**:

- **구현체(Implementation)**: Python은 언어 사양(스펙)이고, CPython은 그 사양을 실제로 실행하는 프로그램입니다. PyPy, Jython 같은 다른 구현체도 존재하지만, 터미널에서 `python` 명령어를 칠 때 실행되는 것은 대부분 CPython입니다.
- **GIL의 근원**: 뒤에서 다룰 GIL(Global Interpreter Lock)은 CPython이라는 구현체가 갖는 특성이지, Python 언어 자체가 반드시 가져야 하는 제약은 아닙니다.

---

### 저수준 언어 vs 고수준 언어 (Low-level vs High-level Language)

**정의**: 컴퓨터(하드웨어)에 얼마나 가까운 언어인지를 구분하는 개념입니다.

**CS적 의의**:

- **저수준 언어**: 어셈블리어, C처럼 메모리 주소를 직접 다루는 언어입니다. 실행은 빠르지만 사람이 읽고 쓰기 어렵습니다.
- **고수준 언어**: Python, JavaScript처럼 메모리 관리를 언어가 대신 처리해주는 언어입니다. 사람이 이해하기 쉽지만 상대적으로 느립니다.
- **추상화(Abstraction)**: 고수준 언어일수록 하드웨어의 세부사항을 추상화해서 감춥니다. `numbers = [1, 2, 3]`이라는 한 줄 뒤에는 실제로 메모리 할당, 포인터 연결 같은 저수준 작업이 숨어 있습니다.

| | 저수준 언어 | 고수준 언어 |
|---|---|---|
| 예시 | 어셈블리어, C | Python, JavaScript |
| 실행 속도 | 빠름 | 상대적으로 느림 |
| 가독성 | 낮음 | 높음 |

## 2. 기본 문법

### 동적 타이핑 (Dynamic Typing)

**정의**: 변수의 자료형(타입)을 미리 선언하지 않고, 값이 대입되는 시점에 자동으로 타입이 결정되는 방식입니다.

**CS적 의의**:

- **정적 타이핑 vs 동적 타이핑**: Java, TypeScript 같은 정적 타이핑 언어는 변수 선언 시 타입을 명시해야 합니다. Python은 동적 타이핑 언어로, `x = 5`라고만 써도 인터프리터가 실행 시점에 타입을 판단합니다.
- **타입은 변수가 아니라 값에 붙는다**: Python에서 타입은 변수 자체가 아니라 그 변수가 가리키는 값(객체)에 속합니다. 그래서 같은 변수에 다른 타입의 값을 재할당할 수 있습니다.

```python
x = 5        # 이 시점에 x는 int 타입으로 판단됨
x = "다섯"    # 같은 변수에 str 타입 값을 재할당해도 에러 없음
```

> 동적 타이핑은 빠르게 코드를 짤 수 있게 해주지만, 프로젝트가 커지면 타입 관련 버그가 실행 시점에야 발견될 수 있습니다. FastAPI/Pydantic이 이 문제를 보완하는 Type Hint를 적극 활용합니다.

---

### 들여쓰기 & IndentationError / TabError (Indentation)

**정의**: Python이 코드 블록(범위)을 구분하는 유일한 문법적 방법입니다. C나 JS의 중괄호 `{}`를 대신합니다.

**CS적 의의**:

- **오프사이드 룰(Off-side Rule)**: 들여쓰기로 코드 블록을 구분하는 문법 규칙을 컴퓨터과학에서 "오프사이드 룰"이라고 부릅니다. Python, Haskell 등이 이 규칙을 채택합니다.
- **IndentationError**: 들여쓰기 수준이 문법 규칙에 맞지 않을 때 발생하는 에러입니다.
- **TabError**: 한 파일 안에서 Tab과 Space를 섞어 쓸 때 Python 3에서 발생하는 에러입니다. PEP 8은 Space 4칸 사용을 권장합니다.

```python
if age >= 18:
    print("성인")   # 스페이스 4칸 들여쓰기로 if에 속함을 표시
```

---

### PEP 8

**정의**: Python 공식 코드 스타일 가이드 문서입니다. 들여쓰기, 네이밍 등 코드 작성 관례를 정의합니다.

**CS적 의의**:

- **코드 컨벤션(Code Convention)의 역할**: 여러 사람이 협업할 때 코드 스타일이 통일되어 있지 않으면 가독성이 떨어지고 Git diff 비교도 어려워집니다. PEP 8은 Python 커뮤니티 전체가 따르는 코드 컨벤션입니다.
- **PEP(Python Enhancement Proposal)**: Python 언어 자체나 관례를 제안하는 공식 문서 체계입니다. PEP 8은 그중 코드 스타일을 다루는 문서 번호입니다.

## 3. 언어 생태계

### 표준 라이브러리 (Standard Library)

**정의**: Python 설치 시 기본으로 포함되어 있어 별도 설치 없이 바로 사용할 수 있는 내장 모듈 모음입니다.

**CS적 의의**:

- **"배터리 포함(Batteries Included)" 철학**: Python은 파일 처리, 날짜 계산, 네트워크 통신 등 실무에서 자주 필요한 기능을 표준 라이브러리로 기본 제공합니다.
- **import 문**: 표준 라이브러리를 사용할 때는 `import` 문으로 불러와서 사용합니다 (설치 불필요).

```python
import json   # 표준 라이브러리 — 별도 설치 없이 바로 사용 가능
```

---

### PyPI (Python Package Index)

**정의**: 전 세계 개발자들이 만든 Python 패키지(라이브러리)를 등록하고 배포하는 공식 저장소입니다.

**CS적 의의**:

- **패키지 매니저(Package Manager)**: `pip`는 PyPI에서 패키지를 설치/관리하는 도구입니다. `pip install 패키지명` 명령으로 설치합니다.
- **의존성(Dependency)**: 프로젝트가 외부 패키지에 의존하는 관계를 "의존성"이라고 부릅니다. 앞으로 배울 FastAPI, SQLAlchemy도 모두 PyPI에서 설치하는 외부 의존성입니다.

```bash
pip install fastapi   # PyPI에서 FastAPI 패키지 설치
```

## 4. 실행 모델

### GIL (Global Interpreter Lock)

**정의**: CPython 인터프리터 내부에서, 한 번에 하나의 스레드만 Python 바이트코드를 실행하도록 제한하는 락(잠금) 메커니즘입니다.

**CS적 의의**:

- **스레드(Thread)**: 하나의 프로그램 안에서 독립적으로 실행되는 흐름 단위입니다. 여러 스레드를 동시에 실행하면 이론적으로 여러 작업을 병렬 처리할 수 있습니다.
- **GIL의 한계**: CPython은 메모리 관리를 단순하고 안전하게 하기 위해 GIL을 둡니다. 그 결과, CPU 연산이 많은 작업(예: 대량 계산)은 멀티스레딩으로 진짜 병렬화되지 않습니다.
- **우회 방법**: I/O 대기가 많은 작업(네트워크 요청 등)은 GIL의 영향을 덜 받습니다. CPU 연산이 많은 작업은 멀티프로세싱(multiprocessing)으로 우회합니다. (추후 FastAPI 비동기 처리에서 다시 다룰 개념)

## Key Questions

**Q1.** Python은 실행 속도가 느린 편인데, 왜 실무에서 이렇게 많이 쓰이나요?

> **A.** 고수준 언어 + 인터프리터 언어라는 특성이 "실행 속도"보다 "개발 속도"를 우선시하기 때문입니다.
> 프로젝트 초기에는 기능을 빠르게 만들고 검증하는 것이 더 중요한 경우가 많습니다.
> 또한 실제 서비스에서 병목이 되는 부분(대량 연산 등)은 C로 작성된 라이브러리(NumPy 등)를 불러 써서 속도를 보완할 수 있고, FastAPI 같은 프레임워크는 비동기 처리로 I/O 대기 시간을 효율적으로 활용해 체감 속도를 높입니다.

---

**Q2.** 들여쓰기를 Tab으로 하면 왜 안 되나요?

> **A.** 에디터마다 Tab의 너비가 다르게 표시되기 때문입니다.
> 어떤 에디터는 Tab을 2칸으로, 어떤 에디터는 4칸이나 8칸으로 표시해서 같은 코드가 사람마다 다르게 보입니다.
> 또한 Python 3에서는 한 파일 안에 Tab과 Space를 섞어 쓰면 `TabError`가 발생해 아예 실행이 안 됩니다.
> Git으로 코드를 비교(diff)할 때도 Tab은 표시가 애매해서 협업 시 충돌 원인이 되기 쉽습니다. 그래서 PEP 8은 Space 4칸을 권장합니다.

---

**Q3.** GIL이 있으면 여러 작업을 동시에 못 한다는 건데, FastAPI는 왜 빠르다고 하나요?

> **A.** GIL은 "CPU 연산"을 진짜 동시에 처리하지 못하게 막는 것이지, "I/O 대기"까지 막는 것은 아니기 때문입니다.
> 웹 서버가 하는 일의 대부분은 데이터베이스 응답 대기, 네트워크 요청 대기 같은 I/O 작업입니다.
> FastAPI는 `async`/`await`로 이런 대기 시간에 다른 요청을 처리할 수 있게 만들어서, GIL의 제약 안에서도 높은 처리량을 냅니다.
>
> ```python
> # FastAPI 예시 — 대기 중에도 다른 요청을 처리할 수 있음
> @app.get("/data")
> async def get_data():
>     result = await db.fetch("SELECT * FROM users")  # DB 응답을 기다리는 동안 다른 요청 처리 가능
>     return result
> ```
