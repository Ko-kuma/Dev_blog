## 1. 시스템 소프트웨어 (System Software)

### 운영체제 (OS, Operating System)

**정의**: 컴퓨터 하드웨어와 응용 프로그램 사이에서 자원을 관리하고 중재하는
시스템 소프트웨어입니다. 컴퓨터를 켜면 가장 먼저 실행됩니다.

**CS적 의의**:

- **자원 관리(Resource Management)**: CPU 시간, 메모리, 저장장치, 입출력 장치 등
  하드웨어 자원을 여러 프로그램이 충돌 없이 공유하도록 중재합니다.
- **추상화(Abstraction)**: 개발자가 하드웨어 세부사항을 몰라도
  파일을 읽고 쓸 수 있는 것은 OS가 복잡한 하드웨어 동작을 추상화해주기 때문입니다.

---

### 커널 (Kernel)

**정의**: 운영체제의 핵심 구성 요소로, 하드웨어를 직접 제어하는
가장 낮은 수준(Low-level)의 소프트웨어입니다.
항상 메모리에 상주하며 시스템 자원 전체를 관리합니다.

**CS적 의의**:

- **커널 공간 vs 사용자 공간(Kernel Space vs User Space)**: 커널은 보안과 안정성을 위해
  일반 프로그램이 접근할 수 없는 보호된 메모리 영역(커널 공간)에서 실행됩니다.
  우리가 짜는 코드는 사용자 공간에서 실행되고, 하드웨어 접근이 필요할 때만
  시스템 콜(System Call)을 통해 커널에 요청합니다.
- **실행 흐름**:

```
사용자 프로그램 (User Space)
        ↓  시스템 콜 (System Call)
    커널 (Kernel Space)
        ↓
    하드웨어 (Hardware)
```

---

### 쉘 (Shell)

**정의**: 사용자와 커널 사이에서 명령어를 해석하고 실행하는
인터페이스 프로그램입니다. 터미널을 열면 실행되는 것이 쉘이에요.

**CS적 의의**:

- **REPL 구조**: 쉘은 Read(입력 읽기) → Evaluate(명령어 해석) → Print(결과 출력) →
  Loop(반복) 구조로 동작합니다. Python 인터프리터, Node.js REPL과 같은 구조예요.
- **환경 변수 관리**: 쉘은 `PATH`, `HOME`, `SHELL` 같은 환경변수를 관리하며
  프로세스 실행 시 자식 프로세스에 전달합니다.

| 쉘 | 특징 |
|---|---|
| bash | POSIX 표준 준수. Linux 기본 쉘 |
| zsh | bash 상위 호환. macOS 기본 쉘 |
| fish | 현대적 UX, POSIX 비호환 |

---

### CLI vs GUI

**정의**:
- **CLI (Command Line Interface)**: 텍스트 명령어로 컴퓨터를 조작하는 인터페이스
- **GUI (Graphical User Interface)**: 아이콘, 버튼 등 그래픽으로 조작하는 인터페이스

**CS적 의의**:

- **자동화(Automation)**: CLI는 명령어를 스크립트 파일에 저장해 반복 실행할 수 있어요.
  서버 배포, 테스트, 빌드 자동화가 전부 CLI 기반인 이유입니다.
- **원격 접속**: SSH로 원격 서버에 접속하면 GUI가 없는 CLI 환경만 제공됩니다.
  Linux 명령어를 모르면 서버를 조작할 수 없는 이유예요.

## 2. 파일 시스템 (File System)

### 파일 시스템 (File System)

**정의**: 운영체제가 저장장치(HDD, SSD)에 데이터를 저장하고 관리하는 방식과 구조입니다.
파일과 디렉토리를 계층적 트리 구조로 관리합니다.

**CS적 의의**: Linux 파일 시스템은 **단일 루트 트리 구조**를 사용합니다.
모든 경로는 `/`(루트)에서 시작하며, Windows의 `C:\`, `D:\` 같은
드라이브 개념이 없습니다. 외장 드라이브도 `/mnt/` 아래에 마운트되어
하나의 트리 안에 포함됩니다.

```
/                   ← 루트 (최상위)
├── home/
│   └── heejin/     ← 홈 디렉토리 (~)
│       ├── Documents/
│       └── Projects/
├── etc/            ← 시스템 설정 파일
├── usr/            ← 사용자 프로그램
└── var/            ← 로그, 캐시 등 가변 데이터
```

---

### 절대 경로 vs 상대 경로 (Absolute Path vs Relative Path)

**정의**:
- **절대 경로**: 루트(`/`)부터 시작하는 전체 경로. 어디서 실행해도 동일한 위치를 가리킵니다.
- **상대 경로**: 현재 위치(Working Directory)를 기준으로 하는 경로.

**CS적 의의**: 절대 경로는 항상 동일한 위치를 가리키므로 **재현성(Reproducibility)**이
보장됩니다. 스크립트나 설정 파일에 경로를 하드코딩할 때 절대 경로를 쓰는 이유예요.
반면 상대 경로는 현재 위치에 따라 달라지므로 **이식성(Portability)**이 높습니다.

```bash
# 절대 경로 — 어디서 실행해도 동일
cd /Users/heejin/Documents

# 상대 경로 — 현재 위치 기준
cd Documents       # 현재 폴더 안의 Documents
cd ..              # 현재 폴더의 상위 폴더
cd ../sibling      # 현재 폴더와 같은 레벨의 sibling 폴더
```

---

### 재귀 (Recursion)

**정의**: 함수나 연산이 자기 자신을 반복적으로 호출하는 프로그래밍 기법입니다.
`rm -r`, `cp -r`에서 `-r` 플래그가 이 개념입니다.

**CS적 의의**: 디렉토리는 내부에 또 다른 디렉토리를 포함할 수 있는
**재귀적 자료구조(트리)**입니다. 폴더 전체를 삭제하려면
내부 폴더를 먼저 삭제하고, 그 내부 폴더의 내부 폴더를 먼저 삭제하는 식으로
재귀적으로 순회해야 합니다. `-r` 옵션이 이 재귀 순회를 활성화합니다.

---

### 표준 스트림 (Standard Streams)

**정의**: Unix/Linux에서 프로세스가 기본으로 갖는 3가지 입출력 채널입니다.

| 스트림 | 번호 | 설명 |
|---|---|---|
| stdin (표준 입력) | 0 | 키보드 입력 |
| stdout (표준 출력) | 1 | 터미널 화면 출력 |
| stderr (표준 오류) | 2 | 오류 메시지 출력 |

**CS적 의의**: `echo "텍스트" > 파일명`에서 `>`는 **리다이렉션(Redirection)** 연산자입니다.
stdout을 터미널 화면 대신 파일로 방향을 바꾸는 것이에요.
`>>`는 덮어쓰기 대신 이어쓰기(Append)를 합니다.

```bash
# stdout을 파일로 리다이렉션 (덮어쓰기)
echo "Hello" > output.txt

# stdout을 파일로 리다이렉션 (이어쓰기)
echo "World" >> output.txt

# stderr를 파일로 리다이렉션
ls 없는폴더 2> error.log
```

### 오픈소스 (Open Source)

**정의**: 소스코드가 공개되어 누구나 열람, 수정, 배포할 수 있는 소프트웨어 개발 방식입니다.
Linux 커널이 대표적인 오픈소스 프로젝트입니다.

**CS적 의의**:

- **라이선스(License)**: 오픈소스라고 모두 같은 조건이 아닙니다.
  Linux 커널은 GPL(GNU General Public License)을 따르며,
  GPL 코드를 사용한 소프트웨어는 마찬가지로 소스코드를 공개해야 합니다.
- **커뮤니티 기반 개발**: Linux 커널은 전 세계 수천 명의 개발자가
  기여하는 분산 협업 모델로 개발됩니다. Git이 이 협업을 위해 만들어진 도구예요.

## Vocabulary

| 용어 | 원어 | 정의 |
|---|---|---|
| OS | Operating System, 운영체제 | 하드웨어와 응용 프로그램 사이에서 자원을 관리하는 시스템 소프트웨어 |
| Kernel | 커널 | OS의 핵심. 하드웨어를 직접 제어하는 가장 낮은 수준의 소프트웨어 |
| Shell | 쉘 | 사용자 명령어를 해석해 커널에 전달하는 인터페이스 프로그램 |
| CLI | Command Line Interface | 텍스트 명령어로 컴퓨터를 조작하는 인터페이스 |
| GUI | Graphical User Interface | 그래픽(아이콘, 버튼)으로 컴퓨터를 조작하는 인터페이스 |
| File System | 파일 시스템 | OS가 저장장치에 데이터를 저장하고 관리하는 구조 |
| Absolute Path | 절대 경로 | 루트(/)부터 시작하는 전체 경로 |
| Relative Path | 상대 경로 | 현재 위치(Working Directory)를 기준으로 하는 경로 |
| Recursion | 재귀 | 함수나 연산이 자기 자신을 반복 호출하는 기법. rm -r의 -r이 재귀 순회 |
| stdin / stdout / stderr | 표준 입력/출력/오류 | 프로세스가 기본으로 갖는 3가지 입출력 채널 |
| Redirection | 리다이렉션 | 표준 스트림의 방향을 파일 등으로 바꾸는 것. > 와 >> 연산자 |
| System Call | 시스템 콜 | 사용자 프로그램이 커널 기능을 요청하는 인터페이스 |
| Open Source | 오픈소스 | 소스코드가 공개되어 누구나 열람·수정·배포 가능한 소프트웨어 |
| REPL | Read-Eval-Print Loop | 입력→해석→출력→반복 구조. 쉘, Python 인터프리터가 이 구조 |

## Key Questions

**Q1.** `rm -rf` 명령어가 위험한 CS적 이유는?

> **A.** `-r`은 디렉토리를 재귀적으로 순회하며 삭제하고,
> `-f`는 삭제 전 확인(confirmation) 프롬프트를 생략하는 강제(force) 옵션입니다.
> Linux 파일 시스템은 삭제 시 휴지통(Trash)으로 이동하지 않고
> 파일 시스템의 인덱스(inode)에서 직접 참조를 제거합니다.
> 데이터는 물리적으로 즉시 사라지는 것은 아니지만 OS가 해당 공간을
> 덮어쓸 수 있는 상태로 표시하므로 사실상 복구가 불가능합니다.

---

**Q2.** `echo "텍스트" > 파일명`과 `echo "텍스트" >> 파일명`의 차이를 CS적으로 설명하면?

> **A.** `>`는 stdout을 파일로 리다이렉션하면서 파일을 **O_TRUNC** 플래그로 열어
> 기존 내용을 모두 지우고 처음부터 씁니다.
> `>>`는 **O_APPEND** 플래그로 열어 파일 포인터를 끝으로 이동시킨 뒤
> 내용을 이어씁니다. 두 경우 모두 파일이 없으면 새로 생성됩니다.

---

**Q3.** Linux에서 모든 경로가 `/`(루트)에서 시작하는 단일 트리 구조를 사용하는 이유는?

> **A.** Unix 설계 철학인 **"모든 것은 파일이다(Everything is a file)"** 원칙에서 비롯됩니다.
> 하드디스크, USB, 네트워크 드라이브, 심지어 프로세스 정보까지
> 전부 파일 시스템 트리 안의 경로로 표현됩니다.
> 단일 트리 구조 덕분에 위치에 상관없이 동일한 파일 접근 인터페이스(read, write, open)를
> 사용할 수 있어 시스템 설계가 단순해집니다.
