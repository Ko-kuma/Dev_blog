## 1. super()와 메서드 오버라이딩

### super()와 초기화 위임

**정의**: 자식 클래스 안에서 부모 클래스의 메서드를 명시적으로 호출할 수 있게 해주는 내장 함수입니다.

**CS적 의의**:

- **위임(Delegation)**: 자식이 부모의 로직을 직접 다시 작성하지 않고, "이 부분은 부모에게 맡긴다"고 위임하는 것입니다. 코드 중복을 줄이면서도, 부모의 구현이 바뀌면 자식도 자동으로 그 변경을 따라가게 됩니다.
- **협력적 상속(Cooperative Inheritance)**: `super()`를 통해 부모의 초기화를 항상 거치도록 관례를 지키면, 상속 계층이 깊어져도 각 단계의 초기화가 순서대로 누락 없이 실행됩니다.

---

### 메서드 오버라이딩 (Method Overriding)과 동적 디스패치

**정의**: 자식 클래스가 부모로부터 물려받은 메서드를 같은 이름으로 다시 정의해서, 호출 시 자식의 버전이 실행되도록 하는 것입니다.

**CS적 의의**:

- **다형성(Polymorphism)의 실현**: 상속 CS 정리(Python10)에서 다형성을 짧게 언급했는데, 오버라이딩이 바로 그 다형성을 실제로 구현하는 방법입니다. `a.sound()`라는 동일한 코드가 `a`의 실제 클래스(`Dog`인지 `Cat`인지)에 따라 다른 결과를 냅니다.
- **동적 디스패치(Dynamic Dispatch)**: `a.sound()`를 실행할 때 어떤 클래스의 `sound()`를 실행할지가, 코드를 작성하는 시점이 아니라 프로그램이 실제로 실행되는 시점에 `a`의 실제 타입을 보고 결정됩니다. 이렇게 실행 시점에 대상이 결정되는 방식을 동적 디스패치라고 합니다.

---

## 2. 다중상속

### 다중상속(Multiple Inheritance)과 MRO(Method Resolution Order)

**정의**: 하나의 클래스가 둘 이상의 부모 클래스를 동시에 상속받는 것이며, MRO는 이름이 겹치는 메서드를 어떤 부모부터 찾을지 정해둔 순서입니다.

**CS적 의의**:

- **선형화(Linearization)**: 여러 부모가 뒤섞인 상속 관계를, Python은 C3 선형화라는 알고리즘으로 하나의 명확한 순서(일렬로 늘어선 리스트)로 정리합니다. `Duck.__mro__`로 그 결과를 직접 확인할 수 있습니다.
- **다이아몬드 문제(Diamond Problem)**: 두 부모가 공통의 조상을 가지면서 같은 메서드를 각자 재정의하고 있을 때, 어느 쪽을 따를지 애매해지는 상황을 다이아몬드 문제라고 부릅니다. MRO는 이 모호함을 "정해진 순서"로 해소하기 위한 장치입니다.

---

### 조합(Composition) — 상속의 대안

**정의**: 상속으로 기능을 물려받는 대신, 다른 클래스의 인스턴스를 속성으로 갖고 있으면서 그 기능을 활용하는 설계 방식입니다.

**CS적 의의**:

- **IS-A와 HAS-A**: 상속은 "자식은 부모의 한 종류다(IS-A)"라는 관계를 표현하는 반면, 조합은 "이 클래스는 저 클래스를 가지고 있다(HAS-A)"는 관계를 표현합니다. 관계의 성격이 다르므로, "정말 하위 종류인가" 아니면 "그냥 어떤 기능을 활용할 뿐인가"를 구분해서 상속과 조합 중 하나를 선택해야 합니다.
- **복잡도 관리**: 다중상속은 MRO처럼 추가적인 규칙을 이해해야 하지만, 조합은 "이 객체가 저 객체를 갖고 있고, 필요할 때 그 메서드를 호출한다"는 단순한 구조로 같은 효과를 낼 수 있어 실무에서 더 선호되는 경우가 많습니다.

## Vocabulary

| 용어 | 원어 | 정의 |
|---|---|---|
| 위임 | Delegation | 자신이 처리하지 않고 다른 대상에게 작업을 맡기는 것 |
| 협력적 상속 | Cooperative Inheritance | 상속 계층 전체가 순서대로 초기화를 이어가는 방식 |
| 다형성 | Polymorphism | 같은 이름의 호출이 대상에 따라 다르게 동작하는 성질 |
| 동적 디스패치 | Dynamic Dispatch | 실행 시점에 실제 타입을 보고 호출 대상이 정해지는 방식 |
| 선형화 | Linearization | 여러 갈래의 관계를 하나의 순서로 정리하는 것 |
| 다이아몬드 문제 | Diamond Problem | 다중상속에서 공통 조상의 기능이 어느 경로로 와야 할지 모호해지는 문제 |
| 조합 | Composition | 상속 대신 다른 객체를 속성으로 가지는 설계 방식 |

## Key Questions

**Q1.** `super().__init__()`을 안 쓰고 그냥 부모의 속성을 다시 적으면 안 되나요?

> **A.** 동작은 하지만 유지보수 관점에서 위험합니다.
> `self.name = name`을 자식 클래스 안에 직접 다시 적어도 당장은 똑같이 동작하지만, 나중에 부모 클래스의 초기화 로직이 바뀌면(예: 이름 앞뒤 공백을 제거하는 로직 추가) 자식 클래스는 그 변경을 전혀 반영하지 못하고 낡은 방식 그대로 남습니다.
> `super().__init__()`으로 위임해두면, 부모의 로직이 바뀔 때 자식도 자동으로 그 변경을 따라가게 되어 코드 중복과 불일치를 동시에 예방합니다.

---

**Q2.** 오버라이딩과, 기초에서 배운 함수 오버로딩(같은 이름, 다른 매개변수)은 같은 건가요?

> **A.** 이름은 비슷하지만 다른 개념입니다.
> 오버라이딩(Overriding)은 상속 관계에서 부모의 메서드를 자식이 재정의하는 것이고, 오버로딩(Overloading)은 같은 이름의 함수를 매개변수 개수나 타입만 다르게 여러 버전으로 정의하는 것입니다.
> 참고로 Python은 오버로딩을 언어 차원에서 직접 지원하지 않습니다. 같은 이름의 함수를 여러 번 정의하면 마지막 정의만 남고 이전 정의는 덮어써집니다. 매개변수 개수가 다른 경우를 다루고 싶다면 기본값(`=None`)이나 `*args`를 활용하는 것이 Python다운 방식입니다.

---

**Q3.** 다중상속이 있는데 왜 실무에서는 조합을 더 권장하나요?

> **A.** 다중상속은 클래스 관계가 복잡해질수록 MRO를 직접 추적해야 하는 부담이 커지기 때문입니다.
> 부모가 두세 개만 되어도 "이 메서드가 정확히 어느 부모에서 왔는지, 왜 이 순서로 실행됐는지"를 파악하려면 `__mro__`를 확인해야 하는 경우가 생깁니다.
> 조합은 "이 객체가 저 객체를 속성으로 가지고 있고, 필요할 때 그 메서드를 명시적으로 호출한다"는 단순한 구조라 관계가 코드에 그대로 드러나며, 큰 프로젝트일수록 이 명확함이 유지보수 비용을 크게 줄여줍니다.
