## 1. 정적 메서드와 클래스 메서드

### 정적 메서드 (Static Method)

**정의**: 인스턴스나 클래스 상태에 접근하지 않고, 클래스라는 이름공간 안에만 소속되어 있는 함수입니다.

**CS적 의의**:

- **네임스페이스(Namespace)의 재활용**: 모듈에서 배운 네임스페이스 개념이 클래스 안에서도 그대로 적용됩니다. `is_valid_age`라는 함수를 모듈 최상위에 두는 대신 `Dog` 클래스 안에 넣어두면, "이 함수는 `Dog`와 관련된 로직"이라는 사실이 이름공간을 통해 드러납니다.
- **결합도(Coupling) 낮추기**: 정적 메서드는 `self`나 `cls`에 의존하지 않기 때문에, 인스턴스 상태가 바뀌어도 영향을 받지 않는 독립적인 로직임이 코드 구조로 보장됩니다. 이런 독립성을 낮은 결합도라고 부르며, 테스트하거나 재사용하기 쉬운 코드의 특징입니다.

---

### 클래스 메서드 (Class Method)와 팩토리 메서드 패턴

**정의**: 인스턴스가 아니라 클래스 자기 자신(`cls`)을 받아서 동작하는 메서드로, 흔히 다른 형태의 데이터로부터 인스턴스를 만들어내는 데 쓰입니다.

**CS적 의의**:

- **팩토리 메서드 패턴(Factory Method Pattern)**: 객체를 생성하는 로직을 `__init__` 하나로 고정하지 않고, 여러 개의 "대안 생성 경로"를 제공하는 설계 방식을 팩토리 메서드 패턴이라고 부릅니다. `from_string`처럼 이름 자체가 "무엇으로부터(from) 만드는지"를 드러내는 것이 관례입니다.
- **다형성(Polymorphism)과 cls 바인딩**: 클래스 이름을 직접 하드코딩(`Dog(...)`)하지 않고 `cls(...)`로 적어두면, 이 메서드를 상속받은 자식 클래스에서 호출했을 때도 자식 클래스의 인스턴스가 만들어집니다. 같은 코드 한 줄이 호출한 클래스에 따라 다르게 동작하는 것으로, 다형성의 한 형태입니다.

---

## 2. property와 캡슐화 심화

### 프로퍼티 (Property)

**정의**: 메서드를 속성처럼 괄호 없이 접근하게 해주면서, 값을 읽거나 쓸 때 임의의 로직(검증, 계산 등)을 실행할 수 있게 해주는 장치입니다.

**CS적 의의**:

- **정보 은닉(Information Hiding)**: 클래스 기초에서 배운 캡슐화가 "데이터와 기능을 하나로 묶는 것"이었다면, 프로퍼티는 한 걸음 더 나아가 "그 데이터에 접근하는 방식 자체를 통제"합니다. 외부 코드는 `my_dog.age`라는 단순한 문법만 알면 되고, 내부적으로 검증이 일어나는지 계산이 일어나는지는 몰라도 됩니다.
- **인터페이스와 구현의 분리(Separation of Interface and Implementation)**: 처음에 `self.age`로 단순 저장했다가 나중에 검증 로직이 필요해져도, `@property`로 바꾸기만 하면 그 속성을 사용하던 외부 코드는 전혀 수정할 필요가 없습니다. "무엇을 하는지(인터페이스)"와 "어떻게 하는지(구현)"가 분리되어 있기 때문입니다.

---

### 읽기 전용 속성과 불변성

**정의**: `setter`를 정의하지 않은 프로퍼티처럼, 한 번 정해진 값이 이후에 바뀌지 않도록 막아둔 속성입니다.

**CS적 의의**:

- **불변성(Immutability)**: 값이 예상치 못하게 바뀌는 것을 원천 차단하면, 프로그램의 다른 부분에서 그 값을 안심하고 신뢰할 수 있습니다. 튜플이 리스트와 달리 값을 바꿀 수 없었던 것과 같은 원리가, 클래스 설계 단계에서도 의도적으로 적용될 수 있습니다.
- **실패를 빠르게(Fail Fast)**: `setter` 안에서 잘못된 값을 즉시 `ValueError`로 막아두면, 잘못된 데이터가 시스템 깊숙한 곳까지 흘러 들어가서 나중에 원인을 찾기 어려운 버그가 되는 것을 예방할 수 있습니다.

| 팩토리 메서드 패턴 | Factory Method Pattern | 객체 생성 로직을 대안 경로로 제공하는 설계 방식 |

## Key Questions

**Q1.** 클래스 메서드에서 `Dog` 대신 `cls`를 꼭 써야 하는 이유가 뭔가요?

> **A.** 상속 관계에서 다형성이 깨지기 때문입니다.
> `from_string` 안에서 `Dog(...)`라고 직접 이름을 적어두면, 나중에 `Puppy(Dog)`라는 자식 클래스가 이 메서드를 그대로 물려받아 호출해도 항상 `Dog` 인스턴스만 만들어집니다.
> `cls(...)`로 적어두면 "지금 이 메서드를 호출한 클래스가 누구든 그 클래스로 만들어라"는 뜻이 되어, `Puppy.from_string(...)`을 호출하면 실제로 `Puppy` 인스턴스가 만들어집니다.

---

**Q2.** `@staticmethod`로 클래스 안에 넣으나, 그냥 모듈 최상위에 함수로 두나 결국 똑같은 것 아닌가요?

> **A.** 동작은 비슷하지만 코드가 전달하는 의미와 이름공간 구조가 다릅니다.
> 모듈 최상위에 두면 그 함수가 특정 클래스와 관련 있다는 사실이 코드만 봐서는 드러나지 않고, 이름이 겹칠 위험도 커집니다.
> 클래스 안에 `@staticmethod`로 넣어두면 `Dog.is_valid_age`처럼 "이 로직은 Dog와 관련된 것"이라는 관계가 이름공간을 통해 명확하게 드러나고, 관련 로직끼리 한 클래스 아래 모여있어 코드를 찾기도 쉬워집니다.

---

**Q3.** `@property`가 캡슐화의 완성이라고 하는 이유는 뭔가요?

> **A.** 캡슐화는 "데이터와 기능을 묶는 것"에서 그치지 않고, "그 데이터에 접근하는 방식까지 통제하는 것"으로 완성되기 때문입니다.
> `self.age`를 그냥 공개 속성으로 두면 누구나 `my_dog.age = -100`처럼 잘못된 값을 자유롭게 대입할 수 있습니다.
> `@property`와 `setter`로 감싸두면, 겉보기엔 여전히 `my_dog.age = -100`처럼 단순하게 접근하지만 내부적으로 검증 로직을 반드시 거치게 되어, 클래스가 자기 자신의 데이터 무결성을 스스로 지킬 수 있게 됩니다.
