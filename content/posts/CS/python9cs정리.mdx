## 1. 모듈과 임포트

### 모듈 (Module)

**정의**: 변수, 함수, 클래스를 담아둔 파이썬 파일 하나를 가리키는 단위입니다.

**CS적 의의**:

- **네임스페이스(Namespace)**: 모듈은 코드를 담는 그릇이자 동시에 하나의 독립된 이름 공간입니다. `math_tools.add`처럼 모듈 이름을 붙여서 접근하면, 서로 다른 모듈에 같은 이름의 함수가 있어도 충돌하지 않습니다.
- **관심사의 분리(Separation of Concerns)**: 기능별로 파일을 나누는 것은, 하나의 파일이 너무 많은 역할을 떠맡지 않도록 코드를 조직화하는 기본적인 설계 원칙입니다.

---

### 모듈은 한 번만 로드된다 (Import Caching)

**정의**: 같은 모듈을 여러 파일에서 import해도, 그 모듈의 코드는 프로그램 전체에서 딱 한 번만 실행됩니다.

**CS적 의의**:

- **캐싱(Caching)**: Python은 처음 import된 모듈을 내부 저장소(`sys.modules`)에 기억해두고, 이후 같은 모듈을 다시 import하면 새로 실행하지 않고 기억해둔 결과를 그대로 재사용합니다. 이는 반복 작업을 피해 성능을 높이는 캐싱이라는 일반적인 기법의 한 사례입니다.
- **부작용(Side Effect)의 통제**: 모듈이 매번 다시 실행된다면, 모듈 안의 초기화 코드(예: 전역변수 설정)가 여러 번 실행되어 예상치 못한 상태가 될 수 있습니다. 한 번만 실행되는 규칙이 이런 혼란을 막아줍니다.

---

### `__name__ == "__main__"`

**정의**: 파일이 직접 실행되었는지, 아니면 다른 파일에서 import되었는지를 구분하는 데 쓰이는 특수 변수와 그 패턴입니다.

**CS적 의의**:

- **진입점(Entry Point)의 구분**: 프로그램이 시작되는 지점을 진입점이라고 부릅니다. 하나의 파일이 "재사용 가능한 라이브러리"와 "실행 가능한 스크립트" 역할을 동시에 겸할 때, 이 패턴으로 "지금 이 파일이 진입점으로 쓰이고 있는가"를 판단할 수 있습니다.
- **암묵적 컨텍스트 변수**: `__name__`은 개발자가 직접 값을 대입하지 않아도 Python 인터프리터가 실행 상황에 따라 자동으로 채워주는 특수 변수입니다. 직접 실행 시 `"__main__"`, import 시에는 모듈 이름이 자동으로 들어갑니다.

```python
if __name__ == "__main__":
    print("직접 실행될 때만 여기 도달함")
```

## 2. 패키지와 외부 라이브러리

### 패키지 (Package)

**정의**: 여러 모듈을 하나의 폴더로 묶고, 그 폴더를 하나의 단위로 import할 수 있게 만든 구조입니다.

**CS적 의의**:

- **계층적 구조화(Hierarchical Organization)**: 모듈이 파일 하나 단위의 조직화라면, 패키지는 폴더 단위로 한 단계 더 큰 규모의 조직화입니다. 프로젝트 규모가 커질수록 이런 계층 구조가 코드 탐색과 유지보수를 쉽게 만듭니다.

---

### pip과 PyPI

**정의**: `pip`은 Python 패키지를 설치·관리하는 도구이고, PyPI(Python Package Index)는 그 패키지들이 등록되어 있는 공식 저장소입니다.

**CS적 의의**:

- **패키지 매니저(Package Manager)**: 외부에서 만든 코드를 직접 복사해오지 않고, 이름과 버전만 지정하면 자동으로 찾아 설치해주는 도구를 패키지 매니저라고 부릅니다. 대부분의 현대 언어가 이런 도구를 갖추고 있습니다(JS의 npm, Java의 Maven 등).
- **중앙 저장소(Central Registry)**: PyPI처럼 패키지들을 한 곳에 모아 등록해두는 저장소가 있으면, 전 세계 개발자가 같은 이름으로 같은 패키지를 정확히 찾아 쓸 수 있습니다.

---

### 의존성 (Dependency)과 requirements.txt

**정의**: 의존성은 내 프로젝트가 정상적으로 동작하기 위해 필요로 하는 외부 패키지를 뜻하며, `requirements.txt`는 그 의존성 목록을 기록해둔 파일입니다.

**CS적 의의**:

- **재현 가능성(Reproducibility)**: 의존성 목록을 파일로 남겨두면, 다른 컴퓨터나 서버에서도 동일한 패키지 구성을 정확히 재현할 수 있습니다. 이는 "내 컴퓨터에서는 되는데 서버에서는 안 된다"는 문제를 예방하는 핵심 장치입니다.

---

### 버전 고정 (Version Pinning)

**정의**: `패키지명==버전번호` 형식으로 정확한 버전을 명시해서, 항상 그 버전이 설치되도록 강제하는 것입니다.

**CS적 의의**:

- **결정론적 빌드(Deterministic Build)**: 버전을 고정하지 않으면 설치 시점마다 다른 최신 버전이 설치될 수 있어, 같은 코드라도 실행 결과가 시점에 따라 달라질 위험이 있습니다. 버전을 고정하면 언제 설치하든 항상 같은 결과를 보장할 수 있습니다.
- **파괴적 변경(Breaking Change)에 대한 방어**: 패키지의 새 버전이 기존 사용법과 호환되지 않게 바뀌는 것을 파괴적 변경이라 부릅니다. 버전 고정은 이런 예상치 못한 변경으로부터 프로젝트를 보호하는 역할을 합니다.

---

## 3. 환경 격리

### 가상환경 (Virtual Environment)

**정의**: 프로젝트마다 독립된 패키지 설치 공간을 만들어주는 기능입니다.

**CS적 의의**:

- **환경 격리(Environment Isolation)**: 여러 프로젝트가 컴퓨터 전체의 패키지 공간을 공유하면, 프로젝트마다 필요한 패키지 버전이 서로 충돌할 수 있습니다. 가상환경은 이런 충돌을 막기 위해 프로젝트별로 독립된 실행 환경을 분리하는 기법입니다.
- **컨테이너(Container)와의 유사성**: 가상환경은 이후 DevOps 단계에서 배울 Docker 컨테이너와 "독립된 실행 환경을 만든다"는 목적이 비슷합니다. 다만 가상환경은 Python 패키지만 격리하고, Docker는 운영체제 수준까지 통째로 격리한다는 점에서 격리 범위가 다릅니다.

## Key Questions

**Q1.** 모듈이 한 번만 로드된다는 게 왜 중요한가요?

> **A.** 모듈 안에 있는 초기화 코드가 여러 번 실행되는 것을 막아주기 때문입니다.
> 만약 매번 새로 실행된다면, 여러 파일에서 같은 모듈을 import할 때마다 그 모듈의 전역변수가 초기화되거나 출력문이 반복 실행되는 등 예상치 못한 부작용이 생길 수 있습니다.
> Python이 import된 모듈을 내부적으로 기억해두고 재사용하는 덕분에, 몇 번을 import하든 항상 예측 가능하게 동작합니다.

---

**Q2.** `if __name__ == "__main__":`이 없으면 무슨 문제가 생기나요?

> **A.** 다른 파일에서 import할 때 원치 않는 코드까지 함께 실행될 수 있습니다.
> 예를 들어 테스트 출력이나 실행용 코드가 모듈 최상단에 그대로 있으면, 그 모듈을 단순히 함수 하나만 가져다 쓰려고 import해도 그 코드가 실행되어버립니다.
> `if __name__ == "__main__":`으로 감싸두면, 그 파일이 "직접 실행되는 진입점"으로 쓰일 때만 해당 코드가 실행되고, 단순히 import되어 라이브러리처럼 쓰일 때는 실행되지 않습니다.

---

**Q3.** 가상환경과 `requirements.txt`는 어떤 관계인가요?

> **A.** 가상환경은 "지금 이 컴퓨터의 이 프로젝트 폴더에 실제로 무엇이 설치되어 있는가"를 담당하고, `requirements.txt`는 "무엇이 설치되어야 하는지"를 기록한 설계도 역할을 합니다.
> 가상환경 자체는 용량이 크고 컴퓨터마다 미묘하게 달라질 수 있어 그대로 공유하지 않고, 대신 `requirements.txt`라는 가벼운 텍스트 파일만 공유합니다.
> 이 파일을 받은 사람은 자기 컴퓨터에 새 가상환경을 만들고 `pip install -r requirements.txt`로 똑같은 패키지 구성을 재현합니다.
