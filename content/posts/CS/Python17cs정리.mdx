## 1. 컨텍스트 매니저 프로토콜

### 컨텍스트 매니저 프로토콜 (Context Manager Protocol)

**정의**: `__enter__`와 `__exit__` 두 메서드를 구현하면 `with`문에서 쓸 수 있게 되는 약속입니다.

**CS적 의의**:

- **또 하나의 프로토콜**: 이터레이터 프로토콜(Python16)이 `__iter__`/`__next__`로 순회를 표준화했듯, 컨텍스트 매니저 프로토콜은 `__enter__`/`__exit__`로 "자원의 시작과 끝"을 표준화합니다. Python에서 반복적으로 등장하는 설계 방식 — 정해진 이름의 메서드 몇 개를 구현하면 언어의 특정 문법을 그대로 쓸 수 있게 되는 방식 — 이 여기서도 나타납니다.
- **자원 관리의 일반화**: 파일 입출력(Python10)에서 배운 "자원은 반드시 정리해야 한다"는 원칙이, 파일이라는 특정 대상에 국한되지 않고 DB 연결, 락, 네트워크 소켓 등 모든 자원에 동일한 방식으로 적용될 수 있게 됩니다.

---

### RAII(자원 획득 즉 초기화)와의 비교

**정의**: 자원의 획득과 해제를 객체의 생명주기에 묶어서, 객체가 만들어질 때 자원을 얻고 사라질 때 자원을 자동으로 반납하게 하는 설계 개념입니다. (본래 C++에서 유래한 용어입니다.)

**CS적 의의**:

- **보장된 정리(Guaranteed Cleanup)**: RAII의 핵심은 "정리를 개발자의 기억력에 맡기지 않고 언어/구조가 보장한다"는 것입니다. Python의 `with`문과 `__exit__`도 정확히 같은 목표를 다른 방식(가비지 컬렉션이 아니라 명시적인 블록 범위)으로 달성합니다.
- **명시적 범위 vs 자동 소멸**: C++의 RAII는 객체가 스코프를 벗어나 자동으로 소멸될 때 정리가 일어나지만, Python은 가비지 컬렉션 시점이 불명확하기 때문에 `with` 블록이라는 명시적인 범위를 문법으로 만들어 "여기서부터 여기까지"를 개발자가 눈으로 확인할 수 있게 했습니다.

## 2. 예외와 컨텍스트 매니저

### 예외 억제 (Exception Suppression)

**정의**: `__exit__`가 `True`를 반환해서, 블록 안에서 발생한 예외가 바깥으로 전파되지 않고 그 자리에서 처리된 것으로 취급되는 동작입니다.

**CS적 의의**:

- **암묵적 흐름 제어의 위험성**: 예외 처리(Python8)에서 배운 `except`는 예외를 잡는다는 것이 코드에 명시적으로 드러납니다. 반면 `__exit__`의 예외 억제는 `with`문 바깥에서 보면 어떤 예외가 왜 사라졌는지 코드만 봐서는 알기 어려워, 신중하게 사용해야 하는 기능으로 간주됩니다.
- **책임의 위치**: 예외를 억제할지 결정하는 책임이 `with`문을 사용하는 쪽이 아니라, 컨텍스트 매니저를 설계한 쪽(`__exit__` 구현)에 있다는 점이 설계상 중요한 포인트입니다. 컨텍스트 매니저를 만들 때는 "이 예외를 정말 삼켜도 되는가"를 신중히 판단해야 합니다.

---

### `contextlib.contextmanager`와 제너레이터의 재활용

**정의**: 제너레이터 함수 하나에 `@contextmanager`를 붙여서, 클래스를 따로 정의하지 않고도 컨텍스트 매니저를 만들 수 있게 해주는 표준 라이브러리 도구입니다.

**CS적 의의**:

- **기존 메커니즘의 재해석**: 제너레이터(Python16)의 "실행을 멈췄다가 재개한다"는 성질이, 여기서는 "자원을 준비하고(`yield` 이전) → 사용하게 해주고(`yield`) → 정리한다(`yield` 이후)"는 세 단계로 재해석되어 쓰입니다. 완전히 다른 목적처럼 보이는 두 기능(순회와 자원 관리)이 같은 언어 메커니즘 위에 세워져 있음을 보여주는 사례입니다.
- **보일러플레이트(Boilerplate) 감소**: `__enter__`, `__exit__`, `__init__`을 매번 갖춘 클래스를 만드는 대신 함수 하나로 대체함으로써, 반복적으로 작성해야 하는 정형화된 코드(보일러플레이트)를 줄여줍니다.

## Vocabulary

| 용어 | 원어 | 정의 |
|---|---|---|
| 컨텍스트 매니저 프로토콜 | Context Manager Protocol | `__enter__`/`__exit__` 구현으로 `with`문을 지원하는 약속 |
| RAII | Resource Acquisition Is Initialization | 자원 획득과 해제를 객체 생명주기에 묶는 설계 개념 |
| 보장된 정리 | Guaranteed Cleanup | 정리 작업이 반드시 실행되도록 구조로 보장하는 것 |
| 예외 억제 | Exception Suppression | 발생한 예외를 바깥으로 전파하지 않고 처리 완료로 취급하는 것 |
| 보일러플레이트 | Boilerplate | 매번 반복적으로 작성해야 하는 정형화된 코드 |

## Key Questions

**Q1.** `try/finally`만으로도 자원을 정리할 수 있는데, 왜 굳이 컨텍스트 매니저를 따로 만드나요?

> **A.** 재사용성과 실수 방지 때문입니다.
> `try/finally`는 그 자리에서 한 번만 쓰고 마는 코드지만, 컨텍스트 매니저로 만들어두면 똑같은 "준비-정리" 로직을 `with 객체:`라는 한 줄로 어디서든 재사용할 수 있습니다.
> 또한 매번 `try/finally`를 손으로 작성하면 정리 코드를 빼먹는 실수가 생길 수 있지만, 컨텍스트 매니저는 그 로직을 한 곳에 모아두기 때문에 실수할 여지가 줄어듭니다.

---

**Q2.** `__exit__`가 예외를 무조건 잡아서 처리해버리면 편할 것 같은데, 왜 신중해야 하나요?

> **A.** 진짜 문제가 조용히 사라져버릴 수 있기 때문입니다.
> 예를 들어 DB 연결 중 발생한 예외를 `__exit__`에서 무조건 `True`로 억제해버리면, 실제로는 DB 연결에 심각한 문제가 있는데도 프로그램은 아무 일 없다는 듯 계속 실행됩니다.
> 그러면 나중에 데이터 정합성이 깨지는 등 훨씬 찾기 어려운 문제로 이어질 수 있어서, 예외를 억제할 때는 정확히 어떤 예외를, 왜 무시해도 되는지를 명확히 하고 로그도 남기는 것이 안전합니다.

---

**Q3.** `@contextmanager` 방식과 클래스로 만드는 방식 중 뭘 언제 써야 하나요?

> **A.** 상태의 복잡도에 따라 다릅니다.
> `@contextmanager`는 "자원을 준비하고, 쓰게 해주고, 정리한다"는 단순한 흐름일 때 적합하며 코드도 짧습니다.
> 반면 컨텍스트 매니저가 여러 메서드나 복잡한 상태(재시도 횟수, 여러 자원 관리 등)를 가져야 한다면, 클래스로 만들어 `__enter__`/`__exit__` 외에도 필요한 메서드와 속성을 자유롭게 추가하는 쪽이 구조적으로 더 명확합니다.
