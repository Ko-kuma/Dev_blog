## 1. 이터레이터 프로토콜

### 이터러블과 이터레이터, 이터레이터 프로토콜 (Iterator Protocol)

**정의**: 이터러블은 순회 가능한 대상이고, 이터레이터는 그 안에서 실제로 값을 하나씩 꺼내는 역할을 합니다. `__iter__`와 `__next__`라는 정해진 이름의 메서드를 구현하면 이 역할을 할 수 있다는 약속을 이터레이터 프로토콜이라고 부릅니다.

**CS적 의의**:

- **프로토콜을 통한 다형성**: 연산자 오버로딩(Python14)에서 다룬 프로토콜 개념이 여기서도 그대로 적용됩니다. 리스트든, 파일이든, 직접 만든 `Countdown` 클래스든, 정해진 메서드 이름만 구현하면 `for`문이라는 같은 문법으로 다룰 수 있습니다.
- **관심사의 분리**: `for`문을 사용하는 코드는 "어떻게 다음 값을 만들어내는지" 전혀 몰라도 되고, `next()`만 호출하면 된다는 사실만 알면 충분합니다. 값을 만드는 방법(구현)과 값을 소비하는 방법(사용)이 분리되어 있습니다.

---

### StopIteration과 종료 신호

**정의**: 이터레이터에 더 이상 꺼낼 값이 없음을 알리기 위해 발생시키는 예외입니다.

**CS적 의의**:

- **제어 흐름으로서의 예외**: 일반적으로 예외는 "잘못된 상황"을 나타내지만, `StopIteration`은 에러가 아니라 "정상적으로 끝났다"는 신호로 쓰입니다. 예외 처리 메커니즘이 오류 보고 외에 흐름 제어 신호로도 활용될 수 있음을 보여주는 사례입니다.
- **프로토콜의 필수 조건**: 이터레이터 프로토콜을 따르려면 `__next__`가 끝날 시점에 반드시 이 신호를 보내야 하며, 빠뜨리면 무한 루프로 이어집니다.

## 2. 제너레이터와 지연 평가

### 제너레이터(Generator)와 상태 머신(State Machine)

**정의**: `yield`를 포함한 함수를 호출하면 만들어지는 특수한 이터레이터로, 실행 중간에 멈췄다가 다음 호출 때 멈춘 지점부터 다시 실행을 이어갈 수 있습니다.

**CS적 의의**:

- **실행 상태의 보존**: 일반 함수는 호출될 때마다 처음부터 새로 실행되지만, 제너레이터는 `yield`에서 멈춘 시점의 지역 변수와 실행 위치를 그대로 기억합니다. 이렇게 "지금 어느 단계에 있는지"를 스스로 기억하고 있는 구조를 상태 머신이라고 부릅니다.
- **코루틴(Coroutine)의 기초**: 실행을 중간에 멈추고 재개할 수 있는 이 성질은, 나중에 배울 비동기 프로그래밍(`async`/`await`)의 코루틴 개념과 근본 원리가 같습니다. 제너레이터를 이해해두면 비동기 함수의 동작 방식을 이해하기가 훨씬 수월해집니다.

---

### 지연 평가 (Lazy Evaluation)와 메모리 효율

**정의**: 값을 미리 전부 계산해두지 않고, 실제로 필요한 시점에 그때그때 계산하는 방식입니다.

**CS적 의의**:

- **공간 복잡도(Space Complexity)의 개선**: 100만 개의 값을 리스트로 만들면 100만 개 전부가 동시에 메모리에 존재해야 하지만, 제너레이터는 한 번에 하나의 값만 메모리에 있으면 됩니다. 데이터의 크기와 무관하게 항상 일정한(상수) 메모리만 사용할 수 있습니다.
- **무한 시퀀스의 표현**: 지연 평가 덕분에, 끝이 없는 무한한 값의 흐름(예: 무한히 증가하는 숫자)도 제너레이터로는 문제없이 표현할 수 있습니다. 리스트로는 애초에 만들 수 없는 것이, 필요한 만큼만 계산하는 방식 덕분에 가능해집니다.

## Vocabulary

| 용어 | 원어 | 정의 |
|---|---|---|
| 이터러블 | Iterable | 순회 가능한 대상 |
| 이터레이터 | Iterator | 값을 하나씩 꺼내주는 역할을 하는 객체 |
| 이터레이터 프로토콜 | Iterator Protocol | `__iter__`/`__next__` 구현으로 순회를 지원하는 약속 |
| 상태 머신 | State Machine | 현재 어느 단계에 있는지를 스스로 기억하는 구조 |

| 지연 평가 | Lazy Evaluation | 필요한 시점까지 계산을 미루는 방식 |
| 공간 복잡도 | Space Complexity | 알고리즘이 사용하는 메모리 양을 나타내는 척도 |

## Key Questions

**Q1.** 클래스로 이터레이터를 직접 만드는 것과 `yield`를 쓰는 것, 결과가 똑같은데 왜 두 가지 방법이 있나요?

> **A.** 결과는 같지만 필요한 제어 수준이 다르기 때문입니다.
> `__iter__`/`__next__`를 직접 구현하는 방식은 번거로운 대신, 상태를 여러 속성에 나눠 담거나 더 복잡한 흐름을 세밀하게 제어할 수 있습니다.
> `yield`는 대부분의 경우에 필요한 "상태 기억 + 순서대로 값 내어주기"를 훨씬 적은 코드로 자동 처리해주기 때문에, 실무에서는 특별한 이유가 없다면 제너레이터 방식을 기본으로 선택합니다.

---

**Q2.** 리스트 컴프리헨션과 제너레이터 표현식은 대괄호냐 소괄호냐 차이만 있나요?

> **A.** 문법은 비슷하지만 "언제 계산이 일어나는가"가 근본적으로 다릅니다.
> 리스트 컴프리헨션은 그 줄이 실행되는 즉시 모든 값을 계산해서 리스트에 채워 넣습니다.
> 제너레이터 표현식은 그 줄이 실행되어도 아직 아무 값도 계산하지 않고, 나중에 `next()`가 호출되거나 `for`문이 순회할 때 비로소 하나씩 계산합니다. 데이터가 크거나 전부 다 쓸지 알 수 없는 상황에서는 제너레이터 쪽이 불필요한 계산을 피할 수 있습니다.

---

**Q3.** 제너레이터가 나중에 배울 비동기(async/await)와 무슨 상관이 있나요?

> **A.** 실행을 멈췄다가 다시 이어가는 근본 메커니즘을 공유하기 때문입니다.
> 제너레이터가 `yield`에서 멈추고 상태를 기억했다가 나중에 이어서 실행하는 것처럼, `async` 함수도 `await` 지점에서 실행을 멈추고 다른 작업이 끝나면 그 지점부터 이어서 실행합니다.
> 실제로 Python의 비동기 프로그래밍은 초기에 제너레이터를 확장하는 방식으로 구현되었을 만큼, 두 개념은 "실행 중간에 멈췄다가 재개하는 함수"라는 같은 뿌리를 가지고 있습니다.
