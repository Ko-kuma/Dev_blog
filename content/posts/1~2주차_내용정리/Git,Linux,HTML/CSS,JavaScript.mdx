## 웹 개발 5대 영역 — 식당 비유로 다시 보기

| 역할 | 식당 비유 | 기술 | 핵심 키워드 |
|---|---|---|---|
| 프론트엔드 | 홀, 메뉴판 | Next.js / React | 사용자가 보는 모든 것 |
| 백엔드 | 주방 | FastAPI / Python | API — 요청과 응답 |
| 데이터베이스 | 냉장고, 창고 | PostgreSQL | 행(row)·열(column) 구조 |
| 보안 | 경비원 | JWT / CORS | 인증(누구냐) vs 인가(뭘 할 수 있냐) |
| DevOps | 건물주, 관리자 | Docker / Vercel | 무중단으로 계속 돌아가게 |

### 심화 — 인증과 인가는 왜 분리해서 생각해야 하나

로그인했다고 해서 모든 작업을 할 수 있는 건 아니다. **인증(Authentication)** 은 "이 사람이 누구인지" 확인하는 절차이고, **인가(Authorization)** 는 "이 사람이 이 작업을 할 권한이 있는지" 확인하는 절차다. 둘을 하나로 합쳐 처리하면 일반 회원이 관리자 기능에 접근하는 보안 취약점이 생기기 쉽다. 실무에서는 JWT로 인증을 먼저 통과시킨 뒤, 별도의 권한 검사 로직(역할 기반 접근 제어, RBAC)으로 인가를 분리해서 처리한다.

### 심화 — API가 "약속"인 이유

API는 단순히 "데이터를 주고받는 통로"가 아니라, 프론트엔드와 백엔드가 미리 합의한 **요청-응답의 형식 규칙**이다. 어떤 URL로, 어떤 메서드(GET/POST 등)로, 어떤 데이터 형식(JSON)으로 보내야 하는지가 정해져 있어야 양쪽이 독립적으로 개발해도 서로 맞물려 동작한다. FastAPI를 배우면 이 "약속"을 코드로 명시하는 방법(Pydantic 스키마)을 다루게 된다.

# Git / GitHub

## Git의 3가지 공간

```
[작업 폴더] → git add → [스테이지] → git commit → [저장소]
(편집 중)              (저장 대기실)              (기록 완료)
```

| 공간 | 영문 | 비유 |
|---|---|---|
| 작업 폴더 | Working Directory | 실제 편집 공간 |
| 스테이지 | Staging Area | 저장 대기실 |
| 저장소 | Repository | 기록이 완료된 곳 |

## 로컬 핵심 명령어

| 명령어 | 역할 |
|---|---|
| `git init` | 폴더를 Git 관리 대상으로 시작 |
| `git status` | 현재 상태 확인 (가장 자주 씀) |
| `git add .` | 변경사항을 스테이지에 올리기 |
| `git commit -m "메시지"` | 스테이지 내용을 공식 기록으로 저장 |
| `git log --oneline` | 커밋 기록을 한 줄씩 조회 |

### 심화 — 커밋 해시는 왜 암호학적 해시인가

`git log`에 찍히는 길쭉한 영문/숫자 조합(예: `a3f5e21...`)은 단순한 일련번호가 아니라 **SHA-1 해시값**이다. 커밋의 내용, 작성자, 시간, 부모 커밋 정보를 합쳐서 해시 함수에 넣어 만들어낸다. 내용이 단 한 글자라도 바뀌면 해시값이 완전히 달라지기 때문에, 이 해시는 곧 "이 커밋이 절대 변조되지 않았다"는 증명서 역할을 한다. Git이 분산 환경에서도 신뢰성을 유지할 수 있는 이유가 바로 이 구조 때문이다.

## GitHub 연동 — 원격 저장소

```bash
git remote add origin https://github.com/사용자명/레포이름.git
git remote -v                  # 등록된 원격 저장소 확인

git push -u origin main        # 처음엔 -u로 업스트림 등록
git push                       # 이후엔 이것만

git pull                       # GitHub 최신 내용 받아오기
git clone https://github.com/사용자명/레포이름.git   # 통째로 복사
```

> `-u (--set-upstream)`는 "앞으로 push/pull 기본 대상은 origin의 main"이라고 등록하는 옵션. 한 번만 설정하면 이후엔 `git push`만 입력해도 됨.

### 심화 — origin은 그냥 별명이다

`origin`이라는 이름은 Git이 강제하는 고정값이 아니라 관례다. 원격 저장소를 등록할 때 어떤 이름이든 붙일 수 있지만(`upstream`, `myrepo` 등), 거의 모든 협업 환경에서 `origin`을 표준으로 쓰기 때문에 그대로 따르는 게 안전하다. 오픈소스에 기여할 때는 보통 원본 저장소를 `upstream`, 내가 fork한 저장소를 `origin`으로 구분해서 쓴다.

## .gitignore — 절대 올리면 안 되는 것들

| 파일/폴더 | 이유 |
|---|---|
| `.env` | API 키, DB 비밀번호 등 민감 정보 |
| `node_modules/` | 용량이 너무 큼. package.json으로 복원 가능 |
| `__pycache__/`, `venv/` | Python 캐시 / 가상환경. requirements.txt로 복원 가능 |
| `.DS_Store` | Mac 시스템 파일 |
| `*.log` | 빌드 시 자동 생성되는 로그 |

> gitignore.io에서 기술 스택을 입력하면 자동 생성된 `.gitignore` 내용을 바로 복사할 수 있음.

## 브랜치 — 독립 작업 공간

```bash
git branch                  # 브랜치 목록
git switch -c 브랜치이름     # 새로 만들고 이동 (최신 방식)
git switch main             # main으로 돌아가기
git merge 브랜치이름         # main에서 실행, 브랜치 합치기
```

| 브랜치 이름 관례 | 용도 |
|---|---|
| `main` | 배포 가능한 안정 코드 |
| `feature/기능이름` | 새 기능 개발 |
| `fix/버그이름` | 버그 수정 |

### 심화 — merge와 rebase의 차이 (다음 단계 예고)

지금은 `merge`만 다뤘지만, 협업이 늘어나면 `rebase`도 마주치게 된다. `merge`는 두 브랜치의 기록을 그대로 합쳐서 "병합 커밋"을 새로 만드는 방식이고, `rebase`는 내 브랜치의 커밋들을 마치 처음부터 최신 main 위에서 작업한 것처럼 다시 쌓아 올리는 방식이다. merge는 기록이 정직하게 남고, rebase는 기록이 깔끔하게 정리된다. 협업 규칙에 따라 어떤 걸 쓸지가 달라지므로, 지금은 "둘 다 브랜치를 합치는 방법이지만 기록 모양이 다르다" 정도만 기억해두면 충분하다.

# Linux 기본 명령어

## 핵심 명령어 7개

| 명령어 | 의미 | 역할 |
|---|---|---|
| `pwd` | Print Working Directory | 현재 위치 확인 |
| `ls -al` | List | 폴더 내용 보기 (숨김파일 + 상세정보) |
| `cd 폴더명` | Change Directory | 폴더 이동 |
| `mkdir -p 경로` | Make Directory | 폴더 생성 (중간 경로까지 한번에) |
| `touch 파일명` | - | 빈 파일 생성 |
| `cat 파일명` | Concatenate | 파일 내용 출력 |
| `rm -rf 폴더명` | Remove | 강제 삭제 (복구 불가) |

> `rm -rf /` (루트 전체 삭제)는 절대 입력 금지. 삭제 전엔 항상 `pwd`로 위치를 확인하는 습관이 안전하다.

## 절대 경로 vs 상대 경로

- 절대 경로: `/Users/heejin/Documents` — 루트(`/`)부터 시작하는 전체 주소
- 상대 경로: `Documents` — 현재 위치 기준 주소

### 심화 — 커널과 쉘의 관계

터미널에서 명령어를 치면 일어나는 일은 2단계다. 먼저 **쉘(Shell)** 이 입력한 텍스트를 해석한다. 쉘은 사용자와 운영체제 사이의 통역사 역할을 하는 프로그램이다 (bash, zsh 등). 쉘이 명령어를 해석하면, 실제 하드웨어 제어(파일 읽기/쓰기, 메모리 할당 등)는 **커널(Kernel)** 이 수행한다. 커널은 운영체제의 핵심 부분으로, 하드웨어와 직접 통신하는 가장 안쪽 계층이다. `ls`를 입력하면 → 쉘이 명령어를 파싱 → 커널에 "이 폴더의 파일 목록을 달라"고 시스템 콜을 보냄 → 커널이 결과를 돌려줌 → 쉘이 화면에 출력, 이런 흐름이다. 나중에 Docker를 배우면 이 커널 개념이 다시 등장한다(컨테이너는 커널을 공유하면서 격리된 환경을 만드는 기술이기 때문).

### 심화 — Linux를 알아야 하는 진짜 이유

지금은 "터미널 명령어 몇 개"로 느껴지지만, Linux 명령어는 서버, Docker, 클라우드(AWS/GCP) 전부의 공통 언어다. 실제 배포 환경(서버)은 거의 대부분 Linux 기반이고, Docker 컨테이너 내부도 Linux 환경이다. 지금 익히는 `cd`, `ls`, `mkdir` 같은 명령어가 나중에 서버에 접속해서 로그를 확인하거나, Docker 컨테이너 안에 들어가서 디버깅할 때 그대로 쓰인다.

# HTML / CSS

## HTML 뼈대 템플릿

```html
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>페이지 제목</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>
  <header><nav>...</nav></header>
  <main><section>...</section></main>
  <footer>...</footer>
</body>
</html>
```

## 핵심 태그 분류

| 분류 | 태그 |
|---|---|
| 시맨틱 (의미) | `<header>` `<nav>` `<main>` `<section>` `<article>` `<aside>` `<footer>` |
| 텍스트 | `<h1>~<h6>` `<p>` `<strong>` `<span>` |
| 링크/이미지 | `<a href="">` `<img src="" alt="">` |
| 목록 | `<ul>` `<ol>` `<li>` |
| 테이블 | `<table>` `<tr>` `<td>` |
| 폼 | `<form>` `<input>` `<button>` `<textarea>` |
| 레이아웃 | `<div>` (의미 없는 컨테이너) |

### 심화 — div보다 시맨틱 태그를 우선해야 하는 진짜 이유

`<div>`로도 레이아웃은 똑같이 만들 수 있다. 차이는 **의미 전달**이다. 검색 엔진(SEO)은 `<article>`, `<nav>` 같은 태그를 보고 페이지 구조의 중요도를 파악한다. 스크린 리더(시각장애인용 도구)도 시맨틱 태그를 기준으로 "지금 내비게이션 영역입니다", "본문 콘텐츠입니다"라고 안내한다. `<div>`만 쓰면 이 정보가 전부 사라지기 때문에, 접근성과 SEO 모두에서 손해를 본다.

## CSS 단위

| 단위 | 기준 | 주 사용처 |
|---|---|---|
| `px` | 고정 픽셀 | border, 작은 요소 |
| `%` | 부모 요소 | width, height |
| `em` | 현재 요소의 font-size | padding, margin |
| `rem` | 루트(html)의 font-size | font-size (실무 표준) |
| `vw` / `vh` | 화면 크기 | 전체 화면 섹션 |

### 심화 — em이 헷갈리는 이유, rem을 쓰는 이유

`em`은 **부모 요소의 font-size를 기준**으로 계산되기 때문에, 중첩된 요소에서는 계산이 누적된다. 부모가 1.5em, 그 안의 자식도 1.5em이면 실제로는 1.5 × 1.5 = 2.25배가 된다. 이 누적 효과 때문에 깊은 중첩 구조에서 의도치 않게 글자가 커지는 버그가 자주 발생한다. `rem`은 항상 **루트(html) 요소의 font-size만 기준**으로 삼기 때문에 중첩 깊이와 상관없이 일관된 크기를 보장한다. 그래서 실무에서는 font-size에 거의 항상 `rem`을 쓴다.

## 박스 모델

```
margin > border > padding > content
```

```css
* { box-sizing: border-box; }  /* width에 padding, border 포함 — 실무 표준 */
```

## position

| 값 | 기준 | 특징 |
|---|---|---|
| `static` | 일반 흐름 | 기본값 |
| `relative` | 자기 원래 위치 | 주변에 영향 없음 |
| `absolute` | 가장 가까운 relative 부모 | 일반 흐름에서 이탈 |
| `fixed` | 화면(뷰포트) | 스크롤해도 고정 |
| `sticky` | 스크롤 위치 | 특정 지점에서 고정 |

> `absolute`는 부모에 `position: relative`가 없으면 `<html>` 기준으로 배치된다. 그래서 absolute를 쓸 땐 항상 부모에 relative를 먼저 설정하는 게 정석이다.

## Flexbox vs Grid

| 비교 | Flexbox | Grid |
|---|---|---|
| 차원 | 1차원 (가로 or 세로) | 2차원 (가로 + 세로 동시) |
| 주 사용처 | 네비게이션, 카드 나열 | 전체 레이아웃, 갤러리 |
| 기준 | 자식 요소 크기 | 격자(칸) |

```css
/* Flexbox — 완벽한 가운데 정렬 패턴 */
.center {
  display: flex;
  justify-content: center;
  align-items: center;
}

/* Grid — 반응형 자동 조절 패턴 */
.container {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  gap: 20px;
}
```

### 심화 — CSS 변수와 다크모드의 연결고리

CSS 변수(`--primary-color`)는 단순한 단축키가 아니라 **테마 전환의 기반 기술**이다. `:root`에 변수를 선언해두고, `body.dark` 같은 클래스가 붙었을 때 변수값만 바꿔치기하면 페이지 전체의 색상이 한 번에 바뀐다. 실제로 만든 다크모드 토글 기능도 이 원리의 단순화된 버전이다(클래스를 토글하고, CSS에서 그 클래스에 해당하는 색상을 정의). Tailwind CSS, styled-components 같은 최신 도구들도 결국 이 CSS 변수 개념을 기반으로 테마 시스템을 구현한다.

### 심화 — 이벤트 루프, JS가 싱글 스레드인데 어떻게 동시에 일하는가

JS 엔진의 콜 스택(Call Stack)은 하나뿐이라 한 번에 한 가지 작업만 처리한다. 그런데 `setTimeout`이나 `scroll` 이벤트는 어떻게 "기다리는 동안" 다른 코드를 계속 실행할 수 있을까. 비밀은 JS 엔진 바깥에 있는 **브라우저의 Web API**가 비동기 작업(타이머, 네트워크 요청, 이벤트 감지)을 대신 처리해주고, 작업이 끝나면 콜백 함수를 **태스크 큐**에 넣어둔다는 데 있다. **이벤트 루프(Event Loop)** 는 콜 스택이 비어 있는지 계속 감시하다가, 비어 있으면 태스크 큐에서 콜백을 꺼내 콜 스택에 올린다. 즉 JS 코드 자체는 싱글 스레드지만, 브라우저가 멀티스레드로 비동기 작업을 보조해주는 구조다.

## 비동기 — Promise와 async/await

```js
// Promise
const p = new Promise((resolve, reject) => {
  if (ok) resolve("성공!");
  else reject("실패");
});
p.then(r => console.log(r)).catch(e => console.log(e));

// async/await — 더 읽기 쉬움
async function getData() {
  try {
    const res = await fetch("https://api.example.com/data");
    const data = await res.json();
  } catch (err) {
    console.log("오류:", err);
  }
}
```

### 심화 — 콜백 지옥에서 async/await까지, 왜 이렇게 진화했나

JS 초기엔 비동기 처리를 전부 콜백 함수로 했다. 문제는 비동기 작업이 연속되면 콜백 안에 콜백이 중첩되어 코드가 오른쪽으로 계속 깊어지는 **콜백 지옥(Callback Hell)** 이 생긴다는 점이다. 이를 해결하기 위해 등장한 게 Promise — `.then()`으로 체이닝하면서 중첩 대신 수평으로 이어 쓸 수 있게 됐다. 하지만 `.then().then().then()`도 길어지면 가독성이 떨어진다. `async/await`는 이 Promise 체인을 **동기 코드처럼 위에서 아래로 읽히게** 만든 문법(문법적 설탕)이다. 기능은 Promise와 동일하지만, 에러 처리도 `try/catch`로 동기 코드와 똑같은 방식을 쓸 수 있어 가장 읽기 쉬운 방식으로 자리잡았다.
