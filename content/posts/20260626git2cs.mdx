## 1. 분산 시스템 아키텍처 — 원격 저장소

### 원격 저장소 (Remote Repository)

**정의**: 네트워크를 통해 접근 가능한 외부 서버에 위치한 저장소입니다.
로컬 저장소(Local Repository)와 대비되는 개념으로,
GitHub, GitLab, Bitbucket 같은 호스팅 서비스가 이에 해당합니다.

**CS적 의의**:

- **가용성(Availability)**: 원격 저장소는 로컬 장치 장애와 독립적으로 유지됩니다.
  로컬 컴퓨터가 고장나도 원격 저장소에 push된 커밋은 손실되지 않습니다.
- **동기화 모델**: `git push`와 `git pull`은 로컬과 원격 저장소 간의
  단방향 동기화(Unidirectional Sync) 연산입니다.
  양방향 실시간 동기화가 아니라 명시적 명령으로만 데이터가 이동합니다.

---

### 업스트림 (Upstream)

**정의**: 현재 로컬 브랜치가 추적(tracking)하도록 설정된 원격 브랜치를 가리킵니다.
`git push -u origin main`에서 `-u (--set-upstream)` 옵션이 이 관계를 등록합니다.

**CS적 의의**:

- **추적 참조(Tracking Reference)**: 업스트림이 설정되면 Git은 로컬 브랜치와
  원격 브랜치 사이의 커밋 차이(divergence)를 자동으로 계산할 수 있습니다.
  `git status`에서 "ahead 2, behind 1" 같은 정보가 이 추적 관계에서 나옵니다.
- **기본값 바인딩**: 업스트림 설정 후 `git push` / `git pull`만 입력해도
  명시적 대상 없이 동작하는 것은 이 바인딩 덕분입니다.

## 2. 소프트웨어 공학 — 브랜치 모델

### 브랜치 (Branch)

**정의**: Git 저장소 내에서 커밋 히스토리의 특정 지점에서 분기(diverge)하여
독립적인 개발 흐름을 만드는 가변 포인터(Mutable Pointer)입니다.
내부적으로는 특정 커밋 객체의 SHA 해시값을 가리키는 40바이트 참조 파일입니다.

**CS적 의의**:

- **비용**: 브랜치 생성은 단순히 참조 파일(40바이트) 하나를 생성하는 연산이므로
  O(1) 시간 복잡도를 가집니다. SVN 같은 구형 도구는 브랜치 생성 시 전체 디렉토리를
  복사해 O(n) 비용이 발생했던 것과 대조됩니다.
- **HEAD 포인터**: Git은 현재 작업 중인 브랜치를 `HEAD`라는 특수 포인터로 추적합니다.
  `git checkout 브랜치이름`은 HEAD가 가리키는 대상을 변경하는 연산입니다.

---

### 머지 (Merge)

**정의**: 두 개 이상의 브랜치 히스토리를 하나로 통합하는 연산입니다.
Git은 공통 조상 커밋(Common Ancestor)을 기준으로 3-way merge 알고리즘을 사용합니다.

**CS적 의의**:

- **3-way Merge**: 공통 조상(Base), 브랜치 A, 브랜치 B — 세 스냅샷을 비교하여
  충돌 없이 병합 가능한 변경사항을 자동으로 통합합니다.
- **Fast-forward Merge**: 한 브랜치가 다른 브랜치의 직접적인 후손일 경우,
  새로운 커밋 없이 포인터만 이동시키는 최적화된 머지입니다.

---

### 충돌 (Conflict)

**정의**: 두 브랜치에서 동일한 파일의 동일한 라인이 서로 다르게 수정되어
Git이 자동으로 병합을 결정할 수 없는 상태입니다.

**CS적 의의**: 충돌은 3-way merge 알고리즘이 해결 불가능한 모호성(Ambiguity)을
발견했을 때 발생합니다. 이 경우 Git은 자동 결정을 포기하고
충돌 마커(`<<<<<<<`, `=======`, `>>>>>>>`)를 파일에 삽입하여
사람이 직접 판단하도록 위임합니다.

## 3. 보안 — 환경변수와 시크릿 관리

### 환경변수 (Environment Variable)

**정의**: 운영체제 수준에서 프로세스에 전달되는 키-값 쌍의 동적 변수입니다.
`.env` 파일에 저장하고 애플리케이션 실행 시 로드하는 방식이 관례입니다.

**CS적 의의**:

- **관심사 분리(Separation of Concerns)**: 코드(로직)와 설정(환경별 비밀값)을
  분리하는 12-Factor App 방법론의 핵심 원칙입니다.
  같은 코드베이스가 개발/스테이징/프로덕션 환경에서 서로 다른 값으로 동작할 수 있게 합니다.
- **시크릿 유출 방지**: API 키, DB 비밀번호 등 민감한 값을 코드에 하드코딩하면
  GitHub에 노출될 위험이 있습니다. `.gitignore`로 `.env` 파일을 추적 제외하는 것은
  이 위험을 차단하는 기본적인 보안 조치입니다.

---

### 콘텐츠 주소 지정 저장소 (Content-addressable Storage)

**정의**: 데이터의 내용(Content)을 해시 함수에 입력하여 얻은 값을
저장 주소(Key)로 사용하는 저장 방식입니다.

**CS적 의의**: `.gitignore`로 추적 제외된 파일은 Git의 오브젝트 저장소에
아예 등록되지 않습니다. Git 오브젝트 저장소는 내용 기반 주소 지정 방식이므로
한 번 저장된 오브젝트는 내용이 변경될 수 없습니다 (불변성, Immutability).
이 덕분에 Git 히스토리는 변조가 불가능합니다.

## Vocabulary

| 용어 | 원어 | 정의 |
|---|---|---|
| Remote Repository | 원격 저장소 | 네트워크를 통해 접근하는 외부 서버의 저장소. GitHub가 대표적 |
| Upstream | 업스트림 | 로컬 브랜치가 추적하도록 설정된 원격 브랜치 |
| HEAD | HEAD | 현재 작업 중인 브랜치(또는 커밋)를 가리키는 Git의 특수 포인터 |
| Tracking Reference | 추적 참조 | 로컬 브랜치와 원격 브랜치 사이의 추적 관계. git push -u로 설정 |
| 3-way Merge | 3방향 병합 | 공통 조상, 브랜치 A, 브랜치 B 세 스냅샷을 비교해 병합하는 알고리즘 |
| Fast-forward | 빨리감기 병합 | 분기 없이 포인터만 앞으로 이동시키는 최적화된 머지 방식 |
| Conflict | 충돌 | 동일 파일의 동일 라인이 두 브랜치에서 다르게 수정되어 자동 병합 불가한 상태 |
| Environment Variable | 환경변수 | OS 수준에서 프로세스에 전달되는 키-값 쌍. .env 파일에 저장 |
| Separation of Concerns | 관심사 분리 | 코드와 설정(환경변수)을 분리하는 소프트웨어 설계 원칙 |
| Content-addressable Storage | 콘텐츠 주소 지정 저장소 | 데이터 내용의 해시값을 저장 주소로 사용하는 방식. Git 오브젝트 저장소가 채택 |
| Mutable Pointer | 가변 포인터 | 가리키는 대상이 바뀔 수 있는 참조. Git 브랜치가 이에 해당 |
| Divergence | 분기 | 공통 조상 이후 두 브랜치가 서로 다른 커밋을 쌓아 갈라지는 상태 |

## Key Questions

**Q1.** `git push -u origin main`에서 `-u` 옵션이 없을 때와 있을 때의
차이를 CS적으로 설명하면?

> **A.** `-u` 없이 push하면 데이터는 전송되지만 로컬 브랜치와 원격 브랜치 사이에
> 추적 참조(Tracking Reference)가 설정되지 않습니다.
> `-u`를 사용하면 Git이 `.git/config` 파일에 upstream 바인딩을 기록하여
> 이후 `git push` / `git pull` 입력 시 대상을 명시하지 않아도
> 자동으로 `origin/main`을 참조합니다.

---

**Q2.** Git 브랜치 생성이 SVN 브랜치 생성보다 빠른 CS적 이유는?

> **A.** Git 브랜치는 특정 커밋의 SHA 해시값(40바이트)을 담은 참조 파일 하나를
> 생성하는 O(1) 연산입니다. 반면 SVN은 브랜치 생성 시 해당 디렉토리 전체를
> 서버에 복사하는 O(n) 연산을 수행합니다 (n = 파일 수).
> Git의 스냅샷 기반 + 콘텐츠 주소 지정 저장 방식 덕분에 가능한 설계입니다.

---

**Q3.** `.env` 파일을 실수로 GitHub에 push했을 때 단순히 삭제 커밋으로는
완전히 제거되지 않는 이유는?

> **A.** Git은 콘텐츠 주소 지정 저장소이므로 한 번 저장된 오브젝트는
> 히스토리에 영구적으로 남습니다. 삭제 커밋은 최신 스냅샷에서 파일을 제거할 뿐,
> 이전 커밋 오브젝트는 그대로 존재합니다.
> 완전 제거를 위해서는 `git filter-branch` 또는 `git filter-repo`로
> 히스토리 전체를 재작성해야 하며, GitHub에도 force push가 필요합니다.
> 유출된 API 키는 재작성 후에도 반드시 즉시 폐기(revoke)해야 합니다.
