## 지금까지 배운 것과 무엇이 다른가

`@staticmethod`, `@classmethod`, `@property`는 Python이 미리 만들어둔 데코레이터를 가져다 쓴 것이었다. 오늘은 그 `@` 문법 자체를 직접 만들어본다. 이걸 이해하고 나면, FastAPI에서 계속 보게 될 `@app.get("/users")` 같은 코드가 "마법"이 아니라 그냥 함수를 감싸는 함수라는 게 보인다.

## 함수도 값이다

Python은 함수를 변수처럼 다른 함수에 넘기거나, 함수 안에서 새 함수를 만들어 반환할 수 있다.

```python
def greet():
    return "안녕하세요"

my_func = greet     # 함수 자체를 변수에 저장 (호출 아님, 괄호 없음)
print(my_func())    # 안녕하세요

def outer():
    def inner():           # 함수 안에 함수를 정의할 수 있음
        return "내부 함수"
    return inner             # 함수를 결과값으로 반환

f = outer()
print(f())   # 내부 함수
```

실생활 비유: 함수가 "레시피 카드"라고 생각하면, 카드 자체를 다른 사람에게 건네주거나(변수에 저장), 새로운 카드를 즉석에서 만들어서 건네줄 수 있는(함수 반환) 것과 같다. 데코레이터는 바로 이 성질을 이용한다.

## 데코레이터의 기본 구조

데코레이터는 "함수를 받아서, 그 함수를 감싼 새로운 함수를 반환하는 함수"다.

```python
def my_decorator(func):        # 감쌀 대상 함수를 인자로 받음
    def wrapper():                 # func를 감싸는 새 함수
        print("함수 실행 전")
        result = func()             # 원래 함수 실행
        print("함수 실행 후")
        return result
    return wrapper                  # 감싼 함수를 반환

def say_hello():
    print("안녕하세요")

say_hello = my_decorator(say_hello)   # say_hello를 감싼 버전으로 교체
say_hello()
```

```text
함수 실행 전
안녕하세요
함수 실행 후
```

`@` 문법은 위 두 줄(`say_hello = my_decorator(say_hello)`)을 짧게 쓰는 문법일 뿐이다.

```python
@my_decorator          # def say_hello 바로 위에 붙임
def say_hello():
    print("안녕하세요")

say_hello()   # my_decorator(say_hello)()와 완전히 동일하게 동작
```

실생활 비유: 선물(원래 함수)을 포장지(데코레이터)로 감싸는 것과 같다. 포장을 열어보면(호출하면) 포장지가 하는 일(실행 전후 메시지)이 먼저 벌어지고, 그 안의 원래 선물(원래 함수의 실행 결과)이 나온다. 선물 내용 자체는 건드리지 않고 겉모습만 감싼다.

## *args, **kwargs — 원래 함수의 인자를 그대로 전달하기

위 예시의 `wrapper()`는 인자가 없는 함수만 감쌀 수 있다. 인자가 있는 함수도 감싸려면 `*args`, `**kwargs`로 "어떤 인자가 오든 그대로 받아서 그대로 전달"해야 한다.

```python
def my_decorator(func):
    def wrapper(*args, **kwargs):     # 어떤 위치 인자, 키워드 인자든 다 받음
        print(f"{func.__name__} 호출, 인자: {args}, {kwargs}")
        return func(*args, **kwargs)    # 받은 그대로 원래 함수에 전달
    return wrapper

@my_decorator
def add(a, b):
    return a + b

print(add(3, 5))   # add 호출, 인자: (3, 5), {} \n 8
```

> 함정: `wrapper()`에 `*args, **kwargs`를 빼먹으면, 인자가 있는 함수를 감쌌을 때 `TypeError: wrapper() takes 0 positional arguments but 2 were given` 에러가 난다. 데코레이터는 "어떤 함수에든 재사용 가능해야" 의미가 있으므로 거의 항상 `*args, **kwargs`를 붙인다.

## functools.wraps — 원래 함수의 정체성 보존하기

데코레이터를 씌우면 `wrapper`라는 새 함수로 바뀌기 때문에, 원래 함수의 이름과 문서(docstring) 같은 메타데이터가 사라진다.

```python
@my_decorator
def add(a, b):
    """두 수를 더한다"""
    return a + b

print(add.__name__)   # wrapper — 원래 이름 add가 사라짐!
```

```python
from functools import wraps

def my_decorator(func):
    @wraps(func)              # func의 이름, 문서 등을 wrapper에 그대로 복사해줌
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@my_decorator
def add(a, b):
    """두 수를 더한다"""
    return a + b

print(add.__name__)   # add — 원래 이름이 유지됨
```

**실무에서는 이렇게 쓴다**: 데코레이터를 만들 때 `@wraps(func)`를 빼먹지 않는 것이 관례다. 안 붙이면 디버깅 도구나 자동 문서 생성기(FastAPI가 API 문서를 자동 생성하는 것도 이 원리에 의존한다)가 실제 함수 정보를 제대로 읽지 못한다.

## 인자를 받는 데코레이터

`@app.get("/users")`처럼 데코레이터 자체에 괄호와 인자가 붙은 경우, 함수를 한 겹 더 감싸야 한다.

```python
def repeat(times):              # 데코레이터에 전달할 인자를 받는 바깥 함수
    def decorator(func):           # 실제 데코레이터
        @wraps(func)
        def wrapper(*args, **kwargs):
            for _ in range(times):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return decorator

@repeat(times=3)     # repeat(3)이 먼저 실행되어 decorator를 반환하고, 그 decorator가 say_hi를 감쌈
def say_hi():
    print("안녕!")

say_hi()   # 안녕! 이 3번 출력됨
```

함수가 세 겹으로 중첩되는 이유: `repeat(times=3)`이 실행되어 `decorator` 함수를 만들어내고, 그 `decorator`가 `say_hi`를 인자로 받아 진짜 데코레이터 역할을 한다. `@app.get("/users")`도 똑같은 구조로, `get("/users")`가 먼저 실행되어 실제 데코레이터를 만들어낸다.

## JS와 비교

| Python | JavaScript |
|---|---|
| `@my_decorator` (안정적인 정식 문법, 오래전부터 지원) | 데코레이터 문법이 아직 실험 단계(Stage 3 제안, 표준 미확정) |
| 함수/클래스 모두에 자유롭게 사용 | 주로 클래스와 클래스 메서드에 한정해서 논의됨 |

Python의 데코레이터는 언어 초기부터 안정적으로 자리 잡은 문법이라, 함수든 클래스든 자유롭게 감쌀 수 있다. JS는 아직 표준화가 진행 중이라 실무에서 데코레이터 문법 자체를 보기는 상대적으로 드물다.

## 실무에서는 이렇게 쓴다

FastAPI의 `@app.get("/users")`, `@app.post("/users")`는 정확히 오늘 배운 "인자를 받는 데코레이터" 구조다. 그 함수를 특정 URL 경로와 HTTP 메서드에 연결(등록)해주는 역할을 한다. 로그인 여부를 검사하는 `@login_required`, 실행 시간을 재는 `@timer`처럼, "여러 함수에 공통으로 끼워 넣고 싶은 부가 기능"을 정의할 때 데코레이터를 쓰는 것이 표준 패턴이다.

## 함정: 데코레이터를 여러 개 겹쳐 쓸 때 순서

```python
@decorator_a
@decorator_b
def func():
    pass

# 실행 순서: decorator_a(decorator_b(func))
# 즉 func에 가장 가까운(아래쪽) 데코레이터가 먼저 적용된다
```

함수 바로 위에 있는 데코레이터부터 안쪽에서 바깥쪽 순서로 감싸진다. 여러 데코레이터를 쓸 때는 이 적용 순서가 결과에 영향을 줄 수 있으므로 주의해야 한다.

## 요약

- 데코레이터는 함수를 인자로 받아, 그 함수를 감싼 새로운 함수를 반환하는 함수다. `@decorator` 문법은 `func = decorator(func)`를 짧게 쓴 것이다
- `wrapper(*args, **kwargs)`로 원래 함수의 인자를 그대로 전달받아 넘겨야, 어떤 함수에든 재사용 가능한 데코레이터가 된다
- 함정: `*args, **kwargs`를 빼먹으면 인자가 있는 함수를 감쌀 때 `TypeError`가 난다
- `@functools.wraps(func)`를 붙이지 않으면 원래 함수의 이름과 문서가 `wrapper`의 것으로 덮어써진다
- `@repeat(times=3)`처럼 인자를 받는 데코레이터는 함수를 한 겹 더 감싼 구조(3중 중첩)로 만든다
- 데코레이터를 여러 개 겹치면, 함수에 가장 가까운(아래쪽) 데코레이터부터 안쪽에서 바깥쪽 순서로 적용된다
- JS는 데코레이터 문법이 아직 표준화 진행 중이지만, Python은 오래전부터 안정적으로 지원한다
- FastAPI의 `@app.get("/users")`도 오늘 배운 "인자를 받는 데코레이터" 구조 그대로다
