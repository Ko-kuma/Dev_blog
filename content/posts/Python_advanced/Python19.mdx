## 지금까지 배운 것과 무엇이 다른가

제너레이터에서 "함수가 실행을 멈췄다가 나중에 이어서 실행할 수 있다"는 걸 배웠다. 오늘 배울 비동기 프로그래밍은 정확히 그 성질을 이용해서, "기다리는 동안 다른 일을 하다가, 기다리던 게 끝나면 돌아와서 이어 하는" 프로그램을 만든다. 심화 단원의 마지막 주제이자, FastAPI가 `async def`로 빠른 API 서버를 만들 수 있는 핵심 원리다.

## 동기 코드의 문제: 기다리는 동안 아무것도 못 한다

```python
import time

def fetch_data(name, seconds):
    print(f"{name} 요청 시작")
    time.sleep(seconds)   # 네트워크 요청을 흉내낸 대기 (실제로는 API 호출 등)
    print(f"{name} 완료")
    return f"{name} 결과"

start = time.time()
fetch_data("A", 2)   # 2초 대기
fetch_data("B", 2)   # 또 2초 대기
print(f"총 소요 시간: {time.time() - start:.1f}초")   # 약 4초
```

`A`를 기다리는 2초 동안 프로그램은 아무것도 하지 않고 그냥 멈춰(block) 있는다. `B`도 마찬가지라, 둘을 순서대로 처리하면 대기 시간이 그대로 더해진다. 실제로는 CPU가 계산하는 시간이 아니라 네트워크 응답을 "기다리는" 시간이 대부분이라, 이 기다림을 겹쳐서 처리할 수 있다면 훨씬 빨라질 것이다.

## async와 await — 기본 문법

```python
import asyncio

async def fetch_data(name, seconds):        # def 앞에 async를 붙이면 코루틴 함수가 됨
    print(f"{name} 요청 시작")
    await asyncio.sleep(seconds)              # await: 여기서 기다리는 동안 다른 작업에 순서를 양보함
    print(f"{name} 완료")
    return f"{name} 결과"

async def main():
    start = asyncio.get_event_loop().time()
    await asyncio.gather(                      # 여러 코루틴을 동시에 실행하고 전부 끝날 때까지 기다림
        fetch_data("A", 2),
        fetch_data("B", 2),
    )
    print("총 소요 시간: 약 2초")   # 동시에 진행되어 4초가 아니라 2초 정도만 걸림

asyncio.run(main())   # 비동기 프로그램의 진입점 — main()이라는 코루틴을 실제로 실행시킴
```

`async def`로 정의된 함수를 코루틴 함수라고 부른다. `await`가 붙은 지점에서 "이 작업이 끝날 때까지 여기서 기다리되, 기다리는 동안에는 다른 코루틴에게 실행 순서를 양보한다." `A`가 2초 대기하는 동안 CPU는 놀지 않고 `B`의 코드를 실행하기 시작하기 때문에, 둘을 합쳐도 총 시간은 4초가 아니라 2초에 가깝다.

실생활 비유: 식당 종업원이 한 테이블 주문을 넣고 음식이 나올 때까지 그 자리에 멍하니 서서 기다리는 게 동기 방식이다. 비동기 방식은, 주문을 넣어두고(`await` 지점) 음식이 준비되는 동안 다른 테이블 주문을 받으러 가는 것과 같다. 종업원(스레드)은 한 명뿐이지만, "기다리는 시간"을 낭비하지 않고 여러 테이블을 돌본다.

## 코루틴과 제너레이터의 관계

```python
async def say_hello():
    return "안녕하세요"

result = say_hello()   # 함수를 호출했는데...
print(result)          # <coroutine object say_hello at 0x7f...> — 아직 실행 안 됨!
```

> 함정: `async def`로 만든 함수는 그냥 호출하면 실행되지 않고, "실행할 준비가 된 코루틴 객체"만 돌려준다. 제너레이터를 호출하면 바로 실행되지 않고 제너레이터 객체만 만들어지던 것(Python16)과 완전히 같은 원리다. 실제로 실행시키려면 `await`를 붙이거나 `asyncio.run()`으로 실행해야 한다.

```python
async def main():
    result = await say_hello()   # await를 붙여야 실제로 실행되고 결과를 받음
    print(result)                  # 안녕하세요

asyncio.run(main())
```

코루틴은 내부적으로 제너레이터와 매우 비슷한 메커니즘(실행을 멈추고 상태를 기억했다가 재개하는 것)으로 만들어졌다. 제너레이터를 이해해두면 코루틴이 "왜 이렇게 동작하는지" 훨씬 쉽게 받아들여진다.

## 이건 무료로 빨라지는 마법이 아니다

> 함정: 비동기는 "기다리는 시간"(네트워크 응답, 파일 입출력 같은 I/O 작업)을 효율적으로 겹쳐 쓰는 것이지, CPU가 실제로 계산하는 시간을 줄여주는 게 아니다. 복잡한 수학 계산처럼 CPU를 계속 바쁘게 쓰는 작업(CPU bound)은 `async`로 감싸도 전혀 빨라지지 않는다. `async`/`await`는 API 호출, DB 쿼리, 파일 읽기처럼 "기다리는 시간이 많은"(I/O bound) 작업에만 효과가 있다.

## JS와 비교

이 주제는 JS와 문법이 거의 동일하다.

| Python | JavaScript |
|---|---|
| `async def func():` | `async function func() {}` |
| `await coroutine`  | `await promise` |
| `asyncio.gather(a, b)` | `Promise.all([a, b])` |
| `asyncio.run(main())` | 최상위 `await` 또는 즉시 실행 async 함수 |

`async`/`await` 키워드와 "기다리는 동안 다른 작업으로 넘어간다"는 동작 원리 자체가 JS와 사실상 같다. JS를 다뤄봤다면 Python의 비동기 문법은 이름과 세부 API(`Promise.all` ↔ `asyncio.gather`)만 다를 뿐, 개념적으로 거의 그대로 옮겨온다고 보면 된다.

## 실무에서는 이렇게 쓴다: FastAPI가 빠른 이유

```python
from fastapi import FastAPI
import httpx

app = FastAPI()

@app.get("/users/{user_id}")
async def get_user(user_id: int):             # async def로 정의된 엔드포인트
    async with httpx.AsyncClient() as client:
        response = await client.get(f"https://api.example.com/users/{user_id}")
        return response.json()
```

FastAPI 엔드포인트를 `async def`로 만들면, 하나의 요청이 외부 API나 DB 응답을 기다리는 동안 서버는 그 시간 동안 다른 사용자의 요청을 처리할 수 있다. 동기 방식으로 짰다면, 한 요청이 응답을 기다리는 동안 서버는 다른 요청을 전혀 처리하지 못하고 멈춰 있었을 것이다. 이 차이가 FastAPI가 동시에 많은 요청을 효율적으로 처리할 수 있는 핵심 이유다.

## 요약

- 동기 코드는 I/O(네트워크, 파일 등)를 기다리는 동안 아무 일도 하지 못하고 멈춰(block) 있는다
- `async def`로 코루틴 함수를 만들고, `await`로 "기다리되 그동안 다른 작업에 순서를 양보하는" 지점을 표시한다
- `asyncio.gather(a, b)`로 여러 코루틴을 동시에 실행하면, 순차 실행보다 훨씬 빠르게 끝난다(대기 시간이 겹쳐지므로)
- 함정: `async def` 함수를 `await` 없이 호출하면 실행되지 않고 코루틴 객체만 반환된다 — 제너레이터를 호출만 하고 순회하지 않은 것과 같은 상태다
- 코루틴은 제너레이터와 같은 "멈췄다 재개하는" 메커니즘 위에 세워져 있다
- 함정: 비동기는 I/O 대기 시간을 효율적으로 겹쳐 쓸 뿐, CPU 연산 자체를 빠르게 하지 않는다. CPU를 많이 쓰는 작업에는 효과가 없다
- JS의 `async`/`await`/`Promise.all`과 문법·개념이 거의 동일하다
- FastAPI는 `async def` 엔드포인트로, 한 요청이 응답을 기다리는 동안 다른 요청을 함께 처리할 수 있어 빠르다
