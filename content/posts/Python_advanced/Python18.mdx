## 지금까지 배운 것과 무엇이 다른가

지금까지 `name = "초코"`, `def bark(self):`처럼 변수나 매개변수에 타입을 따로 적은 적이 없다. Python은 값이 실제로 무슨 타입인지 실행하는 순간에야 알 수 있는 동적 타입 언어이기 때문이다. 오늘은 "이 변수는 이런 타입일 것이다"라고 미리 표시해두는 타입 힌트 문법을 배운다. 이건 FastAPI/Pydantic이 코드를 읽고 자동으로 검증·문서화를 해내는 핵심 원리이기도 하다.

## 기본 문법

```python
name: str = "초코"          # 변수 이름 뒤에 콜론(:)과 타입을 적음
age: int = 3

def bark(name: str, times: int) -> str:   # 매개변수마다 타입, 화살표(->) 뒤에 반환 타입
    return f"{name}: " + "멍! " * times

print(bark("초코", 2))   # 초코: 멍! 멍!
```

`name: str`은 "이 변수에는 문자열이 들어올 것이다"라는 힌트이고, `-> str`은 "이 함수는 문자열을 반환한다"는 힌트다.

실생활 비유: 택배 상자에 붙이는 "이 안에는 유리 제품이 들어있습니다" 스티커와 같다. 스티커가 상자를 못 열게 막아주지는 않는다 — 누구든 열어서 다른 걸 넣을 수 있다. 하지만 스티커 덕분에 배송 담당자와 받는 사람 모두 "이 안에 뭐가 들어있어야 하는지" 미리 알 수 있다.

## 타입 힌트는 강제되지 않는다

```python
def add(a: int, b: int) -> int:
    return a + b

print(add("3", "5"))   # "35" — 문자열을 넣어도 에러 없이 그냥 실행됨!
```

> 함정: 타입 힌트는 Python 실행기(인터프리터)가 직접 검사하지 않는다. `int`라고 적어놓고 문자열을 넘겨도 프로그램은 에러 없이 그대로 실행된다. 타입 힌트는 어디까지나 "힌트"일 뿐이며, 실제로 타입이 맞는지 검사하려면 `mypy` 같은 별도의 정적 타입 검사 도구를 돌리거나, Pydantic처럼 런타임에 직접 검증하는 라이브러리를 써야 한다.

**그럼 왜 쓰는가**: 검사는 안 해줘도, VSCode 같은 편집기가 자동완성과 오류 경고를 훨씬 정확하게 해준다. 그리고 코드를 읽는 사람(미래의 나 포함)이 함수 시그니처만 보고도 "여기 뭘 넣어야 하는지" 바로 알 수 있어 문서 역할을 한다.

## 여러 타입 조합하기

```python
from typing import Optional, Union

def find_dog(dog_id: int) -> Optional[str]:   # str이거나 None일 수 있음
    dogs = {1: "초코", 2: "보리"}
    return dogs.get(dog_id)   # 없는 id면 None 반환

def parse_input(value: Union[int, str]) -> str:   # int 또는 str 둘 다 허용
    return str(value)

# Python 3.10 이상에서는 | 로 더 짧게 쓸 수 있다
def find_dog_new(dog_id: int) -> str | None:      # Optional[str]과 완전히 같은 의미
    ...
```

`Optional[str]`은 `Union[str, None]`을 줄여 쓴 것으로, "문자열이거나 아무것도 없을 수 있다(`None`)"는 뜻이다. 함수가 항상 값을 찾아 반환한다고 보장할 수 없을 때 이 힌트를 쓴다.

## 리스트, 딕셔너리 안의 타입까지 명시하기 (제네릭)

```python
def get_names(dogs: list[dict[str, int]]) -> list[str]:
    # dogs: "문자열 key와 int value를 가진 딕셔너리"들의 리스트
    return [dog["name"] for dog in dogs]
```

그냥 `list`라고만 적으면 "리스트다"까지만 알 수 있지만, `list[str]`처럼 대괄호로 안의 타입까지 적으면 "문자열이 담긴 리스트"라는 것까지 명시할 수 있다. (Python 3.9 이전 버전은 `list[str]` 대신 `typing.List[str]`을 써야 했지만, 지금은 내장 타입에 바로 대괄호를 쓸 수 있다.)

## JS(TypeScript)와 비교

Python의 타입 힌트는 JS보다는 TypeScript와 훨씬 가깝다.

| Python | TypeScript |
|---|---|
| `name: str` | `let name: string` |
| `def bark(name: str) -> str:` | `function bark(name: string): string` |
| `Optional[str]` / `str \| None` | `string \| null` |
| 실행 시점에는 검사 안 됨(주석 취급) | 컴파일 시점에 타입 검사, 컴파일 후 JS로 변환되며 타입 정보는 사라짐 |

둘 다 "타입을 미리 적어두고 도구로 검사한다"는 철학은 같다. 차이는 TypeScript는 컴파일이라는 단계에서 타입 오류를 강제로 걸러내지만, Python은 별도로 `mypy` 같은 도구를 돌리지 않는 한 타입이 틀려도 그냥 실행된다는 점이다.

## 실무에서는 이렇게 쓴다: Pydantic과 FastAPI

지금까지 배운 타입 힌트가 "강제되지 않는 힌트일 뿐"이라고 했는데, Pydantic은 이 타입 힌트를 런타임에 직접 읽어서 실제로 검증해주는 라이브러리다.

```python
from pydantic import BaseModel

class User(BaseModel):
    name: str
    age: int

User(name="혜진", age="스물여덟")   # ValidationError! Pydantic이 age가 int가 아님을 실제로 검사해서 막음
```

일반 함수의 타입 힌트는 그냥 무시되고 넘어가지만, Pydantic 모델의 타입 힌트는 클래스가 인스턴스로 만들어지는 순간 실제로 검사되어, 맞지 않으면 에러를 낸다. FastAPI는 이 Pydantic 모델을 요청/응답에 그대로 사용해서, 타입 힌트 하나로 데이터 검증과 API 문서 자동 생성을 동시에 해결한다.

## 요약

- `변수: 타입`, `def 함수(매개변수: 타입) -> 반환타입:` 형태로 타입 힌트를 붙인다
- 함정: 타입 힌트는 Python 실행기가 강제로 검사하지 않는다. 틀린 타입을 넣어도 에러 없이 실행된다 — 어디까지나 힌트이자 문서다
- `Optional[X]`(= `X | None`)는 값이 없을 수도 있음을, `Union[X, Y]`(= `X | Y`)는 여러 타입 중 하나일 수 있음을 나타낸다
- `list[str]`, `dict[str, int]`처럼 대괄호로 컨테이너 안의 타입까지 명시할 수 있다
- 타입 힌트는 강제하지 않는 대신, 편집기의 자동완성/오류 경고와 코드 문서화에 큰 도움을 준다. 실제 검사가 필요하면 `mypy` 같은 정적 타입 검사 도구를 쓴다
- TypeScript의 타입 표기와 철학이 비슷하지만, TS는 컴파일 시점에 강제 검사하는 반면 Python은 별도 도구 없이는 검사되지 않는다
- Pydantic은 타입 힌트를 런타임에 실제로 읽어 검증하는 라이브러리이며, FastAPI는 이를 이용해 요청 데이터 검증과 API 문서를 타입 힌트 하나로 자동화한다
