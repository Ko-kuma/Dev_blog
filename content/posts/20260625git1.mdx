##  왜 Git을 배우는가?

코드를 짜다 보면 반드시 이런 상황이 옵니다:

- "어제까지 잘 됐는데 오늘 고쳤더니 망가졌다"
- "예전 버전으로 되돌아가고 싶다"
- "이 코드 언제, 누가, 왜 바꿨지?"

Git은 **코드의 변경 기록을 저장하는 시스템**입니다.
쉽게 말하면 **코드용 타임머신**이에요.

나중에 배울 GitHub에 코드를 올리면 포트폴리오가 되고,
협업할 때도 Git 없이는 아무것도 못 합니다.
그래서 가장 먼저 배웁니다.

---

## 개념 — Git vs GitHub

가장 먼저 헷갈리는 것부터 정리합니다.

| | Git | GitHub |
|---|---|---|
|? | 내 컴퓨터에 설치하는 프로그램 | 코드를 올려두는 인터넷 창고 |
| 저장 | 내 컴퓨터 안 (로컬) | 인터넷 위 (클라우드) |
| 비유 | 일기장 | 일기장을 보관하는 클라우드 금고 |

> Git은 내 컴퓨터 안에서 동작합니다.
> GitHub는 그 결과물을 인터넷에 올려두는 곳이에요.
> 둘은 다른 개념인데 항상 같이 써서 헷갈리는 거예요.

---

## 비유

Git을 **요리 레시피 수정 기록**에 비유해볼게요.

```
레시피 v1: 계란 2개, 소금 약간
레시피 v2: 계란 3개로 수정 (더 맛있을 것 같아서)
레시피 v3: 역시 2개가 나았다 → v1으로 되돌리기
```

Git이 이 모든 수정 기록을 저장합니다.
언제든 이전 버전으로 돌아갈 수 있고, 누가 무엇을 바꿨는지도 다 남아요.

## Git의 3가지 공간 

Git을 쓸 때 코드는 세 곳을 거쳐 이동합니다.
이걸 모르면 명령어가 왜 이렇게 쓰이는지 이해가 안 돼요.

```
[작업 폴더]  →  git add  →  [스테이지]  →  git commit  →  [저장소]
(내가 편집중)              (저장 대기실)                  (기록 완료)
```

| 공간 | 이름 | 비유 |
|---|---|---|
| 작업 폴더 | Working Directory | 실제로 코드를 편집하는 곳 |
| 스테이지 | Staging Area | "이것들 저장할게요" 대기실 |
| 저장소 | Repository (repo) | 기록이 완료된 타임머신 본체 |

##  핵심 명령어

### 1. `git init` — 이 폴더를 Git이 관리하도록 시작

```bash
# 원하는 폴더로 이동한 뒤 
cd 원하는 폴더 명
git init
```

실행하면 폴더 안에 숨김 폴더 `.git`이 생깁니다.
이게 생겼다는 건 Git이 이 폴더를 감시하기 시작했다는 뜻이에요.

> `.git` 폴더를 직접 건드리거나 삭제하면 안 됩니다.
> 모든 버전 기록이 여기에 저장되어 있어요.

---

### 2. `git status` — 지금 상태 확인

```bash
git status
```

가장 자주 쓰게 될 명령어입니다.

- 어떤 파일이 변경됐는지
- 스테이지에 올라간 파일이 뭔지
- 아직 추적 안 된 파일이 뭔지

를 한눈에 보여줍니다.
뭔가 이상하면 `git status`부터 치는 게 습관이 됩니다.

---

### 3. `git add` — 스테이지에 올리기

```bash
# 특정 파일만 올리기
git add 파일명.txt

# 현재 폴더의 변경된 파일 전부 올리기
git add .
```

저장하고 싶은 파일을 "대기실"에 올리는 과정입니다.
`git add .` 의 `.` 은 "현재 폴더 전체"를 의미.

---

### 4. `git commit` — 기록 저장 (스냅샷 찍기)

```bash
git commit -m "첫 번째 커밋: README 파일 추가"
```

스테이지에 올린 파일들을 **공식 기록으로 저장**합니다.
`-m` 뒤의 텍스트는 **커밋 메시지** — "무엇을 왜 바꿨는지" 설명하는 메모입니다.

> 커밋 메시지는 나중에 내가 (또는 팀원이) 읽게 됩니다.
> "수정함", "업데이트" 같은 메시지는 쓸모가 없고 아래 내용처럼 구체적으로 작성.
> "로그인 버튼 색상 파란색으로 변경" 처럼 구체적으로 쓰는 게 좋습니다.

---

### 5. `git log` — 기록 조회

```bash
git log
```

지금까지 쌓인 커밋 기록 전체를 보여줍니다.
각 커밋마다 고유한 ID(해시값), 작성자, 날짜, 메시지가 표시돼요.

한 줄로 간결하게 보고 싶을 땐:

```bash
git log --oneline
```

---

## 전체 흐름 정리

```
1. git init          → 이 폴더를 Git 관리 시작
2. 파일 수정/생성
3. git status        → 현재 상태 확인
4. git add .         → 변경사항 스테이지에 올리기
5. git commit -m ""  → 공식 기록으로 저장
6. git log           → 기록 확인
```

---

## 실행해보기

```bash
# 1. 실습 폴더 만들고 이동
mkdir my-first-git
cd my-first-git

# 2. Git 시작
git init

# 3. 파일 하나 만들기
touch README.md

# 4. 상태 확인 (README.md가 "추적 안 된 파일"로 보임)
git status

# 5. 스테이지에 올리기
git add README.md

# 6. 다시 상태 확인 (이제 "커밋 준비 완료"로 바뀜)
git status

# 7. 커밋
git commit -m "첫 번째 커밋: README 파일 추가"

# 8. 기록 확인
git log --oneline
```

##커밋이 안되거나 이메일 설정하라는 경고문이 뜰 시 

### "커밋했는데 기록이 안 남아요"
→ `git add` 를 빠트린 경우가 99%입니다. 스테이지에 먼저 올려야 해요.

### "이름이랑 이메일 설정하라고 떠요"
처음 커밋할 때 Git이 누가 커밋했는지 기록하려고 물어봐요.

```bash
git config --global user.name "이름"
git config --global user.email "이메일@example.com"
```

한 번만 설정하면 이후로는 반복하지 않아도 됩니다.
