## 변수란?

변수는 **라벨이 붙은 서랍** 같은 것이다. 서랍에 "양말"이라고 라벨을 붙여두면 그 안에 어떤 양말을 넣든 꺼내든 상관없이 "양말 서랍"이라고 부를 수 있다. 

나중에 서랍 안 내용물을 바꿔도(양말 → 속옷) 서랍 자체의 이름(라벨)은 그대로 쓸 수 있는 것처럼, 변수도 이름은 그대로 두고 안에 담긴 값을 바꿀 수 있다.

```python
age = 25          # age라는 서랍에 25를 담음
name = "혜진"      # name이라는 서랍에 문자열을 담음
```

동적 타이핑(변수의 종류를 미리 정해두지 않아도 되는 방식)을 다시 떠올려보면, Python은 서랍에 "숫자만 넣는 서랍"처럼 미리 이름표를 정해두지 않아도 된다. 

아무 서랍에나 아무 값이나 넣을 수 있다.

JS와 비교하면:

| 구분 | JavaScript | Python |
|---|---|---|
| 선언 키워드 | let, const 필요 | 키워드 없이 바로 이름 = 값 |
| 재할당 가능 여부 | let은 가능, const는 불가 | 기본적으로 항상 재할당 가능 |
| 타입 고정 | 없음 (동적) | 없음 (동적) |

Python은 let/const 구분이 따로 없다. 

그냥 다 재할당 가능한 변수라고 보면 된다. (상수를 흉내내고 싶으면 관례상 대문자로 이름 짓지만, 문법적으로 강제되진 않는다. 자세한 건 바로 아래에서 다룬다.)

### 변수 이름을 지을 때 반드시 지켜야 할 규칙

네이밍 컨벤션(잠시 후에 배울 snake_case 등)이 "이렇게 짓는 게 좋다"는 **권장 사항**이라면, 이건 어기면 아예 에러가 나는 **강제 규칙**이다.

1. **숫자로 시작할 수 없다**: `1st_place = "금메달"` → `SyntaxError`. 반드시 문자나 언더스코어로 시작해야 한다. (`first_place`, `_place` 는 가능)
2. **예약어(Reserved Word)는 변수명으로 쓸 수 없다**: 예약어란 Python 문법 자체가 이미 특별한 의미로 사용하고 있는 단어다. 이걸 변수명으로 쓰면 문법이 충돌해서 에러가 난다.
3. **띄어쓰기나 대부분의 특수문자는 쓸 수 없다**: 언더스코어(`_`)만 허용된다. (`user name`, `user-name` 은 불가능, `user_name` 은 가능)
4. **대문자와 소문자를 다른 이름으로 구분한다**: `age`와 `Age`는 완전히 다른 변수다. (Case-sensitive라고 부른다)

**실생활 비유**: 예약어는 회사에서 이미 다른 용도로 예약된 회의실 이름(예: "1번 회의실")과 같다. 그 이름표가 붙은 방은 이미 정해진 용도가 있어서, 내가 새로 "내 물건 창고"라는 뜻으로 그 이름을 쓸 수 없는 것과 같다.

**Python의 예약어 목록** (총 35개, 전부 소문자 — 대문자로 바꾸면 예약어가 아니게 되지만 헷갈리니 피하는 게 좋다):

```
False   None    True    and     as      assert  async   await
break   class   continue def     del     elif    else    except
finally for     from    global  if      import  in      is
lambda  nonlocal not     or      pass    raise   return  try
while   with    yield
```

```python
# 예약어를 변수명으로 쓰려고 하면 에러 발생
class = "수학"     # SyntaxError: invalid syntax (class는 "클래스 정의용" 예약어)
return = 10        # SyntaxError (return은 "함수 결과 반환용" 예약어)

# 대신 이렇게 피해서 짓는다
subject = "수학"    # OK
result = 10         # OK
```

> 궁금하면 파이썬에서 직접 예약어 목록을 확인할 수도 있다.
> ```python
> import keyword
> print(keyword.kwlist)   # 현재 Python 버전의 예약어 전체 목록 출력
> ```

**실무에서는 이렇게 쓴다**: 실무에서 자주 걸리는 실수는 `list`, `dict`, `type`, `id` 처럼 예약어는 아니지만 Python이 기본으로 제공하는 함수/자료형 이름을 변수명으로 덮어써버리는 것이다. 예를 들어 `list = [1, 2, 3]`이라고 쓰면 문법 에러는 안 나지만, 그 순간부터 원래의 `list()` 기능을 그 파일 안에서 못 쓰게 되어버린다. 이런 이름은 예약어는 아니어도 되도록 피하는 게 안전하다.

### 변수 이름 규칙 (네이밍 컨벤션)

변수, 함수, 클래스에 이름을 지을 때 "단어를 어떻게 이어붙일지"에 대한 약속을 **네이밍 컨벤션(Naming Convention)** 이라고 한다. 

여러 단어(user, name)를 하나의 이름(username)으로 합칠 때 띄어쓰기를 쓸 수 없기 때문에, 대신 대소문자나 언더스코어로 단어 경계를 표시하는 것이다.

대표적인 방식들:

| 이름 | 표기 방식 | 예시 | 주로 쓰는 곳 |
|---|---|---|---|
| camelCase (카멜 표기법) | 첫 단어는 소문자, 이후 단어는 첫 글자만 대문자 (낙타 등처럼 울퉁불퉁해서 이런 이름) | userName, isActive | JavaScript의 변수/함수 |
| PascalCase (파스칼 표기법) | 모든 단어의 첫 글자를 대문자로 (camelCase와 거의 같은데 첫 글자도 대문자) | UserName, PaymentService | 클래스/컴포넌트 이름 (Python, JS 공통) |
| snake_case (스네이크 표기법) | 전부 소문자 + 단어 사이는 언더스코어(_)로 연결 (뱀처럼 길게 이어진 모양) | user_name, is_active | **Python의 변수/함수** |
| SCREAMING_SNAKE_CASE (대문자 스네이크) | snake_case인데 전부 대문자로 (소리치듯 강조) | MAX_SIZE, DATABASE_URL | 상수(값이 바뀌지 않는다고 약속한 변수) |
| kebab-case (케밥 표기법) | 전부 소문자 + 단어 사이는 하이픈(-)으로 연결 (꼬치처럼 이어진 모양) | user-name, my-component | URL 경로, 파일 이름, CSS 클래스명 |

**Python에서는 실제로 이렇게 쓴다**:

```python
user_name = "혜진"          # 변수 → snake_case
is_active = True           # 변수 → snake_case

def get_user_info():       # 함수 → snake_case
    pass

class UserProfile:         # 클래스 → PascalCase
    pass

MAX_LOGIN_ATTEMPTS = 5     # 상수(바뀌지 않을 값) → SCREAMING_SNAKE_CASE
```

JS와 비교하면:

| 대상 | JavaScript | Python |
|---|---|---|
| 변수/함수 | camelCase (userName) | snake_case (user_name) |
| 클래스/컴포넌트 | PascalCase (UserProfile) | PascalCase (UserProfile) — 동일 |
| 상수 | SCREAMING_SNAKE_CASE (MAX_SIZE) | SCREAMING_SNAKE_CASE (MAX_SIZE) — 동일 |

즉, 클래스 이름과 상수 이름 규칙은 Python과 JS가 똑같고, **변수와 함수 이름만 카멜(JS) vs 스네이크(Python)로 다르다.**

**실생활 비유**: 네이밍 컨벤션은 마치 옷장 정리 방식과 같다. 어떤 사람은 옷을 색깔별로 정리하고(camelCase), 어떤 사람은 종류별로 정리한다(snake_case). 방식 자체에 정답은 없지만, 한 옷장 안에서는 반드시 하나의 방식으로 통일해야 나중에 찾기 쉽다. 코드도 마찬가지로, 한 언어 안에서는 정해진 방식을 일관되게 따라야 협업이 수월해진다.

**실무에서는 이렇게 쓴다**: 실무에서 Python 변수 이름에 camelCase(`userName`)를 쓰면 문법 오류는 안 나지만, 코드 리뷰에서 지적받는 대표적인 항목이다. PEP 8(Python 공식 스타일 가이드)이 snake_case를 공식 권장하기 때문에, 대부분의 Python 팀과 오픈소스 프로젝트가 이를 따른다. 반대로 Next.js(JS/TypeScript) 프로젝트에서는 camelCase가 기본이라, 하나의 풀스택 프로젝트 안에서도 "백엔드(Python)는 snake_case, 프론트엔드(JS)는 camelCase"로 자연스럽게 나뉘게 된다.

### 기본 자료형

값의 종류를 자료형이라고 한다. 

실생활 비유 + 실무 활용 예시와 함께 살펴보면 이해가 쉽다.

| 자료형 | 설명 | 실생활 비유 | 실무 활용 예시 |
|---|---|---|---|
| int | 정수 (소수점 없는 숫자) | 사과 개수를 세는 것 — 사과 반 개짜리는 없음 | 사용자 ID, 페이지네이션(`offset=0, limit=20`), 재고 수량 |
| float | 실수 (소수점 있는 숫자) | 저울로 무게를 재는 것 — 152.3kg처럼 소수점까지 나옴 | 상품 가격, 평점(4.5점), 위도/경도 좌표 |
| str | 문자열 | 메모지에 적어놓은 글자 | 사용자 이름, API 응답 메시지, 파일 경로 |
| bool | 참/거짓 | 전등 스위치 — 켜짐 아니면 꺼짐, 중간은 없음 | 상태 플래그 (`is_active`, `is_deleted`, `is_verified`) |
| NoneType | 값이 없음을 의미 | 아직 아무것도 넣지 않은 빈 상자 | 함수의 기본 반환값, 아직 값이 없는 선택적(optional) 필드 |

> **플래그(flag)란?** 참(True) 또는 거짓(False)만으로 어떤 상태를 켜고 끄듯이 표시하는 값을 말한다. 예를 들어 `is_active`가 True면 "이 계정은 활성 상태", False면 "비활성 상태"를 뜻한다. 전등 스위치를 켜고 끄는 것과 똑같은 개념이라 bool과 잘 어울린다.

```python
# 실무에서 자주 보는 형태
is_active = True          # 이 계정이 활성 상태인가? (플래그)
is_deleted = False        # 이 게시글이 삭제됐는가? (플래그)
user_id = 1024            # 사용자 고유 번호
rating = 4.5              # 평점
nickname = None           # 아직 닉네임을 설정하지 않음 (선택 항목)
```

JS와 비교하면:

| Python | JavaScript | 비고 |
|---|---|---|
| int, float | number | JS는 정수/실수 구분 없이 하나(number)로 통일 |
| str | string | 거의 동일 |
| bool | boolean | Python은 True/False (대문자 시작), JS는 true/false |
| None | null / undefined | Python은 "없음"이 None 하나뿐. JS는 null과 undefined로 나뉨 |

### 숫자(int, float) 표기할 때 알아두면 좋은 것들

숫자를 다루다 보면 자주 마주치는 표기 방법 몇 가지가 있다.

```python
# 큰 숫자에 언더스코어로 자릿수 구분 (사람이 읽기 쉽게, 실제 값에는 영향 없음)
population = 51_780_000     # 51780000과 완전히 같은 값

# 나눗셈의 두 가지 방식
print(7 / 2)     # 3.5   → 일반 나눗셈 (항상 float로 결과가 나옴)
print(7 // 2)    # 3     → 정수 나눗셈 (몫만, 소수점 버림)
print(7 % 2)     # 1     → 나머지 (실무에서 짝수/홀수 판별, 순환 로직에 자주 씀)

# 거듭제곱
print(2 ** 10)   # 1024  → 2의 10제곱
```

**실무에서는 이렇게 쓴다**: `%`(나머지 연산)는 "5개씩 묶어서 표시하기", "짝수 페이지만 다른 배경색 주기"처럼 일정 간격으로 반복되는 로직에 자주 쓰인다. `_`(언더스코어) 구분은 `MAX_FILE_SIZE = 10_485_760`처럼 큰 숫자 상수를 코드에서 읽기 쉽게 적을 때 쓴다.

### 부동소수점의 함정

float(실수)를 다룰 때 반드시 알아야 할 함정이 있다. 직접 확인해보자.

```python
print(0.1 + 0.2)   # 예상: 0.3, 실제 결과: 0.30000000000000004
```

버그가 아니라 **컴퓨터가 실수를 저장하는 방식 자체의 한계**다.

**실생활 비유**: 자로 정확히 1/3(0.333...)을 재려고 하면 어떻게 될까? 자에는 눈금이 딱딱 떨어지게(1mm, 2mm...) 그어져 있어서, 아무리 정밀한 자라도 1/3 지점을 정확히 표시할 수는 없다. 아주 가까운 근사치에 눈금을 맞출 뿐이다.

**왜 이런 일이 생길까**: 우리가 평소 쓰는 숫자 체계는 0~9까지 열 개의 숫자를 쓰는 **10진법**이다. 그런데 컴퓨터 내부는 전기 신호가 켜짐/꺼짐 두 가지 상태만 구분할 수 있어서, 모든 숫자를 0과 1만으로 표현하는 **2진법**을 사용한다. 사람에게는 딱 떨어지는 0.1 같은 10진수가 2진법으로 바꾸면 딱 떨어지지 않는 무한소수가 되어버린다. 그래서 0.1을 저장하는 순간부터 이미 아주 미세한 오차를 안고 시작하고, 이런 값들을 더하면 오차가 눈에 보이는 수준으로 드러난다.

**왜 문제가 되는가**:

```python
# 이렇게 비교하면 안 됨
price = 0.1 + 0.2
print(price == 0.3)   # False! (0.30000000000000004 != 0.3)
```

**실무에서는 이렇게 쓴다**: 쇼핑몰 결제 금액, 정산 로직처럼 정확도가 생명인 계산에서 float를 직접 쓰면 오차가 누적돼서 실제 서비스 버그가 될 수 있다. 그래서 실무에서는 금액 계산에 float 대신 아래처럼 처리한다.

```python
# 방법 1: 반올림해서 비교
print(round(price, 1) == 0.3)   # True

# 방법 2: 정확한 소수 계산이 필요하면 decimal 모듈 사용 (실제 결제 로직에서 많이 씀)
from decimal import Decimal
price = Decimal("0.1") + Decimal("0.2")
print(price)   # 0.3 (정확하게 출력됨)
```

> 지금 단계에서는 "float는 완벽하게 정확하지 않을 수 있다"는 사실만 기억해두면 충분하다. 실제 결제/금액 계산 로직을 만들 때(PostgreSQL 연동 단계에서) decimal 타입을 다시 자세히 다룰 것이다.

### 타입 확인하기: type()

내가 만든 변수가 무슨 자료형인지 궁금할 때 `type()` 함수로 확인할 수 있다.

```python
age = 25
print(type(age))       # <class 'int'>

price = 9.99
print(type(price))     # <class 'float'>

name = "혜진"
print(type(name))      # <class 'str'>

is_valid = True
print(type(is_valid))  # <class 'bool'>
```

**실무에서는 이렇게 쓴다**: 코드에 문제가 생겼을 때 원인을 찾아 고치는 작업을 **디버깅**이라고 한다. `type()`은 디버깅할 때 "이 변수에 왜 예상과 다른 값이 들어있지?"를 확인하는 용도로 자주 쓴다. 특히 API로 데이터를 받았을 때, 숫자인 줄 알았던 값이 실제로는 문자열(`"25"`)로 들어오는 경우가 흔해서 `type()`으로 먼저 확인하는 습관이 버그를 줄여준다.

### 형변환

자료형을 다른 자료형으로 바꾸는 것을 **형변환**이라고 한다. 실생활로 치면 "저울에 잰 무게(kg)를 파운드로 단위 변환"하는 것과 비슷하다 — 같은 값을 다른 형식으로 바꿔서 쓰는 것이다. JS도 Number(), String() 같은 게 있었는데, Python도 비슷하다.

```python
# 문자열 → 정수
age_str = "25"
age_int = int(age_str)   # 25 (숫자로 변환됨)

# 정수 → 문자열
count = 10
count_str = str(count)   # "10" (문자열로 변환됨)

# 문자열 → 실수
price_str = "9.99"
price_float = float(price_str)  # 9.99
```

> 주의: `int("혜진")`처럼 숫자로 바꿀 수 없는 문자열을 변환하려 하면 `ValueError`(값이 잘못됐다는 뜻의 에러)가 발생한다. 형변환은 가능한 값만 바꿀 수 있다.

**실무에서는 이렇게 쓴다**: 웹 폼(사용자가 입력하는 화면의 입력창)이나 API 요청은 모든 값을 일단 문자열로 주고받는 경우가 많다. 예를 들어 사용자가 입력한 나이 "25"는 문자열로 넘어오는데, 이걸 계산에 쓰려면 반드시 `int()`로 형변환해야 한다. 앞으로 배울 FastAPI + Pydantic(데이터 형식을 자동으로 검사해주는 도구)을 쓰면 이 형변환을 자동으로 처리해주는데, 그 전에 원리를 알아두는 것이다.

### 예시

```python
# variables.py
user_name = "혜진"        # str, snake_case로 이름 지음
age = 30                 # int
height = 165.5           # float
is_studying = True       # bool
job = None               # 아직 정해지지 않음

print(f"{user_name}님은 {age}살이고, 키는 {height}cm입니다.")
print(f"공부 중인가요? {is_studying}")
print(type(age))  # <class 'int'>
```

### 요약

- 변수는 라벨 붙은 서랍과 같다. 이름은 그대로 두고 안에 담긴 값만 바꿀 수 있다
- 변수 이름 강제 규칙: 숫자로 시작 불가, 예약어(True, False, None, if, class 등 35개) 사용 불가, 띄어쓰기·특수문자 대부분 불가(언더스코어만 허용), 대소문자 구분됨
- 네이밍 컨벤션(권장 스타일): camelCase(JS 변수), PascalCase(클래스, Python·JS 공통), snake_case(Python 변수·함수), SCREAMING_SNAKE_CASE(상수, 공통), kebab-case(URL·파일명). **Python 변수/함수는 snake_case**가 공식 권장 방식(PEP 8)
- 실무에서는 비밀번호나 API 키 같은 값을 환경변수(.env 파일)로 따로 관리하고, 코드에서 불러와 변수에 담아 사용한다
- 기본 자료형: int(사과 개수처럼 딱 떨어지는 정수, 실무에서는 ID·페이지네이션), float(저울처럼 소수점 있는 실수, 실무에서는 가격·평점), str(문자열, 실무에서는 메시지·경로), bool(전등 스위치처럼 참/거짓, 실무에서는 is_active 같은 플래그), NoneType(빈 상자, 실무에서는 선택적 필드의 기본값)
- 숫자 표기: `_`로 큰 숫자 자릿수 구분 가능, `/`는 일반 나눗셈(float 결과), `//`는 정수 나눗셈, `%`는 나머지, `**`는 거듭제곱
- JS는 number로 정수/실수를 통합하지만, Python은 int/float로 분리
- JS는 null/undefined로 없음을 나누지만, Python은 None 하나로 통일
- 부동소수점 함정: 컴퓨터는 2진법(0과 1)으로만 숫자를 저장하는데, 10진법의 0.1 같은 소수는 2진법으로 딱 떨어지지 않아 0.1 + 0.2 같은 계산에서 미세한 오차가 생김. 실무에서 금액 계산은 decimal 모듈 사용
- type()으로 자료형을 확인하고, int()/str()/float()로 형변환 가능. 실무에서는 API로 받은 문자열 값을 계산용으로 형변환할 때 자주 사용
- 변환 불가능한 값을 형변환하려 하면 ValueError 발생
