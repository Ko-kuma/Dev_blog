## 딕셔너리 (Dictionary)

**정의**: 값을 순서가 아니라 "이름표(key)"로 찾아서 관리하는 자료구조다. 각 값(value)마다 고유한 키가 하나씩 붙어있다.

실생활 비유: 진짜 사전과 같다. 페이지 번호(인덱스)로 단어를 찾는 게 아니라, "사과"라는 단어(key)를 찾으면 그 뜻(value)이 바로 나온다.

### 변수 대신 왜 딕셔너리를 쓰는가

변수는 "몇 개가 필요한지, 이름이 뭔지 미리 다 아는" 소수의 고정된 값에 적합하다. 

하지만 회원이 3명이 아니라 500명이라면, 또는 key 자체가 프로그램이 실행되는 도중에 정해진다면 변수만으로는 대응이 안 된다.

결정적인 차이는, 딕셔너리는 key를 변수처럼 실행 중에 동적으로 다룰 수 있다는 점이다.

```python
person = {"name": "혜진", "age": 30}

field = input("어떤 정보를 보고 싶나요? ")   # 사용자가 "age"라고 입력했다고 가정
print(person[field])                          # person["age"]와 같은 효과, 30 출력
```

`person.age`처럼 변수/속성 이름은 코드에 미리 박혀 있어야 하지만, `person[field]`는 `field` 값이 실행 중에 뭐가 되든 그 이름으로 찾아준다. 이건 변수로는 흉내낼 수 없는 딕셔너리만의 능력이다.

```python
person = {
    "name": "혜진",
    "age": 30,
    "job": "developer",
}
```

### 값 꺼내기, 추가하기, 수정하기

```python
print(person["name"])      # 혜진 (key로 값을 꺼냄)

person["job"] = "backend developer"   # 이미 있는 key면 값을 수정
person["city"] = "성남"                 # 없는 key면 새로 추가

print(person)
```

### 값이 없는 key에 접근하면?

```python
print(person["phone"])   # KeyError! "phone"이라는 key가 없음
```

리스트에서 `fruits[10]`처럼 없는 인덱스에 접근하면 `IndexError`가 나듯, 딕셔너리도 없는 key에 접근하면 `KeyError`가 난다.

**안전하게 값을 꺼내는 법**: `get()`을 쓰면 key가 없어도 에러 대신 기본값(또는 `None`)을 돌려준다.

```python
print(person.get("phone"))          # None (에러 대신 None 반환)
print(person.get("phone", "없음"))   # "없음" (기본값을 직접 지정 가능)
```

> 함정: `딕셔너리["key"]`처럼 대괄호로 접근하면 key가 없을 때 프로그램이 그대로 멈춰버릴 수 있다. 사용자 입력이나 API 응답처럼 "key가 있을지 확신할 수 없는" 데이터를 다룰 때는 `get()`을 쓰는 습관을 들이는 게 안전하다.

### key로 값을 삭제하기: pop()

어제 리스트에서 `pop()`은 인덱스로 값을 꺼내며 삭제했는데, 딕셔너리의 `pop()`은 key로 값을 꺼내며 삭제한다.

```python
person = {"name": "혜진", "age": 30}
age = person.pop("age")   # "age" key의 값을 꺼내면서 삭제
print(age)                  # 30
print(person)                # {"name": "혜진"}
```

**실무에서는 이렇게 쓴다**: 임시로 쓰고 버릴 값을 꺼내면서 동시에 딕셔너리에서 지우고 싶을 때 쓴다. 예를 들어 요청 데이터에서 비밀번호처럼 민감한 값을 꺼내 별도로 처리하고, 나머지 데이터에서는 완전히 제거하고 싶을 때 유용하다.

### 두 딕셔너리 합치기: update()

```python
default_settings = {"theme": "light", "language": "ko"}
user_settings = {"theme": "dark"}

default_settings.update(user_settings)   # user_settings의 내용으로 덮어씀
print(default_settings)   # {"theme": "dark", "language": "ko"}
```

실생활 비유: 기본 설정지(default_settings) 위에 사용자가 수정한 부분(user_settings)만 겹쳐서 도장 찍는 것과 같다. 겹치는 항목은 새 값으로 덮어쓰고, 겹치지 않는 항목은 그대로 남는다.

**실무에서는 이렇게 쓴다**: 기본 설정값에 사용자가 입력한 값만 덮어씌울 때, 여러 곳에서 가져온 데이터를 하나로 합칠 때 자주 쓰인다.

### 중첩 딕셔너리 (Nested Dictionary)

어제 중첩 리스트를 배웠듯, 딕셔너리 안에도 또 다른 딕셔너리를 넣을 수 있다. 실제 API에서 주고받는 데이터는 대부분 이런 중첩 구조를 띤다.

```python
user = {
    "name": "혜진",
    "address": {
        "city": "판교",
        "zipcode": "13529",
    },
}
print(user["address"]["city"])   # 판교 (key를 두 번 이어서 접근)
```

실생활 비유: 회원 정보 서류철 안에 "주소" 폴더가 따로 들어있고, 그 폴더를 열어야 도시·우편번호 같은 세부 정보가 나오는 것과 같다.

**실무에서는 이렇게 쓴다**: 회원 정보 안에 주소, 결제 정보처럼 여러 하위 항목이 묶여 있는 API 응답 데이터를 다룰 때 거의 항상 이런 중첩 구조를 마주치게 된다.

### key, value, 쌍(pair)을 한 번에 순회하기

```python
person = {"name": "혜진", "age": 30}

for key in person.keys():        # key만 순회
    print(key)

for value in person.values():    # value만 순회
    print(value)

for key, value in person.items():   # key와 value를 동시에 순회
    print(key, value)
```

### key가 있는지 확인하기

멤버십 연산자(`in`)를 딕셔너리에도 그대로 쓸 수 있다. 다만 딕셔너리에서 `in`은 **value가 아니라 key를 기준으로 확인**한다.

```python
person = {"name": "혜진", "age": 30}
print("name" in person)   # True (key 확인)
print("혜진" in person)   # False (value는 in으로 확인 안 됨)
```

JS와 비교하면:

| 구분 | JavaScript | Python |
|---|---|---|
| 자료구조 이름 | Object 또는 Map | Dictionary (dict) |
| 선언 | `const person = {name: "혜진"}` | `person = {"name": "혜진"}` |
| 값 꺼내기 | `person.name` 또는 `person["name"]` | `person["name"]` |
| 안전하게 꺼내기 | `person?.phone` (옵셔널 체이닝) | `person.get("phone")` |
| key 존재 확인 | `"name" in person` | `"name" in person` — 동일 |

가장 큰 차이는 JS는 key에 따옴표를 생략할 수 있지만(`{name: "혜진"}`), Python은 문자열 key에 항상 따옴표를 붙여야 한다는 점이다(`{"name": "혜진"}`).

**실무에서는 이렇게 쓴다**: API를 통해 주고받는 데이터(JSON)는 사실상 딕셔너리와 구조가 동일하다. 회원 정보, 게시글 데이터 등 "이름표가 붙은 여러 값의 묶음"을 다루는 거의 모든 실무 코드에서 딕셔너리가 쓰인다.

```python
# 실무에서 자주 보는 형태 (API 응답을 흉내낸 데이터)
user_response = {
    "id": 1,
    "name": "혜진",
    "is_active": True,
}
print(user_response.get("email", "이메일 없음"))   # 없는 key라 "이메일 없음" 출력
```

**데이터베이스와의 연결**: PostgreSQL 같은 DB는 표(테이블) 형태로 데이터를 저장하는데, 한 줄(row)을 Python으로 가져오면 컬럼 이름이 key, 값이 value가 되어 딕셔너리 형태로 다뤄진다.

```python
# DB에서 가져온 한 줄을 딕셔너리로 표현하면 이런 모양
user = {
    "id": 1,
    "name": "혜진",
    "email": "hj@test.com",
}
```

전체 흐름으로 보면 `PostgreSQL 질의 → 결과가 딕셔너리(또는 비슷한 객체)로 변환 → FastAPI가 JSON으로 응답 → Next.js가 JSON을 받아 화면 표시` 순서로 이어진다. 딕셔너리와 JSON은 사실상 같은 구조라서, 이 과정 내내 데이터가 딕셔너리 모양을 그대로 유지하며 흘러간다.

> 정확히 짚으면: raw SQL을 직접 쓸 때는 결과가 튜플로 오는 경우도 많고, 나중에 배울 SQLAlchemy(ORM)를 쓰면 "객체" 형태로 온다. 하지만 이 객체도 딕셔너리처럼 값을 꺼낼 수 있고, 최종적으로 API 응답 시점엔 다 딕셔너리(JSON)로 통일된다.

### 딕셔너리의 key로 쓸 수 없는 것

딕셔너리의 key는 **변경 불가능한(Immutable) 자료형만 가능**하다. 어제 배운 리스트(가변)는 key로 쓸 수 없고, 튜플(불변)은 key로 쓸 수 있다.

```python
data = {(1, 2): "좌표1"}   # 튜플은 key로 사용 가능 (OK)
data = {[1, 2]: "좌표1"}   # 리스트는 key로 사용 불가 → TypeError
```

> 함정: 왜 리스트는 안 되고 튜플은 될까? key는 한 번 정해지면 절대 바뀌지 않아야 딕셔너리 내부에서 값을 빠르게 찾을 수 있는 구조(해시)로 관리되기 때문이다. 만약 key로 쓰인 리스트의 내용이 나중에 바뀌면, 딕셔너리 내부적으로 그 값을 다시 찾을 수 없게 되어버린다. 어제 배운 "가변 vs 불변" 개념이 여기서 실제로 적용되는 사례다.

## 셋 (Set)

**정의**: 순서가 없고, 중복된 값을 허용하지 않는 자료구조다.

실생활 비유: 우표 수집첩과 같다. 같은 우표를 두 장 넣으려고 해도 자동으로 한 장만 남는다. "어떤 우표들을 모았는가"만 중요하지, "어떤 순서로 모았는가"는 중요하지 않다.

```python
numbers = {1, 2, 2, 3, 3, 3}
print(numbers)   # {1, 2, 3} — 중복이 자동으로 제거됨
```

### `{ }`는 딕셔너리인가 셋인가?

방금 딕셔너리도 `{ }`, 셋도 `{ }`를 쓰는 걸 봤을 것이다. 같은 괄호를 쓰는데 Python은 어떻게 구분할까?

**기준은 안에 콜론(`:`)이 있는지 여부다.**

```python
a = {"name": "혜진"}   # 콜론이 있음 → 딕셔너리 (key: value)
b = {1, 2, 3}            # 콜론이 없음 → 셋 (값만 나열)
```

실생활 비유: `{ }`라는 같은 모양의 상자라도, 안에 이름표가 붙어 있으면(콜론) "사전"이고, 이름표 없이 물건만 담겨 있으면 "중복 없는 수집함"이라고 구분하는 것과 같다.

> 함정: 그런데 `{ }`처럼 **아무것도 안 든 빈 괄호는 딕셔너리로 취급된다.** 콜론이 있는지 확인할 대상 자체가 없기 때문에, Python은 이걸 기본값인 딕셔너리로 해석한다. 빈 셋을 만들고 싶다면 `{ }`가 아니라 반드시 `set()`이라고 써야 한다.

```python
empty = {}          # 딕셔너리! (셋이 아님)
print(type(empty))   # <class 'dict'>

empty_set = set()    # 진짜 빈 셋
print(type(empty_set))   # <class 'set'>
```

### 값 추가/삭제

```python
fruits = {"사과", "바나나"}
fruits.add("포도")        # 값 추가
fruits.remove("바나나")   # 값 삭제 (없는 값 삭제 시도하면 에러)
fruits.discard("수박")    # 값 삭제 (없어도 에러 안 남, remove보다 안전)
```

### 집합 연산 — 교집합, 합집합, 차집합

셋의 진짜 힘은 수학의 집합 연산을 그대로 쓸 수 있다는 점이다.

```python
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}

print(a & b)   # {3, 4} — 교집합 (둘 다 있는 값)
print(a | b)   # {1, 2, 3, 4, 5, 6} — 합집합 (둘 중 하나라도 있는 값)
print(a - b)   # {1, 2} — 차집합 (a에는 있고 b에는 없는 값)
print(a ^ b)   # {1, 2, 5, 6} — 대칭차집합 (겹치지 않는 값들만)
```

실생활 비유: 두 사람이 각자 좋아하는 과일 목록이 있을 때, 교집합은 "둘 다 좋아하는 과일", 합집합은 "둘 중 누구라도 좋아하는 과일 전체", 차집합은 "나만 좋아하고 상대는 안 좋아하는 과일", 대칭차집합은 "취향이 안 겹치는 과일들(둘 중 한 명만 좋아하는 것 전부)"이다.

JS와 비교하면:

| Python | JavaScript |
|---|---|
| `{1, 2, 3}` | `new Set([1, 2, 3])` |
| `a & b` (교집합) | 전용 연산자 없음 — `new Set([...a].filter(x => b.has(x)))`로 직접 구현 |
| `fruits.add("포도")` | `fruits.add("포도")` — 동일 |

> 함정: JS도 `Set` 객체가 있어서 중복 제거는 비슷하게 되지만, Python처럼 교집합·합집합·차집합을 기호(`&`, `|`, `-`) 하나로 바로 계산하는 기능은 없다. JS에서는 배열로 변환하고 `filter()`를 직접 조합해서 구현해야 한다.

**실무에서는 이렇게 쓴다**: 리스트에서 중복된 값을 빠르게 제거할 때(`list(set(중복있는리스트))`), 두 그룹의 공통 권한을 확인할 때(`user_permissions & required_permissions`), 태그 시스템에서 겹치는 태그를 찾을 때 자주 쓰인다.

```python
# 실무 예시: 리스트의 중복을 셋으로 빠르게 제거
emails = ["a@test.com", "b@test.com", "a@test.com"]
unique_emails = list(set(emails))   # 중복 제거 후 다시 리스트로
```

### 셋도 순서가 없고, 안에 리스트를 못 넣는다

딕셔너리의 key와 같은 이유로, 셋 안에도 리스트처럼 가변 자료형은 넣을 수 없다.

```python
tags = {"python", "python", "fastapi"}   # OK, 문자열은 불변이라 가능
groups = {[1, 2], [3, 4]}                 # TypeError, 리스트는 불가능
```

> 함정: 딕셔너리가 삽입 순서를 기억하는 것과 달리(Python 3.7+), 셋은 순서를 아예 보장하지 않는다. `print(numbers)`를 실행할 때마다 순서가 다르게 보일 수 있으니, 순서가 중요한 데이터라면 셋 대신 리스트를 써야 한다.

### frozenset — 셋의 불변 버전

어제 리스트와 튜플의 관계(가변 vs 불변)를 배웠는데, 셋에도 똑같은 관계의 짝이 있다. `frozenset`은 `set`과 똑같이 동작하지만 한 번 만들면 값을 추가·삭제할 수 없다.

```python
tags = frozenset(["python", "fastapi"])
tags.add("django")   # AttributeError! frozenset은 수정 불가

# 형변환을 통한 수정가능 참고자료 연습 확인하고 참고
```

**실무에서는 이렇게 쓴다**: 딕셔너리의 key나 셋의 원소는 불변 자료형만 가능하다. 그래서 "셋을 다른 셋 안에 넣고 싶을 때"는 일반 `set` 대신 `frozenset`을 써야 한다.

```python
groups = {frozenset(["a", "b"]), frozenset(["c", "d"])}   # OK, frozenset은 불변이라 가능
```

## dict/set vs list/tuple — 언제 뭘 쓰나

| 자료구조 | 순서 | 중복 | 접근 방식 | 대표 용도 |
|---|---|---|---|---|
| list | 있음 | 허용 | 인덱스(위치) | 순서가 중요한 목록 |
| tuple | 있음 | 허용 | 인덱스(위치) | 고정된 값 묶음 |
| dict | 있음(삽입 순서) | key 중복 불가 | key(이름표) | 이름표가 붙은 데이터 |
| set | 없음 | 불가 | 값 자체 | 중복 제거, 집합 연산 |
| frozenset | 없음 | 불가 | 값 자체 | 다른 셋/딕셔너리 안에 넣을 값 |

## 요약

- 딕셔너리는 key-value 쌍으로 값을 저장하며, 인덱스가 아니라 key로 값을 찾는다
- 변수는 개수와 이름을 미리 다 아는 소수의 값에 적합하고, 딕셔너리는 key 자체를 실행 중에 동적으로 다룰 수 있어 개수·이름을 미리 알 수 없는 데이터에 적합하다
- 없는 key에 `딕셔너리["key"]`로 접근하면 `KeyError`. `get()`을 쓰면 안전하게 기본값을 받을 수 있다
- `pop(key)`로 값을 꺼내면서 동시에 삭제할 수 있고, `update()`로 두 딕셔너리를 합칠 수 있다
- 딕셔너리 안에 또 다른 딕셔너리를 넣는 중첩 구조로, 실제 API 데이터 대부분이 이런 형태를 띤다
- `keys()`, `values()`, `items()`로 각각 key만, value만, 쌍을 순회할 수 있다
- 함정: 딕셔너리의 key와 셋의 원소는 리스트 같은 가변 자료형이 될 수 없다. 튜플(불변)은 key로 사용 가능
- 셋은 순서가 없고 중복을 허용하지 않는 자료구조로, `&`(교집합), `|`(합집합), `-`(차집합), `^`(대칭차집합) 연산을 지원한다
- 함정: 딕셔너리와 셋 모두 `{ }`를 쓰지만, 콜론(`:`)이 있으면 딕셔너리, 없으면 셋으로 구분된다. 빈 `{ }`는 셋이 아니라 딕셔너리로 취급되므로, 빈 셋은 반드시 `set()`으로 만들어야 한다
- `frozenset`은 셋의 불변 버전으로, 다른 셋이나 딕셔너리의 key 안에 셋을 넣고 싶을 때 사용한다
- 함정: JS에도 Set이 있지만 교집합/합집합/차집합을 기호로 바로 계산하는 기능은 없다
- 함정: 셋은 순서를 보장하지 않으므로, 순서가 중요하면 리스트를 써야 한다
- 실무에서는 API 데이터(dict), 중복 제거·권한 비교(set)에 각각 자주 쓰인다
- DB(PostgreSQL)에서 가져온 데이터도 결국 딕셔너리(또는 유사한 구조)로 다뤄지며, 딕셔너리와 JSON은 사실상 같은 구조라 DB → API 응답까지 이 형태가 그대로 이어진다
