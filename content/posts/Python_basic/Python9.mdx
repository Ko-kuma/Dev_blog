## 모듈이란?

변수, 함수, 클래스 같은 코드를 담아둔 파이썬 파일(.py) 하나를 모듈이라고 부른다.

실생활 비유: 공구함 안의 서랍 하나와 같다. "드라이버 서랍", "망치 서랍"처럼 도구를 종류별로 나눠 담아두면, 필요할 때 해당 서랍만 열어서 도구를 꺼내 쓸 수 있다. 코드도 기능별로 파일(모듈)에 나눠 담아두면 필요할 때 가져다 쓸 수 있다.

```python
# math_tools.py (모듈 파일)
def add(a, b):
    return a + b

def subtract(a, b):
    return a - b
```

## import — 모듈 가져오기

`import`로 다른 모듈을 가져와서 그 안의 함수나 변수를 쓸 수 있다.

```python
import math_tools   # math_tools.py 전체를 가져옴

print(math_tools.add(3, 2))   # 5, 모듈 이름을 붙여서 사용
```

**표준 라이브러리**: Python은 설치할 때부터 다양한 모듈을 기본으로 포함하고 있는데, 이걸 표준 라이브러리라고 부른다. 별도 설치 없이 바로 `import`해서 쓸 수 있다.

```python
import random
print(random.randint(1, 10))   # 1~10 사이 랜덤 정수

import datetime
print(datetime.date.today())    # 오늘 날짜
```

### import는 한 번만 실행된다

같은 모듈을 여러 파일에서 import해도, 그 모듈 안의 코드는 처음 한 번만 실행되고 이후로는 이미 실행된 결과를 재사용한다.

```python
# greeting.py
print("모듈이 로드됩니다")   # import되는 순간 딱 한 번만 출력됨

def hello():
    return "안녕!"
```

```python
# main.py
import greeting   # "모듈이 로드됩니다" 출력됨
import greeting   # 또 import해도 다시 출력되지 않음 (이미 로드된 걸 재사용)
```

실생활 비유: 도서관에서 책을 처음 대출할 때는 책을 서가에서 꺼내오는 과정이 필요하지만, 이미 대출한 책을 다시 찾을 때는 내 책상 위에 있는 걸 그대로 쓰면 되는 것과 같다. Python은 한 번 가져온 모듈을 내부적으로 기억해두고 재사용한다.

### `if __name__ == "__main__":` — 직접 실행과 import를 구분하기

파일을 두 가지 방식으로 쓸 수 있다: 터미널에서 직접 실행하거나, 다른 파일에서 `import`해서 가져다 쓰거나. 이 둘을 구분하고 싶을 때 쓰는 관용구다.

```python
# calculator.py
def add(a, b):
    return a + b

print(f"2 + 3 = {add(2, 3)}")   # 이 줄은 항상 실행됨 (직접 실행이든 import든)
```

```python
# main.py
from calculator import add
print(add(10, 20))
```

`main.py`를 실행하면 `calculator.py`를 import하는 순간, 원치 않았던 `"2 + 3 = 5"`라는 출력까지 딸려 나온다. 이걸 막으려면 "직접 실행될 때만 실행"하고 싶은 코드를 아래처럼 감싼다.

```python
# calculator.py (개선)
def add(a, b):
    return a + b

if __name__ == "__main__":       # 이 파일이 직접 실행될 때만 True
    print(f"2 + 3 = {add(2, 3)}")   # import될 때는 실행 안 됨
```

`__name__`이라는 특수 변수는, 파일이 직접 실행되면 `"__main__"`이라는 값을 갖고, 다른 파일에서 import되면 그 모듈의 이름(`"calculator"`)을 갖는다. 이 차이를 이용해 "테스트용 코드"나 "직접 실행할 때만 필요한 코드"를 분리해둘 수 있다.

실생활 비유: 요리책(모듈)에 "이 레시피를 참고용으로만 읽는 사람"과 "직접 이 요리책을 펴놓고 요리하는 사람"을 구분해서, 후자에게만 보이는 "지금 바로 시작하세요" 안내문을 넣어두는 것과 같다.

**실무에서는 이렇게 쓴다**: 하나의 파일이 "다른 곳에서 가져다 쓰는 라이브러리 역할"과 "그 자체로 실행 가능한 스크립트 역할"을 동시에 할 때 거의 항상 이 패턴을 쓴다. FastAPI 프로젝트의 진입점 파일(`main.py`)에서도 흔히 볼 수 있다.

## from ... import — 필요한 것만 콕 집어오기

모듈 전체가 아니라 특정 함수나 클래스 하나만 가져오고 싶을 때 쓴다.

```python
from math_tools import add   # add 함수만 가져옴

print(add(3, 2))   # 5, 모듈 이름 없이 바로 사용 가능
```

`import math_tools`는 서랍 전체를 가져와서 "서랍.도구"처럼 꺼내 쓰는 것이고, `from math_tools import add`는 서랍에서 도구 하나만 미리 꺼내서 손에 쥐고 있는 것과 같다. 자주 쓰는 것 하나둘 정도만 필요하면 후자가 코드를 더 짧게 만들어준다.

## as — 별칭 붙이기

모듈이나 함수 이름이 길거나 겹칠 때, `as`로 짧은 별명을 붙일 수 있다.

```python
import datetime as dt   # dt라는 별칭으로 사용

print(dt.date.today())   # datetime.date.today()와 동일
```

**실무에서는 이렇게 쓴다**: `import pandas as pd`, `import numpy as np`처럼 특정 별칭이 커뮤니티 전체의 관례로 굳어진 경우가 많다. 이런 관례를 따르면 다른 사람의 코드를 읽을 때도 낯설지 않다.

## 패키지란?

여러 개의 모듈을 폴더 하나로 묶어놓은 것이다. 모듈이 서랍 하나라면, 패키지는 그 서랍들을 여러 개 모아놓은 공구함 전체에 가깝다.

```
my_package/
├── __init__.py
├── math_tools.py
└── string_tools.py
```

`__init__.py`는 "이 폴더는 그냥 폴더가 아니라 패키지입니다"라고 Python에게 알려주는 표시 파일이다. 지금 단계에서는 "패키지 폴더에는 이 파일이 있어야 한다" 정도만 기억해두면 충분하다.

```python
from my_package import math_tools
print(math_tools.add(3, 2))
```

## pip — 패키지 설치 도구

지금까지는 내가 직접 만든 모듈이나 Python에 기본 포함된 표준 라이브러리를 썼는데, 전 세계 개발자들이 만들어 올려둔 패키지도 가져다 쓸 수 있다. 이때 쓰는 도구가 `pip`이다.

```bash
pip install requests      # requests라는 패키지를 설치
pip list                    # 지금 설치된 패키지 목록 확인
pip uninstall requests    # 패키지 제거
```

`pip`가 설치해주는 패키지들은 PyPI(Python Package Index)라는 공식 저장소에 등록되어 있다. `pip install fastapi`라고 치면, PyPI에서 fastapi 패키지를 찾아 내려받아 설치해준다.

## requirements.txt — 설치할 패키지 목록 관리하기

프로젝트에 필요한 패키지 목록을 파일 하나에 정리해두고, 그 파일로 한 번에 설치할 수 있다.

```
fastapi==0.115.0
requests==2.32.0
```

```bash
pip install -r requirements.txt   # 파일에 적힌 패키지를 한꺼번에 설치
pip freeze > requirements.txt      # 현재 설치된 패키지 목록을 파일로 저장
```

실생활 비유: 이사 갈 집에 필요한 가구 목록을 미리 종이에 적어두는 것과 같다. 새 집(새로운 컴퓨터, 서버)에 도착했을 때 그 종이 한 장만 있으면 필요한 가구를 정확히 똑같이 다시 채워 넣을 수 있다.

**실무에서는 이렇게 쓴다**: 팀원이 새로 프로젝트에 합류했을 때, `pip install -r requirements.txt` 한 줄이면 나와 똑같은 개발 환경을 그대로 재현할 수 있다. GitHub에 코드를 올릴 때 이 파일도 함께 올려두는 게 표준 관례다.

### 버전을 왜 고정해둘까

`fastapi==0.115.0`처럼 `==`로 정확한 버전을 명시해두는 것을 버전 고정(Pinning)이라고 한다.

패키지는 시간이 지나면 새 버전이 계속 나오는데, 버전이 바뀌면서 기존 코드와 안 맞게 동작이 바뀌는 경우가 흔하다. 버전을 고정해두지 않으면, 오늘 내 컴퓨터에서는 잘 되던 코드가 몇 달 뒤 다른 사람 컴퓨터(또는 배포 서버)에서 최신 버전이 새로 설치되면서 갑자기 안 되는 상황이 생길 수 있다.

실생활 비유: 요리 레시피에 "소금 약간"이 아니라 "소금 5g"이라고 정확히 적어두는 것과 같다. "약간"은 사람마다 다르게 해석되지만, 정확한 숫자는 언제 누가 만들어도 똑같은 결과가 나온다.

## 가상환경(venv)이 필요한 이유

`pip install`로 패키지를 설치하면 기본적으로 컴퓨터 전체에 딱 하나뿐인 공간에 설치된다. 그런데 프로젝트를 여러 개 진행하다 보면 문제가 생긴다.

```
프로젝트 A: FastAPI 0.100 버전이 필요함
프로젝트 B: FastAPI 0.115 버전이 필요함
```

컴퓨터 전체에 패키지가 하나만 설치되는 구조라면, 두 프로젝트가 서로 다른 버전을 동시에 쓸 수 없다.

실생활 비유: 원룸 하나에서 여러 명이 각자 물건을 뒤섞어 놓고 사는 것과 같다. 방(가상환경)을 프로젝트별로 따로 나누면, 서로의 물건(패키지 버전)이 섞이지 않고 각자 필요한 것만 깔끔하게 관리할 수 있다.

## 가상환경 만들고 쓰기

```bash
python -m venv venv          # venv라는 이름의 가상환경 생성

# 활성화 (이 명령을 실행해야 가상환경 "안"으로 들어감)
source venv/bin/activate      # Mac/Linux
venv\Scripts\activate         # Windows

pip install fastapi            # 이제 이 프로젝트의 가상환경 안에만 설치됨

deactivate                     # 가상환경에서 나가기
```

가상환경을 활성화하면, 터미널 프롬프트 앞에 `(venv)`라는 표시가 붙는다. 이게 "지금 가상의 독립된 방 안에 들어와 있다"는 신호다.

> 함정: 가상환경을 활성화하지 않은 채로 `pip install`을 하면, 그 패키지는 프로젝트 전용 공간이 아니라 컴퓨터 전체 공간에 설치되어 버린다. 터미널에 `(venv)` 표시가 있는지 항상 확인하는 습관이 필요하다.

### venv 폴더는 GitHub에 올리지 않는다

가상환경 폴더(`venv`)는 용량이 크고, 컴퓨터마다 경로나 설정이 조금씩 달라서 그대로 공유하면 오히려 문제가 생긴다. 그래서 `.gitignore` 파일에 등록해서 Git이 아예 추적하지 않게 만든다.

```
# .gitignore
venv/
__pycache__/
```

대신 `requirements.txt`만 올려두면, 다른 사람이 그 파일로 자기 컴퓨터에 자기만의 가상환경을 새로 만들 수 있다.

실생활 비유: 이사할 때 방(가상환경) 자체를 통째로 옮기는 게 아니라, "이 방에 뭘 채워 넣어야 하는지" 적힌 목록(requirements.txt)만 넘겨주고, 새 방은 받는 사람이 직접 꾸리게 하는 것과 같다.

**실무에서는 이렇게 쓴다**: `venv`, `__pycache__`처럼 "직접 만든 코드가 아니라 재생성 가능한 것들"은 `.gitignore`에 등록하는 것이 표준 관례다. 프로젝트를 시작할 때 `requirements.txt`와 `.gitignore`를 함께 세팅해두는 것이 일반적인 순서다.

**실무에서는 이렇게 쓴다**: 새 프로젝트를 시작할 때 가장 먼저 하는 일이 가상환경을 만드는 것이다. 이렇게 해야 나중에 다른 프로젝트의 패키지 버전과 충돌하지 않고, `requirements.txt`로 그 프로젝트만의 정확한 패키지 구성을 기록할 수 있다.

JS와 비교하면:

| Python | JavaScript |
|---|---|
| pip | npm (또는 yarn, pnpm) |
| requirements.txt | package.json |
| venv (가상환경 폴더) | node_modules (패키지 설치 폴더) |
| PyPI | npm registry |

가장 큰 차이는, JS(Node.js)는 프로젝트마다 `node_modules` 폴더가 자동으로 분리되어 애초에 가상환경 같은 개념이 필요 없다는 점이다. Python은 기본적으로 컴퓨터 전체가 하나의 공간이라, 프로젝트별로 격리하려면 가상환경을 직접 만들어야 한다.

## 요약

- 모듈은 코드를 담은 파일 하나, 패키지는 여러 모듈을 묶은 폴더다
- 같은 모듈을 여러 곳에서 import해도 내부 코드는 처음 한 번만 실행되고, 이후로는 이미 로드된 결과를 재사용한다
- `if __name__ == "__main__":`으로 "이 파일이 직접 실행될 때만" 실행되는 코드를 구분할 수 있다. import될 때는 이 블록이 실행되지 않는다
- `import 모듈`은 통째로, `from 모듈 import 이름`은 필요한 것만 콕 집어서 가져온다
- `as`로 모듈에 별칭을 붙일 수 있고, `import pandas as pd` 같은 관례가 커뮤니티에 자리잡혀 있다
- `pip`으로 PyPI에 등록된 외부 패키지를 설치·삭제·조회할 수 있다
- `requirements.txt`에 필요한 패키지 목록을 적어두면, 다른 환경에서도 동일하게 설치를 재현할 수 있다
- 버전 고정(`==0.115.0`)으로 시간이 지나도 항상 같은 버전이 설치되게 만들어, 예상치 못한 버전 충돌을 막는다
- 가상환경(venv)은 프로젝트마다 독립된 패키지 설치 공간을 만들어, 프로젝트 간 버전 충돌을 막아준다
- 함정: 가상환경을 활성화하지 않고 설치하면 컴퓨터 전체 공간에 패키지가 설치된다. 터미널의 `(venv)` 표시를 항상 확인해야 한다
- `venv`, `__pycache__` 폴더는 `.gitignore`에 등록해서 GitHub에 올리지 않고, 대신 `requirements.txt`로 환경을 재현하게 한다
- JS의 npm/package.json/node_modules와 대응되지만, Node.js는 프로젝트별 폴더 분리가 기본이라 별도의 가상환경 개념이 필요 없다는 점이 Python과 다르다
