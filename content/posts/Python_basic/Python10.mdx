## 파일 입출력이란?

프로그램이 종료되어도 데이터가 사라지지 않도록, 파일에 내용을 저장하거나 저장된 내용을 읽어오는 것이다.

지금까지 만든 변수나 리스트는 프로그램이 끝나는 순간 전부 사라진다. 파일에 써두면 컴퓨터를 껐다 켜도 그 내용이 그대로 남아있다.

## 파일 열고 닫기: open(), close()

```python
file = open("note.txt", "w")   # "w"는 쓰기 모드로 파일을 염
file.write("안녕하세요")
file.close()                     # 반드시 닫아줘야 함
```

실생활 비유: 냉장고 문을 여는 것과 같다. 문을 열어서(open) 뭔가를 넣거나 꺼낸 다음에는, 반드시 문을 닫아야(close) 안에 있는 것들이 상하지 않는다. 파일도 다 쓰고 나면 닫아줘야 데이터가 확실히 저장되고, 다른 프로그램도 그 파일에 접근할 수 있다.

> 함정: `close()`를 깜빡하면 파일에 쓴 내용이 완전히 저장되지 않을 수도 있고, 파일이 계속 "열린 상태"로 남아 다른 곳에서 접근하지 못하게 될 수도 있다.

## with문으로 안전하게 파일 다루기

`close()`를 깜빡할 위험을 없애주는 문법이 `with`문이다. 예외처리에서 배운 `finally`가 "항상 실행되는 정리 코드"였는데, `with`문은 파일 같은 자원에 대해 이 패턴을 자동으로 처리해준다.

```python
with open("note.txt", "w") as file:
    file.write("안녕하세요")
# with 블록을 빠져나오는 순간 자동으로 close()가 호출됨
```

실생활 비유: 자동으로 닫히는 냉장고 문과 같다. 사람이 깜빡하고 문을 안 닫아도, 일정 시간이 지나면 저절로 닫혀서 안전하게 관리된다. `with`문은 블록이 끝나는 순간 무조건 파일을 닫아주기 때문에, `close()`를 깜빡할 걱정이 없다.

**실무에서는 이렇게 쓴다**: 파일뿐 아니라 데이터베이스 연결처럼 "쓰고 나면 반드시 정리해야 하는" 모든 자원을 다룰 때 `with`문을 쓰는 것이 표준 관례다. 직접 `open()`/`close()`를 쓰는 코드는 실무에서 거의 보기 어렵다.

## 파일 모드

`open()`의 두 번째 인자로 어떤 목적으로 파일을 열지 지정한다.

| 모드 | 의미 |
|---|---|
| `"r"` | 읽기 (Read). 파일이 없으면 에러 |
| `"w"` | 쓰기 (Write). 파일이 있으면 기존 내용을 전부 지우고 새로 씀 |
| `"a"` | 추가 (Append). 기존 내용 뒤에 이어서 씀 |
| `"x"` | 새로 생성. 이미 파일이 있으면 에러 |

> 함정: `"w"` 모드로 열면 기존 파일 내용이 전부 사라진다. 기존 내용을 유지하면서 뒤에 이어 쓰고 싶다면 반드시 `"a"` 모드를 써야 한다.

## 파일 쓰기와 읽기

```python
# 쓰기
with open("note.txt", "w", encoding="utf-8") as file:
    file.write("첫 번째 줄\n")
    file.write("두 번째 줄\n")

# 전체 읽기
with open("note.txt", "r", encoding="utf-8") as file:
    content = file.read()
    print(content)

# 한 줄씩 읽기
with open("note.txt", "r", encoding="utf-8") as file:
    for line in file:          # 파일도 반복문으로 한 줄씩 순회 가능
        print(line.strip())     # strip()으로 줄 끝의 개행 문자 제거

# 모든 줄을 리스트로 한 번에 받기
with open("note.txt", "r", encoding="utf-8") as file:
    lines = file.readlines()    # ["첫 번째 줄\n", "두 번째 줄\n"]
    print(lines[0])               # 첫 번째 줄만 꺼내 쓸 수 있음
```

**`encoding="utf-8"`을 꼭 붙이는 이유**: 운영체제에 따라 파일을 여는 기본 인코딩(문자를 숫자로 바꾸는 방식)이 다를 수 있는데, 이 설정 없이 한글이 포함된 파일을 열면 글자가 깨져서 보일 수 있다. 항상 `encoding="utf-8"`을 명시해두면 이 문제를 예방할 수 있다.

**파일이 없을 때는 예외처리와 함께**: 없는 파일을 `"r"` 모드로 열려고 하면 `FileNotFoundError`가 발생한다.

```python
try:
    with open("없는파일.txt", "r") as file:
        print(file.read())
except FileNotFoundError:
    print("파일을 찾을 수 없습니다")
```

JS와 비교하면:

| Python | JavaScript (Node.js) |
|---|---|
| `with open() as f:` | `fs.readFile()` / `fs.promises.readFile()` |
| 자동으로 파일 닫힘 | 콜백/Promise가 끝나면 별도 정리 필요 없음(Node가 관리) |

**실무에서는 이렇게 쓴다**: 설정 파일을 읽어오거나, 로그를 파일에 계속 추가하거나(`"a"` 모드), 사용자가 업로드한 데이터를 저장할 때 파일 입출력을 쓴다.

## 클래스란?

같은 종류의 데이터와 기능을 하나로 묶어서, 필요할 때마다 똑같은 구조를 여러 개 찍어낼 수 있게 만든 설계도다.

실생활 비유: 붕어빵 틀과 같다. 붕어빵 틀(클래스) 하나만 있으면, 그 틀로 붕어빵을 몇 개든 찍어낼 수 있다. 각 붕어빵(인스턴스)은 서로 다른 개별 붕어빵이지만, 모두 같은 틀에서 나왔기 때문에 같은 모양과 구조를 가진다.

```python
class Dog:                      # Dog이라는 클래스(붕어빵 틀) 정의
    def __init__(self, name, age):
        self.name = name          # 이 강아지만의 이름
        self.age = age             # 이 강아지만의 나이

    def bark(self):
        return f"{self.name}: 멍멍!"

my_dog = Dog("초코", 3)   # 클래스로 실제 강아지(인스턴스, 붕어빵) 하나를 만듦
print(my_dog.bark())        # 초코: 멍멍!
```

## `__init__`과 self

**`__init__`**: 클래스로 인스턴스를 만드는 순간 자동으로 실행되는 특별한 메서드다. "이 인스턴스가 태어날 때 어떤 값을 채워 넣을지"를 정의한다. 붕어빵 틀에 재료(팥, 반죽)를 채워 넣는 과정에 해당한다.

**self**: 클래스 안의 메서드에서, "지금 이 메서드를 호출한 바로 그 인스턴스 자기 자신"을 가리키는 이름이다.

```python
class Dog:
    def __init__(self, name, age):
        self.name = name   # self.name = 이 인스턴스의 name 속성에 저장
        self.age = age

dog1 = Dog("초코", 3)
dog2 = Dog("보리", 5)

print(dog1.name)   # 초코 — dog1만의 name
print(dog2.name)   # 보리 — dog2만의 name (서로 다른 붕어빵이라 내용물도 다름)
```

실생활 비유: `self`는 붕어빵 틀에서 찍어낸 "지금 이 붕어빵"을 가리키는 말이다. 틀(클래스) 자체는 하나지만, 그 틀로 찍어낸 붕어빵(인스턴스)은 여러 개일 수 있고, 각자 자기 몫의 재료(속성)를 따로 가진다.

> 함정: `self`는 실무 관례상 이름을 그렇게 정한 것일 뿐, 진짜 문법 요소는 "메서드의 첫 번째 매개변수가 항상 인스턴스 자기 자신을 받는다"는 규칙이다. 다른 이름을 써도 동작은 하지만, `self`가 아닌 다른 이름을 쓰면 다른 개발자들이 코드를 읽을 때 혼란스러워하므로 절대 바꾸지 않는 게 좋다.

## 속성(Attribute)과 메서드(Method)

- **속성**: 인스턴스가 가지고 있는 데이터(`self.name`, `self.age`)
- **메서드**: 인스턴스가 할 수 있는 행동(`bark()` 같은, 클래스 안에 정의된 함수)

```python
print(my_dog.name)    # 속성 접근 (괄호 없음)
print(my_dog.bark())   # 메서드 호출 (괄호 있음, 함수니까 실행해야 함)
```

이건 함수 자체와 실행 결과를 괄호 유무로 구분하던 것과 같은 원리다. 속성은 그냥 값이라 괄호가 없고, 메서드는 함수이기 때문에 괄호로 호출해야 한다.

### `__str__` — 인스턴스를 print()했을 때 뭐가 나올지 정하기

인스턴스를 그냥 `print()`하면 사람이 읽기 어려운 값이 나온다.

```python
print(my_dog)   # <__main__.Dog object at 0x7f8a1c2b3d90>
```

`__str__`이라는 메서드를 정의해두면, `print()`했을 때 이 메서드가 반환하는 값이 대신 출력된다.

```python
class Dog:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def __str__(self):
        return f"{self.name}({self.age}살)"   # print()할 때 이 문자열이 나오게 됨

my_dog = Dog("초코", 3)
print(my_dog)   # 초코(3살)
```

실생활 비유: 신분증에 인쇄된 정보와 같다. `__str__`이 없으면 신분증 뒷면의 알아보기 힘든 일련번호(메모리 주소)만 보이고, `__str__`을 정의해두면 신분증 앞면의 "이름, 나이"처럼 사람이 알아볼 수 있는 정보가 대신 보인다.

**실무에서는 이렇게 쓴다**: 디버깅할 때 `print(user)` 한 줄로 그 인스턴스의 핵심 정보를 바로 확인할 수 있어서, 거의 모든 클래스에 `__str__`을 기본으로 정의해두는 게 관례다.

### 클래스 변수 vs 인스턴스 변수

지금까지 다룬 `self.name`처럼 `self.`를 붙인 변수는 인스턴스마다 따로 가지는 **인스턴스 변수**다. 이와 달리, 클래스 안에서 `self` 없이 바로 선언한 변수는 **클래스 변수**로, 그 클래스로 만든 모든 인스턴스가 공유한다.

```python
class Dog:
    species = "개"   # 클래스 변수 — 모든 Dog 인스턴스가 공유함

    def __init__(self, name):
        self.name = name   # 인스턴스 변수 — 각자 따로 가짐

dog1 = Dog("초코")
dog2 = Dog("보리")

print(dog1.species)   # 개
print(dog2.species)   # 개 (같은 값을 공유)
print(dog1.name)        # 초코
print(dog2.name)        # 보리 (서로 다름)
```

실생활 비유: 클래스 변수는 같은 학교 학생들이 다 같이 쓰는 교훈(校訓) 같은 것이고, 인스턴스 변수는 학생 개개인의 이름과 같다. 교훈은 학생 전체가 공유하지만, 이름은 학생마다 다르다.

> 함정: 함수의 기본값으로 리스트를 쓰면 위험했던 것과 똑같은 원리로, 클래스 변수에 리스트나 딕셔너리 같은 가변 자료형을 쓰면 모든 인스턴스가 그 하나를 공유하게 되어 예상치 못한 버그가 생긴다. 인스턴스마다 따로 가져야 하는 값은 반드시 `__init__` 안에서 `self.변수 = ...`로 만들어야 한다.

JS와 비교하면:

| Python | JavaScript |
|---|---|
| `class Dog:` | `class Dog {` |
| `def __init__(self, name):` | `constructor(name) {` |
| `self.name = name` | `this.name = name;` |
| `new_dog = Dog("초코", 3)` | `const newDog = new Dog("초코", 3);` (JS는 `new` 키워드 필요) |

가장 큰 차이는 Python은 인스턴스를 만들 때 `new` 같은 별도 키워드 없이 `클래스이름(인자)`로 바로 만들 수 있다는 점이다. 또한 Python은 `self`를 모든 메서드의 첫 매개변수로 명시적으로 써야 하지만, JS는 `this`가 암묵적으로 제공되어 매개변수로 적지 않는다.

## 클래스 vs 인스턴스 — 헷갈리지 않기

- **클래스**: 설계도, 틀 그 자체 (`Dog`)
- **인스턴스**: 그 설계도로 실제로 만들어낸 것 (`my_dog`, `dog1`, `dog2`)

```python
print(type(my_dog))   # <class '__main__.Dog'> — my_dog은 Dog 클래스의 인스턴스
```

실무에서 자주 쓰는 표현으로 "Dog 클래스를 인스턴스화(instantiate)한다"는 말은, 곧 "붕어빵 틀로 실제 붕어빵을 하나 찍어낸다"는 뜻이다.

**실무에서는 이렇게 쓴다**: 회원 한 명, 게시글 하나처럼 "같은 구조를 가진 여러 개의 데이터"를 다룰 때 클래스를 쓴다. 나중에 배울 FastAPI와 SQLAlchemy에서 "User"나 "Post" 같은 데이터 모델이 전부 이런 클래스로 정의된다.

## 클래스 상속 (Inheritance)

이미 예외처리에서 `class InsufficientBalanceError(Exception):`처럼 클래스를 한 번 살짝 써봤는데, 그때 괄호 안에 적었던 `Exception`이 바로 상속이다. 어떤 클래스가 다른 클래스의 속성과 메서드를 그대로 물려받는 것을 상속이라고 한다.

```python
class Animal:                      # 부모 클래스
    def __init__(self, name):
        self.name = name

    def eat(self):
        return f"{self.name}가 밥을 먹는다"

class Dog(Animal):                 # Animal을 상속받는 자식 클래스
    def bark(self):
        return f"{self.name}: 멍멍!"

my_dog = Dog("초코")
print(my_dog.eat())    # 초코가 밥을 먹는다 (부모 클래스의 메서드를 그대로 물려받음)
print(my_dog.bark())   # 초코: 멍멍! (자식 클래스만의 메서드)
```

`Dog(Animal)`이라고 쓰는 순간, `Dog`는 `Animal`이 가진 모든 것(`eat` 메서드 등)을 그대로 물려받고, 거기에 `bark`라는 자기만의 기능을 추가로 갖는다.

실생활 비유: 붕어빵 틀 중에서도 "기본 붕어빵 틀"(부모)을 바탕으로 "초콜릿이 추가된 붕어빵 틀"(자식)을 새로 만드는 것과 같다. 기본 틀의 모양(공통 기능)은 그대로 가져오면서, 초콜릿(추가 기능)만 새로 더하는 것이다.

`class InsufficientBalanceError(Exception):`도 마찬가지로, Python이 미리 만들어둔 `Exception`이라는 부모 클래스의 모든 기능(에러 메시지 저장, `except`로 잡히는 방식 등)을 그대로 물려받으면서, 그 서비스만의 이름을 새로 붙인 것이었다.

**실무에서는 이렇게 쓴다**: "일반 사용자"와 "관리자"처럼 공통된 속성은 많지만 일부 기능만 다른 경우, 공통 부분을 부모 클래스로 만들고 차이 나는 부분만 자식 클래스에서 추가하는 식으로 코드 중복을 줄인다.

## 요약

- 파일 입출력으로 프로그램이 종료돼도 데이터를 남길 수 있다. `open()`으로 열고 `close()`로 닫아야 하며, `with`문을 쓰면 자동으로 닫혀서 더 안전하다
- 파일 모드: `"r"`(읽기), `"w"`(덮어쓰기), `"a"`(이어쓰기). 함정: `"w"`는 기존 내용을 전부 지운다
- 한글 파일을 다룰 때는 `encoding="utf-8"`을 꼭 지정해야 글자가 깨지지 않는다
- `read()`(전체), `for line in file`(한 줄씩), `readlines()`(리스트로 전부) 중 상황에 맞게 골라 읽는다
- 없는 파일을 읽으려 하면 `FileNotFoundError`가 발생하므로 예외처리와 함께 쓴다
- 클래스는 같은 구조의 데이터를 여러 개 찍어낼 수 있는 설계도(붕어빵 틀)이고, 인스턴스는 그 설계도로 실제로 만든 것(붕어빵)이다
- `__init__`은 인스턴스가 만들어질 때 자동 실행되는 메서드이고, `self`는 메서드 안에서 인스턴스 자기 자신을 가리킨다
- 속성은 인스턴스가 가진 데이터(괄호 없이 접근), 메서드는 인스턴스가 할 수 있는 행동(괄호로 호출)
- `__str__`을 정의하면 `print(인스턴스)`했을 때 알아보기 쉬운 문자열이 출력된다
- 함정: 클래스 변수(모든 인스턴스가 공유)에 리스트 같은 가변 자료형을 쓰면, 함수 기본값 함정과 같은 이유로 예상치 못하게 값이 공유된다. 인스턴스마다 달라야 하는 값은 `__init__` 안에서 `self.`로 만들어야 한다
- 상속으로 부모 클래스의 기능을 자식 클래스가 그대로 물려받고, 자식만의 기능을 추가할 수 있다. 예외처리에서 `Exception`을 상속받은 것도 같은 원리다
- JS와 비교하면 Python은 인스턴스를 만들 때 `new` 키워드가 필요 없고, `self`를 메서드 매개변수에 명시적으로 적어야 한다는 점이 다르다
